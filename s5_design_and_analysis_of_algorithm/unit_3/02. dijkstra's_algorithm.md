# Dijkstra's Algorithm

# 1. Introduction

In many real-world situations, we need to travel from **one location** to **another location** while spending the **least possible distance, time, or cost**.

For example,

- Finding the shortest route on Google Maps.
- Finding the fastest route for food delivery.
- Routing data packets over the Internet.
- GPS Navigation Systems.
- Flight Route Planning.
- Railway Route Optimization.
- Robot Path Planning.

In all these situations, our goal is **not just to reach the destination**, but to reach it using the **shortest possible path**.

This type of problem is known as the **Shortest Path Problem**.

One of the most popular algorithms used to solve this problem is **Dijkstra's Algorithm**.

Dijkstra's Algorithm is one of the most important applications of the **Greedy Algorithm** in Graph Theory.

Before understanding Dijkstra's Algorithm, we must first understand some basic concepts related to graphs and shortest paths.

---

# 2. What is a Graph?

A **Graph** is a data structure that consists of a collection of **vertices (nodes)** and **edges (connections)**.

- **Vertices (V):** Represent objects or locations.
- **Edges (E):** Represent the connection between two vertices.

For example,

suppose we have four cities.

```text
Delhi

Mumbai

Ahmedabad

Pune
```

If roads connect these cities, then

- Each city becomes a **Vertex**.
- Each road becomes an **Edge**.

Graph representation

```text
            Delhi
           /     \
          /       \
     Mumbai ----- Ahmedabad
          \
           \
           Pune
```

Here,

```text
Vertices = Cities

Edges = Roads
```

Graphs help us represent relationships between different locations.

---

# 3. What is a Weighted Graph?

Sometimes every edge has a numerical value associated with it.

This value is called the **Weight**.

The weight may represent

- Distance
- Time
- Cost
- Fuel Consumption
- Cable Length
- Traffic Delay

Example

```text
          A
       4 / \ 2
        /   \
       B-----C
         3
```

Here,

```text
A → B = 4 km

A → C = 2 km

B → C = 3 km
```

This graph can also be represented as a table.

| Edge | Weight |
|------|--------|
| A → B | 4 |
| A → C | 2 |
| B → C | 3 |

Since every edge has a weight,

this is called a **Weighted Graph**.

---

## Why Do We Need Weights?

In real-world problems, every connection usually has some cost associated with it.

For example,

| Connection | Possible Weight |
|------------|-----------------|
| Road | Distance (km) |
| Flight | Ticket Cost |
| Railway | Travel Time |
| Internet Network | Transmission Delay |
| Water Pipeline | Installation Cost |
| Electric Cable | Cable Length |

Without weights,

we only know **whether two vertices are connected**.

With weights,

we also know **how expensive or how far that connection is**.

This additional information helps us solve optimization problems such as finding the **Shortest Path**.

---
# 4. What is a Path?

Before understanding **Dijkstra's Algorithm**, we must first understand one of the most basic concepts in Graph Theory called a **Path**.

Suppose you are planning to travel from **Ahmedabad** to **Delhi**.

There are many roads connecting different cities.

You may travel directly if a road exists, or you may have to pass through several intermediate cities before reaching your destination.

For example,

you may travel like this.

```text
Ahmedabad

↓

Udaipur

↓

Jaipur

↓

Delhi
```

Here,

you started your journey from **Ahmedabad**,

passed through **Udaipur** and **Jaipur**,

and finally reached **Delhi**.

The complete sequence of cities that you visited during your journey represents the **Path** from Ahmedabad to Delhi.

The same idea is used in Graph Theory.

Instead of cities,

we use **Vertices (Nodes)**.

Instead of roads,

we use **Edges**.

Whenever we move from one vertex to another through connected edges,

we are following a **Path**.

---

## Definition

> A **Path** is a sequence of vertices connected by edges that allows us to travel from one vertex to another.

In simple words,

a path tells us

> **"Which route should we follow to reach the destination from the source?"**

---

## Example 1

Consider the following graph.

```text
A ----- B ----- C ----- D
```

Suppose we want to travel from **A** to **D**.

The journey will be

```text
A

↓

B

↓

C

↓

D
```

Therefore,

the path is

```text
A → B → C → D
```

Notice that,

every consecutive pair of vertices is connected by an edge.

- A is connected to B.
- B is connected to C.
- C is connected to D.

Since every movement is possible,

this is a **valid path**.

---

## Example 2

Now consider another graph.

```text
      A
     / \
    B   C
     \ /
      D
```

Suppose once again we want to travel from **A** to **D**.

Can we reach the destination in only one way?

The answer is **No**.

There are two possible routes.

First path

```text
A → B → D
```

Second path

```text
A → C → D
```

Both paths correctly connect the source vertex to the destination vertex.

This tells us something very important.

A graph may contain **multiple different paths** between the same pair of vertices.

Later,

Dijkstra's Algorithm will examine all these possibilities indirectly and determine **which path has the minimum total cost**.

---

## Can Every Graph Have Multiple Paths?

Not necessarily.

Some graphs contain only one possible path between two vertices,

while others may contain many different paths.

Example

```text
A ----- B ----- C
```

Here,

there is only one possible path from **A** to **C**.

```text
A → B → C
```

Now consider another graph.

```text
      A
     / \
    B   C
     \ /
      D
```

From **A** to **D**,

there are two different paths.

```text
A → B → D
```

and

```text
A → C → D
```

This shows that

the number of possible paths depends entirely on how the graph is connected.

---

## Important Observations

While studying paths,

remember the following points.

- A path connects one vertex to another.
- A path is formed by following connected edges.
- A graph may contain one path or many different paths between two vertices.
- Different paths may contain different numbers of edges.
- Different paths may also have different total costs.
- Dijkstra's Algorithm does **not** simply choose any path.
- It always tries to find the **shortest possible path**.

# 5. What is Path Cost?

After understanding what a **Path** is,

the next important question is,

> **How do we decide whether one path is better than another?**

The answer is very simple.

In a weighted graph,

every edge has a numerical value associated with it.

This numerical value is called the **Weight**.

The weight may represent

- Distance
- Time
- Fuel Consumption
- Construction Cost
- Cable Length
- Network Delay

Whenever we travel from one vertex to another,

we use several edges.

Since every edge has its own weight,

the total cost of travelling along that route is obtained by adding all those edge weights together.

This total value is called the **Path Cost**.

In simple words,

the Path Cost tells us

> **"How expensive is this complete journey?"**

---

## Definition

> The **Path Cost** is the sum of the weights of all the edges present in a path.

---

## Example

Consider the following weighted graph.

```text
          4

A ------------- B

 \

  \2

   \

    C

     \

      \3

       \

        D

B ------------- D

       6
```

Suppose we travel from **A** to **D** using the following path.

```text
A

↓

C

↓

D
```

The journey contains two edges.

```text
A → C = 2

C → D = 3
```

Therefore,

the Path Cost becomes

```text
2 + 3

=

5
```

So,

the total cost of travelling from **A** to **D** through **C** is

```text
5
```

Now,

let us consider another possible path.

```text
A

↓

B

↓

D
```

The edge weights are

```text
A → B = 4

B → D = 6
```

Therefore,

the Path Cost becomes

```text
4 + 6

=

10
```

Notice that, both paths start from the same source and reach the same destination, but their total costs are completely different.

This is exactly why we calculate the Path Cost before deciding which route is better.

Notice one very important point from the previous example.

Although both paths successfully reach the destination,

they do **not** require the same total cost.

One path requires only

```text
5
```

while the other requires

```text
10
```

This tells us that simply reaching the destination is **not enough**.

We should always try to reach the destination by spending the **minimum possible distance, cost, or time**.

This idea forms the foundation of almost every **Shortest Path Algorithm**, including **Dijkstra's Algorithm**.

---

## Comparison of Both Paths

| Path | Edge Weights | Total Path Cost |
|------|--------------|----------------:|
| A → C → D | 2 + 3 | **5** |
| A → B → D | 4 + 6 | **10** |

From the above table,

we can clearly observe that

```text
5 < 10
```

Therefore,

the path

```text
A → C → D
```

is considered **better** because it has a smaller total cost.

---

## Formula for Calculating Path Cost

Suppose a path contains several edges.

Each edge has its own weight.

The total Path Cost is simply obtained by adding the weights of all those edges.

Mathematically,

```text
Path Cost

=

Weight₁ + Weight₂ + Weight₃ + ... + Weightₙ
```

or

```text
Path Cost

=

Σ (Weight of every edge in the path)
```

where

- **Weight₁** represents the weight of the first edge.
- **Weight₂** represents the weight of the second edge.
- **Weight₃** represents the weight of the third edge.
- ...
- **Weightₙ** represents the weight of the last edge.

In simple words,

we simply add the weights of every edge that we travel through.

---

Clearly,

```text

```

Therefore,

the second path is more efficient.

---

## Does Fewer Edges Always Mean Less Cost?

One of the most common misconceptions among beginners is

> **A path having fewer edges is always the shortest path.**

This statement is **not true**.

The shortest path depends on the **total edge weight**,

not on the number of edges.

Let us understand this with an example.

```text
        100

A --------------- B

 \

  \1

   \

    C

     \

      \1

       \

        B
```

There are two possible paths from **A** to **B**.

### Path 1

```text
A → B
```

Total Cost

```text
100
```

Notice that,

this path contains only **one edge**.

---

### Path 2

```text
A

↓

C

↓

B
```

Total Cost

```text
1 + 1

=

2
```

Although this path contains **two edges**,

its total cost is much smaller.

Since

```text
2 < 100
```

the second path is actually the **better path**.

This example teaches us one very important lesson.

> **The shortest path is decided by the total weight of the edges, not by the number of edges.**

This is exactly why algorithms like **Dijkstra's Algorithm** calculate and compare **Path Costs** instead of simply counting edges.

---

## Important Observations

While studying Path Cost,

remember the following points.

- Every edge in a weighted graph has a numerical value called its **Weight**.
- The total Path Cost is obtained by adding the weights of all the edges in the path.
- Two different paths between the same pair of vertices may have different Path Costs.
- The path having fewer edges is **not necessarily** the cheaper path.
- Dijkstra's Algorithm always compares the **total Path Cost** before deciding which path is better.

# 6. What is the Shortest Path?

Now that we understand

- what a **Path** is, and
- how to calculate its **Path Cost**,

the next logical question is,

> **Among all the available paths, which one should we choose?**

The answer is very simple.

We should choose the path whose **total cost is the smallest**.

Such a path is called the **Shortest Path**.

In Graph Theory,

the word **Shortest** does **not always mean the smallest physical distance**.

Instead,

it simply means

> **the path having the minimum total edge weight.**

Depending on the problem,

the edge weight may represent

- Distance
- Time
- Fuel Consumption
- Money
- Delay
- Cable Length

Therefore,

the shortest path simply means

the **least expensive route** according to the given edge weights.

---

## Example

Consider the following weighted graph.

```text
          4

     A -------- B

      \

       \2

        \

         C

          \

           \3

            \

             D

B -------- D

      6
```

Suppose we want to travel from **A** to **D**.

Can we reach the destination in only one way?

No.

There are two possible paths.

---

### Path 1

```text
A

↓

B

↓

D
```

The total cost becomes

```text
4 + 6

=

10
```

---

### Path 2

```text
A

↓

C

↓

D
```

The total cost becomes

```text
2 + 3

=

5
```

Now,

let us compare both paths.

| Path | Total Cost |
|------|-----------:|
| A → B → D | 10 |
| A → C → D | **5** |

Clearly,

```text
5 < 10
```

Therefore,

the path

```text
A

↓

C

↓

D
```

is called the **Shortest Path** because its total cost is smaller than every other possible path connecting the same source and destination.

---

## Formal Definition

> A **Shortest Path** is the path between two vertices whose **total edge weight is minimum among all possible paths** connecting those vertices.

---

## Why Is Finding the Shortest Path Important?

In many real-world applications,

simply reaching the destination is not enough.

We also want to reduce

- Travel Distance
- Travel Time
- Fuel Consumption
- Transportation Cost
- Network Delay
- Resource Utilization

For example,

when you use **Google Maps**,

Instead,

Similarly,

Internet routers choose the fastest route for sending data packets,

food delivery applications choose the quickest delivery route,

and airlines optimize flight routes to reduce operational costs.

All these systems rely on the same fundamental idea—

**finding the shortest path**.

---

## Important Observations

While studying the Shortest Path concept,

remember the following points.

- A graph may contain many different paths between the same pair of vertices.
- Every path may have a different total cost.
- The shortest path is the one having the **minimum total edge weight**.
- The shortest path is **not necessarily** the path having the fewest edges.
- Dijkstra's Algorithm is designed to efficiently find the shortest paths in a graph containing **non-negative edge weights**.

# 7. What is the Single Source Shortest Path (SSSP) Problem?

After understanding the concept of a **Shortest Path**, let us move one step further.

Until now, we were finding the shortest path between only **two vertices**.

For example,

we found the shortest path from **A** to **D**.

But now, let us think about a slightly different situation.

Suppose you are standing in **Ahmedabad**.

Instead of finding the shortest route to only **Delhi**,

you now want to know the shortest route to

- Mumbai
- Pune
- Jaipur
- Delhi
- Chennai

In other words,

you are interested in finding the shortest route from **one starting location** to **every other location**.

This is exactly the problem solved by **Single Source Shortest Path (SSSP)** algorithms.

---

## Understanding the Meaning of Single Source

Before learning the formal definition,

let us first understand the meaning of the term itself.

The phrase **Single Source Shortest Path** consists of three different words.

### Single

The word **Single** means **only one**.

It indicates that there is only one starting point.

---

### Source

The word **Source** refers to the starting vertex from where our journey begins.

It is the vertex from which we want to calculate the shortest distance to every other vertex.

---

### Shortest Path

The phrase **Shortest Path** means the path having the minimum total edge weight.

Therefore,

combining all three terms,

**Single Source Shortest Path** means

> Finding the shortest path from one fixed starting vertex to every other vertex present in the graph.

---

## Definition

> The **Single Source Shortest Path (SSSP)** problem is the problem of finding the minimum distance from one source vertex to every other vertex in the graph.

Notice one important point.

The **source vertex never changes**.

Only the destination vertices change.

This is the key idea behind the SSSP problem.

---

## Example

Consider the following weighted graph.

```text
              A
           /     \
        4 /       \ 2
         /         \
        B -----3---- C
         \           /
        5 \         / 6
           \       /
              D
```

Suppose

```text
Source Vertex = A
```

Now,

our objective is **not** to find the shortest path only from **A** to **D**.

Instead,

we want to find

- Shortest path from **A** to **B**
- Shortest path from **A** to **C**
- Shortest path from **A** to **D**

In larger graphs,

this may include hundreds or even thousands of vertices.

Still,

the source remains the same.

Only the destinations change.

---

## What Should the Algorithm Produce?

Suppose the graph contains five vertices.

```text
Vertices

A

B

C

D

E
```

If the source vertex is

```text
A
```

then after executing the algorithm,

we should know

```text
Distance from A to A

Distance from A to B

Distance from A to C

Distance from A to D

Distance from A to E
```

Notice that,

the algorithm does not calculate the shortest path for only one destination.

Instead,

it computes the shortest distance from the source vertex to **every other vertex** in a single execution.

This is why Dijkstra's Algorithm is considered very efficient.

---

## Real-Life Analogy

Imagine Google Maps.

Suppose your current location is your home.

Now imagine that instead of searching for only one destination,

you want to know the shortest route from your home to every important place in your city.

For example,

- Hospital
- Railway Station
- Airport
- College
- Shopping Mall

Although the destinations are different,

your starting point remains exactly the same.

This is a perfect example of the **Single Source Shortest Path Problem**.

---

## Why Do We Need the SSSP Problem?

Many real-world applications require finding the shortest path from one location to many different locations.

For example,

a food delivery company may need to calculate the shortest distance from its restaurant to every customer's address.

Similarly,

an Internet router needs to determine the shortest path from itself to every other router in the network.

Instead of solving each destination separately,

algorithms like Dijkstra's Algorithm calculate all shortest distances together.

This saves both time and computational resources.

---

## Important Observations

While studying the Single Source Shortest Path problem,

remember the following points.

- There is only one source vertex.
- The source vertex remains fixed throughout the algorithm.
- The destination vertices may be different.
- The objective is to calculate the minimum distance from the source to every other vertex.
- Dijkstra's Algorithm is one of the most popular algorithms used to solve the Single Source Shortest Path problem when all edge weights are non-negative.

# 9. Why Can't BFS Solve Weighted Graphs?

One of the most common questions asked by beginners is,

> "If BFS can find the shortest path in a graph, then why do we need Dijkstra's Algorithm?"

This is a very important question.

To answer it,

we must first understand how **Breadth First Search (BFS)** works.

BFS always explores the graph level by level.

It assumes that every edge has exactly the same cost.

Because of this assumption,

BFS minimizes the **number of edges travelled**.

However,

it does **not** minimize the **total edge weight**.

This difference is extremely important.

Suppose we have the following weighted graph.

```text
          100

      A -------- B

       \

        \1

         \

          C

           \

            \1

             \

              B
```

Suppose we want to travel from **A** to **B**.

There are two possible paths.

### Path 1

```text
A → B
```

Number of edges

```text
1
```

Total Cost

```text
100
```

---

### Path 2

```text
A → C → B
```

Number of edges

```text
2
```

Total Cost

```text
1 + 1 = 2
```

Now think carefully.

Which path will BFS choose?

Since BFS minimizes the number of edges,

it immediately selects

```text
A → B
```

because it contains only one edge.

However,

is this actually the shortest path?

The answer is **No**.

The second path has a total cost of only

```text
2
```

which is much smaller than

```text
100
```

Therefore,

BFS fails for weighted graphs because it ignores edge weights completely.

It only counts the number of edges.

Dijkstra's Algorithm,

on the other hand, considers the weight of every edge and always tries to minimize the **total path cost**.

This is why Dijkstra's Algorithm is used for weighted graphs instead of BFS.

# 10. Introduction to Dijkstra's Algorithm

After understanding the Single Source Shortest Path problem,

the next question naturally becomes,

> "How can we efficiently find the shortest distance from one source vertex to every other vertex?"

The answer is **Dijkstra's Algorithm**.

Dijkstra's Algorithm is one of the most important shortest path algorithms in Graph Theory.

It was developed by the Dutch computer scientist **Edsger W. Dijkstra** in the year **1956**.

The algorithm is based on the **Greedy Technique**.

Instead of checking every possible path, it repeatedly chooses the vertex having the **smallest currently known distance** from the source.

Once that vertex is selected, the algorithm attempts to improve the distances of all its neighboring vertices.

This improvement process is called **Edge Relaxation**.

The algorithm continues repeating this process until every reachable vertex has obtained its shortest possible distance from the source.

Notice that,

Dijkstra's Algorithm never explores paths randomly.

Every decision is based on the currently smallest known distance, making it both efficient and elegant.

---

# 11. Why is Dijkstra's Algorithm Called a Greedy Algorithm?

Before learning how Dijkstra's Algorithm actually works,

there is one very important question that every student should understand.

> **Why is Dijkstra's Algorithm called a Greedy Algorithm?**

This question is frequently asked during technical interviews because understanding the answer helps us understand the complete intuition behind the algorithm.

To answer this question,

we must first understand what a **Greedy Algorithm** actually means.

---

## What is a Greedy Algorithm?

A **Greedy Algorithm** is an algorithm that solves a problem by making the **best possible choice at the current moment**.

Instead of looking at every possible future outcome,

it makes the decision that appears to be the best **right now**.

In simple words,

a Greedy Algorithm always thinks

> "At this moment, which choice looks the most beneficial?"

It immediately makes that choice and moves forward.

It never goes back to reconsider the previous decisions.

---

## Real-Life Example

Suppose you are participating in a race.

Your objective is to reach the finish line as quickly as possible.

Now imagine that at every junction,

you simply choose the road that currently looks shortest.

You do not stop to analyze every possible road in the city.

Instead,

you always choose the road that appears to be the best option at that particular moment.

This behavior is called a **Greedy Choice**.

Similarly,

Dijkstra's Algorithm also makes a greedy choice at every step.

---

## How Does Dijkstra Make a Greedy Choice?

Suppose we have the following graph.

```text
        A
      /   \
    4/     \2
    /       \
   B ------- C
      3
```

Initially,

our source vertex is

```text
A
```

At the beginning,

the known distances are

| Vertex | Distance |
|---------|---------:|
| A | 0 |
| B | 4 |
| C | 2 |

Now,

among all these vertices,

which vertex is currently the closest to the source?

Clearly,

```text
C
```

because

```text
2 < 4
```

Therefore,

instead of visiting **B** first,

Dijkstra immediately visits **C**.

Notice what happened.

The algorithm selected the vertex having the **smallest currently known distance**.

This is exactly the greedy decision.

---

## Why Is This Decision Correct?

At first,

many beginners think,

> "What if another shorter path is discovered later?"

This is an excellent question.

The answer lies in one important assumption.

**Dijkstra's Algorithm works only when all edge weights are non-negative.**

Since every edge weight is either positive or zero,

travelling further can never reduce an already finalized shortest distance.

In other words,

once a vertex becomes the closest vertex,

its shortest distance can never become smaller later.

Because of this property,

the greedy decision is always correct.

---

## The Greedy Rule Used by Dijkstra

The greedy rule followed by Dijkstra's Algorithm is very simple.

> Always choose the unvisited vertex having the minimum known distance from the source.

Notice that,

the algorithm does **not** choose the smallest edge.

It chooses the **vertex whose current distance from the source is minimum**.

Many beginners confuse these two ideas.

Remember,

Dijkstra is minimizing the **distance from the source**,

not the individual edge weights.

---

## Important Observations

While studying the greedy nature of Dijkstra's Algorithm,

remember the following points.

- Dijkstra is a Greedy Algorithm.
- It always chooses the vertex having the smallest currently known distance.
- Once a vertex is finalized, its shortest distance never changes.
- This property is valid only when all edge weights are non-negative.
- The greedy choice made by Dijkstra eventually leads to the correct shortest paths.

---

# 12. Basic Idea Behind Dijkstra's Algorithm

Now that we know Dijkstra's Algorithm is a Greedy Algorithm,

let us understand its overall working.

Many students directly memorize the code.

However,

doing so makes the algorithm very difficult to remember.

Instead,

if we understand the basic idea,

the code becomes extremely easy.

---

## The Main Objective

Suppose we are given a weighted graph.

Also suppose,

we are standing at one particular source vertex.

Our objective is

> To calculate the shortest distance from the source vertex to every other vertex present in the graph.

To achieve this,

Dijkstra follows a very systematic process.

It never guesses the shortest path.

Instead,

it gradually improves its knowledge about every vertex.

Initially,

the algorithm knows almost nothing.

As it keeps exploring the graph,

it keeps discovering shorter and shorter paths.

Eventually,

every vertex stores its actual shortest distance from the source.

---

## The Complete Thought Process

Imagine that you are standing at the source vertex.

Initially,

you only know one thing with complete certainty.

```text
The distance from the source to itself is always 0.
```

For every other vertex,

you currently have no idea how far it is.

Therefore,

you temporarily assume

```text
Infinity (INT_MAX)
```

This does **not** mean that the vertex is infinitely far away.

It simply means

> "I have not discovered any path to this vertex yet."

This idea is extremely important.

Many beginners misunderstand the purpose of `INT_MAX`.

It is merely a placeholder representing an unknown distance.

Later, as soon as the algorithm discovers a valid path, this value is updated.

---

## What Does the Algorithm Do Next?

Once the source distance becomes

```text
0
```

the algorithm performs the following steps repeatedly.

### Step 1

Select the vertex having the **smallest currently known distance**.

### Step 2

Visit all of its neighboring vertices.

### Step 3

Check whether travelling through the current vertex provides a shorter path to those neighbors.

### Step 4

If a shorter path is found, update the distance.

### Step 5

Repeat the same process until every reachable vertex has been processed.

Notice that, the algorithm never updates distances randomly.

Every update is based on mathematical comparison.

This comparison is called **Edge Relaxation**, which we will study in detail in the next section.

---

## Visual Overview

```text
Start

↓

Source Distance = 0

↓

Choose Minimum Distance Vertex

↓

Visit All Neighbors

↓

Try to Improve Their Distances

↓

Update Distance if Better Path Exists

↓

Repeat Until Priority Queue Becomes Empty

↓

All Shortest Distances Found
```

This flowchart represents the complete intuition behind Dijkstra's Algorithm.

Every line of code that we write later is simply an implementation of these steps.

---

## Why Is This Approach Efficient?

Suppose a graph contains

- Hundreds of vertices.
- Thousands of edges.

Checking every possible path would take an enormous amount of time.

Instead,

Dijkstra always expands the **most promising vertex first**.

Because of this, it avoids unnecessary computations and efficiently reaches the correct answer.

This intelligent strategy is one of the main reasons why Dijkstra's Algorithm is widely used in real-world applications.

#include <iostream>      // Used for input and output operations
#include <vector>        // Used to create dynamic arrays (vectors)
#include <queue>         // Used for Priority Queue (Min-Heap)
#include <climits>       // Used for INT_MAX
using namespace std;

//------------------------------------------------------------
// Function to implement Dijkstra's Algorithm
//------------------------------------------------------------
void dijkstra(int source, vector<vector<pair<int, int>>> &adj, int V)
{
    //--------------------------------------------------------
    // Step 1:
    // Create a distance vector.
    //
    // Purpose:
    // Stores the shortest known distance from the source
    // vertex to every other vertex.
    //
    // Initially,
    // we don't know the shortest distance to any vertex,
    // therefore every value is initialized with INT_MAX
    // which represents Infinity (Unknown Distance).
    //--------------------------------------------------------
    vector<int> dist(V, INT_MAX);

    //--------------------------------------------------------
    // Distance from source to itself is always zero because
    // no travelling is required to reach itself.
    //--------------------------------------------------------
    dist[source] = 0;

    //--------------------------------------------------------
    // Priority Queue (Min Heap)
    //
    // Stores:
    // (Distance, Vertex)
    //
    // Why?
    // We always want to process the vertex having the
    // smallest distance first.
    //
    // greater<pair<int,int>>
    // converts the default Max Heap into a Min Heap.
    //--------------------------------------------------------
    priority_queue<
        pair<int, int>,
        vector<pair<int, int>>,
        greater<pair<int, int>>
    > pq;

    //--------------------------------------------------------
    // Push the source vertex into the priority queue.
    //
    // Pair Format:
    // {distance, vertex}
    //
    // Source distance = 0
    //--------------------------------------------------------
    pq.push({0, source});

    //--------------------------------------------------------
    // Continue until every reachable vertex has been
    // processed.
    //--------------------------------------------------------
    while (!pq.empty())
    {
        //----------------------------------------------------
        // Get the vertex having the minimum distance.
        //----------------------------------------------------
        int currentDistance = pq.top().first;
        int u = pq.top().second;

        //----------------------------------------------------
        // Remove that vertex from the priority queue.
        //----------------------------------------------------
        pq.pop();

        //----------------------------------------------------
        // Sometimes the same vertex gets inserted multiple
        // times with different distances.
        //
        // Example:
        //
        // (10,2)
        // (7,2)
        //
        // Later we may even get
        //
        // (5,2)
        //
        // Only the smallest distance should be processed.
        //
        // If the popped distance is greater than the latest
        // shortest distance stored in dist[],
        // simply ignore it.
        //----------------------------------------------------
        if (currentDistance > dist[u])
            continue;

        //----------------------------------------------------
        // Visit every neighbour of the current vertex.
        //----------------------------------------------------
        for (auto neighbour : adj[u])
        {
            //------------------------------------------------
            // v = Adjacent Vertex
            // weight = Weight of edge (u,v)
            //------------------------------------------------
            int v = neighbour.first;
            int weight = neighbour.second;

            //------------------------------------------------
            // EDGE RELAXATION
            //
            // Check whether travelling through 'u'
            // gives a shorter path to vertex 'v'.
            //
            // Current Known Distance
            //        vs
            // Newly Calculated Distance
            //------------------------------------------------
            if (dist[v] > dist[u] + weight)
            {
                //--------------------------------------------
                // Update the shortest distance.
                //--------------------------------------------
                dist[v] = dist[u] + weight;

                //--------------------------------------------
                // Since we found a better distance,
                // insert the updated pair into the
                // priority queue.
                //
                // Pair Format
                // (Updated Distance, Vertex)
                //--------------------------------------------
                pq.push({dist[v], v});
            }
        }
    }

    //--------------------------------------------------------
    // Print the shortest distance from source to every
    // vertex.
    //--------------------------------------------------------
    cout << "Shortest Distance from Source Vertex "
         << source << ":\n\n";

    for (int i = 0; i < V; i++)
    {
        cout << "Vertex " << i
             << " --> ";

        if (dist[i] == INT_MAX)
            cout << "Not Reachable";
        else
            cout << dist[i];

        cout << endl;
    }
}

//------------------------------------------------------------

{
    //--------------------------------------------------------
    // Number of Vertices
    //--------------------------------------------------------
    int V = 5;

    //--------------------------------------------------------
    // Adjacency List
    //
    // Every vertex stores
    //
    // (Neighbour Vertex, Edge Weight)
    //--------------------------------------------------------
    vector<vector<pair<int, int>>> adj(V);

    //        2
    //   0 ------- 1
