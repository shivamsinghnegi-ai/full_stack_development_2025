## 1. React — Counter Component

**Difficulty:** Easy | **Time:** 15–20 min

### Task
Create a React component `<Counter />` that displays a number and has three buttons: **Increment** (+1), **Decrement** (−1), and **Reset** (back to 0). The count must never go below 0.

### Requirements
- State must be managed using `useState`
- The Decrement button must be **disabled** when count is `0`
- Reset button sets count back to `0`
- Component must work without any props
- Buttons must have clear labels: "+" , "−", "Reset"

### Starter Code
```jsx
function Counter() {
  // your code here
  return <div></div>;
}
```

### Expected Output
```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>{count}</h2>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)} disabled={count === 0}>−</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}
```

### What Interviewer Checks
- Do you know `useState`?
- Do you use `disabled={count === 0}` correctly?
- Do you avoid unnecessary state (no separate "isZero" state)?
- Is JSX clean and readable?

### Hints
- Use `disabled={count === 0}` on the Decrement button — do NOT use an if-check inside the handler.
- Keep it clean. Interviewer watches your component structure, not just whether it works.

### Follow-up Questions
- What if we need two counters that share the same state? How would you lift state up?
- How would you convert this to accept `initialValue` and `step` as props?

## 2. React — Searchable List with Live Filtering

**Difficulty:** Easy | **Time:** 20–25 min

### Task
Given an array of 10 user names (hardcoded or as a prop), render a text input and a list below it. As the user types, filter the list in real time to show only names that include the search string (case-insensitive). Show a "No results found" message when nothing matches.

### Requirements
- Input must be a **controlled component** using `useState`
- Filtering must be **case-insensitive**
- Show `"No results found"` when the filtered array is empty
- Filtering must happen on **every keystroke** — NOT on a button click
- Do NOT store the filtered list in a second `useState` — derive it directly

### Starter Code
```jsx
const users = ["Alice", "Bob", "Charlie", "David", "Eva",
               "Frank", "Grace", "Henry", "Iris", "Jake"];

function SearchList() {
  // your code here
}
```

### Expected Output
```jsx
function SearchList() {
  const [query, setQuery] = useState("");

  const filtered = users.filter(name =>
    name.toLowerCase().includes(query.toLowerCase())
  );

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Search..."
      />
      {filtered.length === 0 ? (
        <p>No results found</p>
      ) : (
        <ul>
          {filtered.map(name => <li key={name}>{name}</li>)}
        </ul>
      )}
    </div>
  );
}
```

### What Interviewer Checks
- Is the input controlled (`value` + `onChange`)?
- Is the filtered list derived from state (not stored in a second useState)?
- Is filtering case-insensitive?
- Is there a proper empty state message?
- Are list items rendered with a `key` prop?

### Hints
- Derive `filtered` directly inside the component body, not inside a `useEffect`. This is the most common mistake.
- Use `.toLowerCase()` on both the query and the name when comparing.

### Follow-up Questions
- How would you add debouncing so filtering only triggers 300ms after the user stops typing?
- How would you highlight the matching part of each name in the list?

## 3. React — Fetch and Display API Data

**Difficulty:** Medium | **Time:** 25–35 min

### Task
Build a component that fetches a list of posts from `https://jsonplaceholder.typicode.com/posts` on mount. Display them in a list showing each post's title and body. Show a loading indicator while fetching and an error message if the request fails.

### Requirements
- Use `useEffect` with an empty dependency array `[]` for the initial fetch
- Manage three separate states: `loading` (boolean), `data` (array), `error` (string or null)
- Show a loading message/spinner while fetching
- Show an error message if the fetch fails
- Each list item shows **post title** (bold/heading) and **post body**
- Limit display to first 10 posts to avoid an overloaded UI

### Starter Code
```jsx
function PostList() {
  // states: loading, data, error
  // useEffect for fetch

  return <div></div>;
}
```

### Expected Output
```jsx
function PostList() {
  const [data, setData] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    async function fetchPosts() {
      try {
        const res = await fetch("https://jsonplaceholder.typicode.com/posts");
        if (!res.ok) throw new Error("Failed to fetch");
        const json = await res.json();
        setData(json.slice(0, 10));
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    }
    fetchPosts();
  }, []);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <ul>
      {data.map(post => (
        <li key={post.id}>
          <strong>{post.title}</strong>
          <p>{post.body}</p>
        </li>
      ))}
    </ul>
  );
}
```

### What Interviewer Checks
- Is `useEffect` used correctly (not async itself)?
- Are all three states (loading, data, error) handled?
- Is `loading` set to false in both success and error cases?
- Are there proper guard returns for loading and error states?
- Is each list item keyed correctly?

### Hints
- Never make the `useEffect` callback itself async. Define `async function fetchPosts()` inside the effect and call it.
- Use a `finally` block to ensure `setLoading(false)` runs in both success and failure.
- Check `res.ok` before parsing JSON — `fetch` does NOT throw on 4xx/5xx responses.

### Follow-up Questions
- How would you add a Retry button to re-trigger the fetch?
- How would you use `AbortController` to cancel the in-flight request when the component unmounts?

## 4. React — Todo List App

**Difficulty:** Medium | **Time:** 30–40 min

### Task
Build a full Todo app. The user types a task in a text input, presses Enter or clicks "Add" to add it. Each todo has a checkbox — checking it marks it done (strikethrough text). A delete button removes the todo. All state updates must be immutable.

### Requirements
- Each todo must have a unique `id` (use `Date.now()` or `crypto.randomUUID()`)
- Toggling `done` must use `.map()` — never mutate the array directly
- Deleting a todo must use `.filter()`
- Completed todos show text with `text-decoration: line-through`
- Input clears after a task is added
- Pressing **Enter** in the input also adds the task (`onKeyDown`)
- Do not add empty todos — validate that `input.trim() !== ""`

### Starter Code
```jsx
function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState("");

  // addTodo, toggleTodo, deleteTodo functions here

  return <div></div>;
}
```

### Expected Output
```jsx
function TodoApp() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState("");

  function addTodo() {
    if (!input.trim()) return;
    setTodos([...todos, { id: Date.now(), title: input.trim(), done: false }]);
    setInput("");
  }

  function toggleTodo(id) {
    setTodos(todos.map(t => t.id === id ? { ...t, done: !t.done } : t));
  }

  function deleteTodo(id) {
    setTodos(todos.filter(t => t.id !== id));
  }

  return (
    <div>
      <input
        value={input}
        onChange={e => setInput(e.target.value)}
        onKeyDown={e => e.key === "Enter" && addTodo()}
        placeholder="Add a task..."
      />
      <button onClick={addTodo}>Add</button>
      <ul>
        {todos.map(todo => (
          <li key={todo.id} style={{ textDecoration: todo.done ? "line-through" : "none" }}>
            <input type="checkbox" checked={todo.done} onChange={() => toggleTodo(todo.id)} />
            {todo.title}
            <button onClick={() => deleteTodo(todo.id)}>Delete</button>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

### What Interviewer Checks
- Are immutable update patterns used (spread, map, filter)?
- Is `key` prop on each `<li>`?
- Is empty input validated before adding?
- Does Enter key work via `onKeyDown`?
- Does the input clear after adding?

### Hints
- The #1 mistake: `todo.done = true` — this mutates state directly and is an instant red flag.
- Always use `{ ...t, done: !t.done }` inside `.map()`.
- Use `todo.id` as key, never the array index.

### Follow-up Questions
- Add a filter bar: All / Active / Completed — how would you implement it without a new state?
- How would you persist todos to `localStorage` using `useEffect`?

## 5. React — Custom useFetch Hook

**Difficulty:** Hard | **Time:** 40–50 min

### Task
Write a reusable custom hook `useFetch(url)` that handles data fetching. It returns `{ data, loading, error }`. Then use this hook inside a `UserList` component to fetch and display users from JSONPlaceholder.

### Requirements
- Hook must **re-fetch if the URL changes** (put `url` in the dependency array)
- Use `AbortController` to cancel in-flight requests on cleanup
- Return `{ data: null, loading: true, error: null }` as initial state
- On success, set `data` to the response JSON and `loading` to false
- On error, set `error` to the error message string (not the full error object)
- Hook must be fully generic — it should work for any URL and any data type
- Ignore `AbortError` — do not set error state when the request is cancelled

### Starter Code
```jsx
function useFetch(url) {
  // return { data, loading, error }
}

function UserList() {
  const { data, loading, error } = useFetch("https://jsonplaceholder.typicode.com/users");
  // render users
}
```

### Expected Output
```jsx
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    async function fetchData() {
      setLoading(true);
      setError(null);
      try {
        const res = await fetch(url, { signal: controller.signal });
        if (!res.ok) throw new Error(`HTTP error: ${res.status}`);
        const json = await res.json();
        setData(json);
      } catch (err) {
        if (err.name !== "AbortError") {
          setError(err.message);
        }
      } finally {
        setLoading(false);
      }
    }

    fetchData();

    return () => controller.abort();
  }, [url]);

  return { data, loading, error };
}

function UserList() {
  const { data: users, loading, error } = useFetch("https://jsonplaceholder.typicode.com/users");

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name} — {user.email}</li>
      ))}
    </ul>
  );
}
```

### What Interviewer Checks
- Is the hook generic and reusable?
- Is `AbortController` used and returned from cleanup?
- Is `AbortError` ignored correctly?
- Is `url` in the dependency array?
- Are all three states reset properly when URL changes?

### Hints
- Return `() => controller.abort()` from inside `useEffect` — that is the cleanup function.
- Check `err.name !== "AbortError"` before setting error state, otherwise you'll show an error every time the component unmounts.
- Reset `loading` to `true` and `error` to `null` at the start of each fetch so stale state doesn't show.

### Follow-up Questions
- How would you add a simple cache (object or Map) so the same URL is not fetched twice?
- How would you modify the hook to accept request options (method, headers, body)?

Create an Express server with routes for a Todos resource. Use an in-memory array (no database). Implement `GET /todos` and `POST /todos`. Test using Postman or curl.

- `POST /todos` reads `req.body.title`, creates `{ id: Date.now(), title, done: false }`, pushes to array, returns **201** with the new todo

```js
  const newTodo = { id: Date.now(), title: title.trim(), done: false };
  todos.push(newTodo);
```

## 7. Node.js — Middleware: Auth Token Check

**Difficulty:** Medium | **Time:** 20–25 min

### Task
Write a custom Express middleware function `requireAuth` that checks for an `Authorization` header in the format `Bearer <token>`. If the token equals the string `"secret123"`, allow the request through. Otherwise, return a 401 response. Apply this middleware to a `GET /dashboard` route.

### Requirements
- Middleware must read `req.headers.authorization`
- If header is missing, return 401: `{ error: "No token provided" }`
- If header format is wrong (not `Bearer ...`), return 401: `{ error: "Invalid token format" }`
- If token does not match `"secret123"`, return 401: `{ error: "Unauthorized" }`
- If valid, call `next()` to proceed to the route handler
- `GET /dashboard` must return `{ message: "Welcome to dashboard" }` when authenticated

### Starter Code
```js
const express = require("express");
const app = express();

function requireAuth(req, res, next) {
  // your middleware logic
}

app.get("/dashboard", requireAuth, (req, res) => {
  // protected route
});

app.listen(3000);
```

### Expected Output
```js
function requireAuth(req, res, next) {
  const authHeader = req.headers.authorization;

  if (!authHeader) {
    return res.status(401).json({ error: "No token provided" });
  }

  const parts = authHeader.split(" ");
  if (parts.length !== 2 || parts[0] !== "Bearer") {
    return res.status(401).json({ error: "Invalid token format" });
  }

  const token = parts[1];
  if (token !== "secret123") {
    return res.status(401).json({ error: "Unauthorized" });
  }

  next();
}

app.get("/dashboard", requireAuth, (req, res) => {
  res.json({ message: "Welcome to dashboard" });
});
```

### What Interviewer Checks
- Is `next()` called correctly when auth passes?
- Are all early-return cases handled with `return`?
- Is the Bearer token extracted from the header correctly?
- Are correct 401 responses returned?

### Hints
- Middleware MUST call `next()` when validation passes — forgetting it hangs the request forever.
- `return res.status(401).json(...)` — always `return` inside middleware error paths.
- In real apps, this token would be a JWT. Interviewers may ask you to extend this with `jsonwebtoken`.

### Follow-up Questions
- How would you replace the hardcoded token with JWT verification using `jsonwebtoken`?
- How would you apply this middleware globally to all routes instead of per-route?

```js
const productSchema = new mongoose.Schema({
  name: { type: String, required: true },
  price: { type: Number, required: true },

const Product = mongoose.model("Product", productSchema);
```

## 9. Node.js — User Registration with Bcrypt

**Difficulty:** Hard | **Time:** 40–50 min

### Task
Build a `POST /register` route that accepts `{ name, email, password }`, validates the input, hashes the password using bcrypt, saves the user to MongoDB, and returns the user without the password field. Also build `POST /login` that verifies the password and returns a JWT.

### Requirements
- Validate that `name`, `email`, and `password` are all present; return 400 if not
- Check if email already exists — return 409 (Conflict) if it does
- Hash the password with `bcrypt.hash(password, 10)` before saving
- Never return the `password` field in the response — use `.select("-password")`
- For login: use `bcrypt.compare(plaintext, hash)` to verify
- On successful login, return a JWT with payload `{ userId: user._id }` signed with a secret key
- JWT should expire in `"7d"`

### Starter Code
```js
const express = require("express");
const mongoose = require("mongoose");
const bcrypt = require("bcrypt");
const jwt = require("jsonwebtoken");

const JWT_SECRET = "your_jwt_secret";

const userSchema = new mongoose.Schema({
  // define here
});

const User = mongoose.model("User", userSchema);
const app = express();
app.use(express.json());

app.post("/register", async (req, res) => { });
app.post("/login", async (req, res) => { });

app.listen(3000);
```

### Expected Output
```js
const userSchema = new mongoose.Schema({
  name: { type: String, required: true },
  email: { type: String, required: true, unique: true },
  password: { type: String, required: true }
});

app.post("/register", async (req, res) => {
  try {
    const { name, email, password } = req.body;
    if (!name || !email || !password) {
      return res.status(400).json({ error: "All fields are required" });
    }
    const existing = await User.findOne({ email });
    if (existing) return res.status(409).json({ error: "Email already registered" });

    const hashedPassword = await bcrypt.hash(password, 10);
    const user = await User.create({ name, email, password: hashedPassword });
    const userResponse = await User.findById(user._id).select("-password");
    res.status(201).json(userResponse);
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

app.post("/login", async (req, res) => {
  try {
    const { email, password } = req.body;
    if (!email || !password) return res.status(400).json({ error: "Fields required" });

    const user = await User.findOne({ email });
    if (!user) return res.status(401).json({ error: "Invalid credentials" });

    const isMatch = await bcrypt.compare(password, user.password);
    if (!isMatch) return res.status(401).json({ error: "Invalid credentials" });

    const token = jwt.sign({ userId: user._id }, JWT_SECRET, { expiresIn: "7d" });
    res.json({ token });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});
```

### What Interviewer Checks
- Is `bcrypt.hash` used (not bcryptjs without justification)?
- Is the password NEVER sent back in the response?
- Is the duplicate email check done before hashing?
- Is `bcrypt.compare` used (not manual comparison)?
- Does JWT include `expiresIn`?

### Hints
- For login, always return the same error `"Invalid credentials"` whether the email is not found OR the password is wrong — this prevents user enumeration attacks.
- Salt rounds of 10 is standard. Do not use 1 or 100.
- Store `JWT_SECRET` in an environment variable (`.env`) — interviewers notice this.

### Follow-up Questions
- How would you protect a route using the JWT returned from login?
- How would you implement token refresh with a refresh token?

## 14. Full Stack — Login Form with JWT Auth

```jsx
// Frontend
function LoginForm() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");
  const [token, setToken] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(false);

  async function handleSubmit(e) {
    e.preventDefault();
    setLoading(true);
    setError(null);
    try {
      const res = await fetch("/api/login", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ email, password })
      });
      const data = await res.json();
      if (!res.ok) throw new Error(data.error);
      localStorage.setItem("token", data.token);
      setToken(data.token);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }

  if (token) return <p>Logged in as {email}</p>;

  return (
    <form onSubmit={handleSubmit}>
      <input type="email" value={email} onChange={e => setEmail(e.target.value)} placeholder="Email" />
      <input type="password" value={password} onChange={e => setPassword(e.target.value)} placeholder="Password" />
      <button type="submit" disabled={loading}>{loading ? "Logging in..." : "Login"}</button>
      {error && <p style={{ color: "red" }}>{error}</p>}
    </form>
  );
}
```

- Is JWT stored in `localStorage`?
- Is `disabled={loading}` on the submit button?

### Hints
- Return the same "Invalid credentials" for both "user not found" and "wrong password" — different messages let attackers know which emails are registered.
- `JWT_SECRET` must be in an environment variable, not hardcoded. Interviewers specifically look for this.
- `fetch` does NOT throw on 4xx responses — check `res.ok` and manually throw the error.

### Follow-up Questions

## Quick Reference — What Interviewers Watch

| Topic | Common Mistakes | What to Do Instead |
|---|---|---|
| useState update | `todos[0].done = true` (mutation) | `setTodos(todos.map(...))` |
| useEffect async | `useEffect(async () => ...)` | Define async fn inside, then call it |
| fetch error handling | Not checking `res.ok` | `if (!res.ok) throw new Error(...)` |
| Express middleware | Forgetting `return` before `res.status()` | Always `return res.status(400).json(...)` |
| express.json() | Missing middleware, `req.body` is undefined | Add `app.use(express.json())` before routes |
| Mongoose update | `findByIdAndUpdate` without `{ new: true }` | Pass `{ new: true }` to get updated doc |
| JWT secret | Hardcoded string in source | Use `process.env.JWT_SECRET` |
| getStaticPaths params | Passing number IDs | Convert to string: `String(p.id)` |
| Next.js middleware | Passing relative URL to redirect | `new URL("/login", request.url)` |
| Password comparison | Manual string compare | Always use `bcrypt.compare()` |
