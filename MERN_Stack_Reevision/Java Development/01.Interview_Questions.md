# Java + Spring Boot Interview Questions

## 1. Core Java — Basics

---

### Q1. What is Java? What are its main features?

**Answer:**

Java is a high-level, object-oriented programming language developed by Sun Microsystems (now Oracle). It follows the principle **"Write Once, Run Anywhere"** — meaning you write the code once and it can run on any platform that has a **JVM (Java Virtual Machine)** installed.

**Key Features of Java:**

| Feature | What it means |
|---|---|
| **Platform Independent** | Java code compiles to *bytecode*, which runs on any OS via JVM |
| **Object-Oriented** | Everything is modeled as objects (classes, instances) |
| **Secure** | No pointer access; strong memory management |
| **Multithreading** | Multiple tasks can run simultaneously |
| **Robust** | Strong exception handling; garbage collection prevents memory leaks |
| **Simple** | Easy syntax, similar to C++ but cleaner |

**How to say it in interview:**
> "Java is a platform-independent, object-oriented programming language. The Java compiler converts the source code into bytecode, which the JVM then interprets and runs on any operating system. This makes Java highly portable."

---

### Q2. What is the difference between JDK, JRE, and JVM?

**Answer:**

This is one of the **most frequently asked** basic questions. Here's the simple breakdown:

```
JDK (Java Development Kit)
 JRE (Java Runtime Environment)
 JVM (Java Virtual Machine)
```

- **JVM (Java Virtual Machine):** It is an engine that executes the Java *bytecode*. It is platform-specific (different JVM for Windows, Linux, Mac) but the bytecode it runs is the same everywhere. JVM also handles memory management and garbage collection.

- **JRE (Java Runtime Environment):** JRE = JVM + Library Classes. It provides the environment needed to *run* a Java program. If you only want to run a Java app, you install JRE.

- **JDK (Java Development Kit):** JDK = JRE + Development Tools (compiler `javac`, debugger, etc.). If you want to *develop/write* Java programs, you need JDK.

**Interview Answer:**
> "JVM is the engine that runs bytecode. JRE includes JVM plus standard libraries needed to run Java programs. JDK includes JRE plus development tools like the compiler. So as a developer, I install JDK on my machine."

### Q3. What is the difference between `==` and `.equals()` in Java?

**Answer:**

This is a **very common trick question** in interviews!

- **`==`** (double equals) checks **reference equality** — it checks if two variables point to the *same object in memory*.
- **`.equals()`** checks **content/value equality** — it checks if two objects have the *same content*.

**Example:**
```java
String a = new String("Hello");
String b = new String("Hello");

System.out.println(a == b); // false → different objects in memory
System.out.println(a.equals(b)); // true → same content
```

**Why does this matter?**
> For primitive types (`int`, `char`, etc.), `==` compares values directly. But for objects like `String`, always use `.equals()` for content comparison.

**Interview Answer:**
> "`==` checks if two references point to the same memory location — it's comparing addresses. `.equals()` checks if the actual data inside two objects is the same. For Strings and objects, we should always use `.equals()` to compare values."

### Q4. What is the difference between `String`, `StringBuilder`, and `StringBuffer`?

**Answer:**

| Class | Mutable? | Thread-Safe? | Performance | Use When |
|---|---|---|---|---|
| `String` | No (Immutable) | Yes | Slow (creates new objects) | Few/no modifications |
| `StringBuilder` | Yes | No | Fast | Single-threaded, many changes |
| `StringBuffer` | Yes | Yes | Medium | Multi-threaded environment |

**Simple Explanation:**
- `String` is **immutable** — once created, it cannot be changed. Every time you do `str + "world"`, Java creates a brand new String object.
- `StringBuilder` is **mutable** and **not synchronized** — best for building strings in a loop when thread safety is not needed.
- `StringBuffer` is like StringBuilder but **thread-safe** (synchronized) — useful in multi-threaded programs.

**Interview Answer:**
> "String is immutable; once created, it can't be changed. StringBuilder is mutable and faster but not thread-safe. StringBuffer is mutable and thread-safe but slower than StringBuilder due to synchronization overhead. For most single-threaded string manipulation, we prefer StringBuilder."

### Q5. What is the difference between `static` and `non-static` methods?

**Answer:**

- A **static method** belongs to the **class itself** and can be called without creating an object.
- A **non-static method** belongs to an **instance (object)** of the class and needs an object to be called.

```java
class Calculator {
 // Static method - no object needed
 public static int add(int a, int b) {
 return a + b;
 }

 // Non-static method - needs an object
 public int multiply(int a, int b) {
 return a * b;
 }
}

// Usage
int sum = Calculator.add(5, 3); // Called on class directly

Calculator calc = new Calculator();
int product = calc.multiply(5, 3); // Called on object
```

**Interview Answer:**
> "Static methods belong to the class and can be accessed directly using the class name without creating an object. Non-static methods belong to instances and require an object to be called. `main()` is a classic example of a static method."

## 2. OOP Concepts

---

### Q6. What are the 4 pillars of OOP?

**Answer:**

OOP stands for **Object-Oriented Programming**. It is built on 4 core principles:

---

#### 1. Encapsulation — "Wrapping data and methods together + hiding internal details"
Binding the data (variables) and methods (functions) into a single unit called a **class**, and restricting direct access to data using access modifiers (`private`).

```java
class BankAccount {
 private double balance; // Hidden from outside

 public double getBalance() { return balance; } // Controlled access
 public void deposit(double amount) { balance += amount; }
}
```

---

#### 2. Inheritance — "Child class inherits from parent class"
A child class can reuse properties and methods of a parent class using the `extends` keyword.

```java
class Animal {
 public void eat() { System.out.println("Eating..."); }
}

class Dog extends Animal {
 public void bark() { System.out.println("Barking..."); }
}

Dog d = new Dog();
d.eat(); // Inherited from Animal
d.bark(); // Dog's own method
```

---

#### 3. Polymorphism — "One name, many forms"
A method can behave differently based on the object. Two types:
- **Compile-time (Method Overloading):** Same method name, different parameters.
- **Runtime (Method Overriding):** Child class redefines parent's method.

```java
// Overloading (compile-time polymorphism)
class MathUtil {
 int add(int a, int b) { return a + b; }
 double add(double a, double b) { return a + b; } // Same name, different params
}

// Overriding (runtime polymorphism)
class Animal {
 public void sound() { System.out.println("Some sound"); }
}
class Cat extends Animal {
 @Override
 public void sound() { System.out.println("Meow"); } // Overrides parent
}
```

---

#### 4. Abstraction — "Show only what's necessary, hide the complexity"
Hiding implementation details and showing only the essential features. Achieved via **abstract classes** or **interfaces**.

```java
abstract class Shape {
 abstract double area(); // No implementation, just declaration
}

class Circle extends Shape {
 double radius;
 Circle(double r) { this.radius = r; }

 @Override
 double area() { return Math.PI * radius * radius; } // Implemented here
}
```

**Interview Answer:**
> "The 4 pillars are: Encapsulation — bundling data and methods while hiding internal state; Inheritance — child classes reusing parent class code; Polymorphism — same method name behaving differently in different contexts; and Abstraction — exposing only what's needed and hiding complexity."

---

### Q7. What is the difference between Abstract Class and Interface?

**Answer:**

This is **asked in almost every Java interview**. Know this cold.

| Feature | Abstract Class | Interface |
|---|---|---|
| Methods | Can have both abstract & concrete methods | Only abstract methods (Java 8+ allows default & static methods) |
| Variables | Can have instance variables | Only `public static final` constants |
| Constructor | Can have constructors | Cannot have constructors |
| Inheritance | `extends` (single class only) | `implements` (multiple interfaces allowed) |
| Access Modifiers | Any (private, protected, public) | Methods are `public` by default |
| Use Case | Partial implementation + shared state | Full contract / multiple behaviors |

**Simple Real-World Analogy:**
> Think of an **interface** as a *job contract* — "You must implement these functions." Think of an **abstract class** as a *base template* — "Here's some code you can use + some you must implement yourself."

```java
// Abstract Class
abstract class Vehicle {
 String brand = "Generic";
 abstract void start();
 void stop() { System.out.println("Stopping..."); } // Concrete method
}

// Interface
interface Flyable {
 void fly(); // Must be implemented
}

class FlyingCar extends Vehicle implements Flyable {
 @Override
 public void start() { System.out.println("Starting flying car"); }

 @Override
 public void fly() { System.out.println("Flying!"); }
}
```

**Interview Answer:**
> "An abstract class can have both abstract and concrete methods and can hold state through instance variables. An interface defines a pure contract — all methods must be implemented by the class. A class can only extend one abstract class but can implement multiple interfaces. We use an abstract class when classes share common code, and an interface when we want to define a capability."

---

### Q8. What is method overloading vs method overriding?

**Answer:**

| Feature | Overloading | Overriding |
|---|---|---|
| Definition | Same method name, different parameters | Child redefines parent's method |
| Binding | Compile-time (Static) | Runtime (Dynamic) |
| Return type | Can differ | Must be same (or covariant) |
| Access modifier | Any | Cannot be more restrictive |
| `@Override` annotation | Not needed | Recommended to use |

**Interview Answer:**
> "Overloading is when multiple methods in the same class share the same name but have different parameters — Java decides which one to call at compile time. Overriding is when a subclass provides its own implementation of a method defined in the parent class — Java decides which one to run at runtime based on the actual object type."

## 3. Java Collections

### Q9. What is the Java Collections Framework?

**Answer:**

The **Java Collections Framework (JCF)** is a set of classes and interfaces that provide ready-made data structures to store and manipulate groups of objects.

**Main Interfaces:**

```
Collection
 List → Ordered, allows duplicates → ArrayList, LinkedList
 Set → Unordered, NO duplicates → HashSet, TreeSet
 Queue → FIFO ordering → LinkedList, PriorityQueue

Map (separate hierarchy)
 Key-Value pairs → HashMap, TreeMap, LinkedHashMap
```

**Interview Answer:**
> "The Java Collections Framework provides classes and interfaces to store groups of objects efficiently. The main types are List (ordered, allows duplicates), Set (unordered, no duplicates), Queue (FIFO), and Map (key-value pairs). Popular implementations include ArrayList, HashSet, and HashMap."

### Q10. What is the difference between `ArrayList` and `LinkedList`?

**Answer:**

| Feature | ArrayList | LinkedList |
|---|---|---|
| Internal Structure | Dynamic array | Doubly linked list |
| Access by index | Fast — O(1) | Slow — O(n) |
| Insert/Delete (middle) | Slow — O(n) (shifting) | Fast — O(1) |
| Memory | Less (just data) | More (stores next/prev pointers) |
| Best for | Read-heavy operations | Insert/delete-heavy operations |

**Interview Answer:**
> "ArrayList uses a dynamic array internally, so accessing elements by index is very fast. But inserting or deleting in the middle is slow because elements need to be shifted. LinkedList uses a doubly linked list, so insertion and deletion are fast, but random access is slow. For most read-heavy use cases, ArrayList is preferred."

---

### Q11. What is the difference between `HashMap` and `Hashtable`?

**Answer:**

| Feature | HashMap | Hashtable |
|---|---|---|
| Thread Safety | Not synchronized | Synchronized |
| Null keys/values | Allows one null key, multiple null values | No null allowed |
| Performance | Faster | Slower (due to sync) |
| Legacy | No (modern) | Yes (old class) |

**Interview Answer:**
> "HashMap is not thread-safe but is faster and allows one null key. Hashtable is thread-safe (synchronized) but slower and doesn't allow null keys. In modern Java, if we need thread safety, we use `ConcurrentHashMap` instead of Hashtable."

## 4. Exception Handling

---

### Q12. What is exception handling in Java?

**Answer:**

Exception handling is a mechanism to **gracefully handle runtime errors** so the program doesn't crash abruptly. Java uses the `try-catch-finally` block.

**Types of Exceptions:**
- **Checked Exceptions:** Must be handled at compile time (e.g., `IOException`, `SQLException`)
- **Unchecked Exceptions (RuntimeExceptions):** Not required to be caught (e.g., `NullPointerException`, `ArrayIndexOutOfBoundsException`)
- **Error:** Serious problems like `OutOfMemoryError` — not meant to be caught by application code

```java
try {
 int result = 10 / 0; // ArithmeticException will be thrown
} catch (ArithmeticException e) {
 System.out.println("Error: " + e.getMessage()); // Handled gracefully
} finally {
 System.out.println("This always runs!"); // Cleanup code
}
```

**Interview Answer:**
> "Exception handling in Java allows us to catch and handle runtime errors without crashing the application. We use try-catch blocks to handle exceptions. Checked exceptions must be declared or handled at compile time, while unchecked exceptions occur at runtime. The `finally` block always executes and is used for cleanup tasks like closing connections."

---

### Q13. What is the difference between `throw` and `throws`?

**Answer:**

- **`throw`** — Used inside a method to *actually throw* an exception object.
- **`throws`** — Used in a method signature to *declare* that the method might throw an exception.

```java
// 'throws' declares that method may throw this exception
public void readFile(String path) throws IOException {
 if (path == null) {
 throw new IOException("Path cannot be null"); // 'throw' actually throws it
 }
}
```

**Interview Answer:**
> "`throw` is used to explicitly throw an exception from a specific point in the code. `throws` is used in the method declaration to indicate that the method can potentially throw one or more checked exceptions, and the caller must handle them."

## 5. Java 8 Features

---

### Q14. What are the main features introduced in Java 8?

**Answer:**

Java 8 was a **revolutionary update** to the language. The key features are:

1. **Lambda Expressions** — Write anonymous functions concisely
2. **Stream API** — Process collections of data in a functional style
3. **Functional Interfaces** — Interfaces with a single abstract method
4. **Default and Static methods in Interfaces** — Add method bodies to interfaces
5. **Optional class** — Avoid NullPointerExceptions
6. **New Date/Time API** — `LocalDate`, `LocalDateTime`, `ZonedDateTime`

**Lambda Expression Example:**
```java
// Before Java 8 (Verbose)
List<String> names = Arrays.asList("Alice", "Bob", "Charlie");
Collections.sort(names, new Comparator<String>() {
 public int compare(String a, String b) { return a.compareTo(b); }
});

// Java 8 Lambda (Clean!)
names.sort((a, b) -> a.compareTo(b));

// Even cleaner with method reference
names.sort(String::compareTo);
```

**Stream API Example:**
```java
List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5, 6);

List<Integer> evenNumbers = numbers.stream()
 .filter(n -> n % 2 == 0) // Keep only even
 .collect(Collectors.toList()); // Collect result

// Result: [2, 4, 6]
```

**Interview Answer:**
> "Java 8 introduced Lambda expressions for writing concise anonymous functions, the Stream API for functional-style data processing, Optional for handling nulls gracefully, default and static methods in interfaces, and a new Date-Time API. These features significantly improved code readability and productivity."

## 6. Spring Framework Basics

---

### Q15. What is the Spring Framework?

**Answer:**

Spring is a **powerful, lightweight, open-source Java framework** designed to simplify enterprise-level Java application development. The core idea of Spring is **Dependency Injection (DI)** and **Inversion of Control (IoC)**.

**Key Modules of Spring:**
- **Spring Core** — IoC Container, Dependency Injection
- **Spring MVC** — Web applications (Model-View-Controller)
- **Spring Data** — Database access
- **Spring Security** — Authentication & Authorization
- **Spring AOP** — Aspect-Oriented Programming (logging, transactions)

**Interview Answer:**
> "Spring Framework is an open-source Java framework that simplifies enterprise application development. Its core feature is Dependency Injection and IoC Container, which manage object creation and dependencies. Spring has many modules like Spring MVC for web apps, Spring Data for database access, and Spring Security for authentication."

---

### Q16. What is Dependency Injection (DI) and Inversion of Control (IoC)?

**Answer:**

These two concepts go hand-in-hand. Let's break them down simply:

**Inversion of Control (IoC):**
> Normally, you create objects yourself. With IoC, you *hand over control* to the Spring framework to create and manage objects.

**Dependency Injection (DI):**
> Instead of a class creating its own dependencies, the Spring framework *injects* them automatically.

**Without DI (tight coupling — bad practice):**
```java
class OrderService {
 private PaymentService paymentService;

 public OrderService() {
 this.paymentService = new PaymentService(); // Creating dependency yourself
 }
}
```

**With DI (loose coupling — good practice):**
```java
@Service
class OrderService {
 private final PaymentService paymentService;

 @Autowired // Spring injects this automatically!
 public OrderService(PaymentService paymentService) {
 this.paymentService = paymentService;
 }
}
```

**Interview Answer:**
> "IoC means we give control of object creation to the Spring container instead of doing it ourselves. Dependency Injection is how IoC is implemented — Spring automatically injects the dependencies a class needs. This promotes loose coupling, making code easier to test and maintain."

---

### Q17. What is a Spring Bean?

**Answer:**

A **Spring Bean** is simply a Java object that is **created, configured, and managed by the Spring IoC container**. Any class you register with Spring (via annotations or XML) becomes a Spring Bean.

**How to create a Spring Bean:**
```java
@Component // Marks this class as a Spring Bean
public class EmailService {
 public void sendEmail(String to, String message) {
 System.out.println("Sending email to: " + to);
 }
}
```

**Common annotations to declare Beans:**
- `@Component` — Generic bean
- `@Service` — Business logic layer
- `@Repository` — Data access layer
- `@Controller` / `@RestController` — Web layer

**Interview Answer:**
> "A Spring Bean is any Java object managed by the Spring IoC container. We declare beans using annotations like @Component, @Service, or @Repository. Spring handles their creation, wiring, and lifecycle. The main advantage is that we don't need to manually instantiate objects — Spring does it for us."

## 7. Spring Boot Core

---

### Q18. What is Spring Boot? How is it different from Spring Framework?

**Answer:**

**Spring Boot** is built on top of the Spring Framework. It was created to **eliminate boilerplate configuration** and make it easier to build production-ready applications quickly.

**Spring vs Spring Boot:**

| Feature | Spring | Spring Boot |
|---|---|---|
| Configuration | Manual XML or Java config | Auto-configuration (minimal setup) |
| Server | External server (Tomcat, etc.) | **Embedded server** (Tomcat built-in) |
| Deployment | WAR file required | Runs as a standalone JAR file |
| Setup time | High | Very Low |
| Dependency management | Manual | **Starter dependencies** handle it |
| Production monitoring | Manual setup | **Actuator** built-in |

**Interview Answer:**
> "Spring Boot is an extension of the Spring Framework that eliminates most of the boilerplate configuration. The biggest advantages are auto-configuration, embedded servers like Tomcat so we don't need to deploy WAR files separately, and starter dependencies that bundle related libraries together. While Spring requires a lot of setup, Spring Boot gets an application running in minutes."

---

### Q19. What is `@SpringBootApplication`? What does it do?

**Answer:**

`@SpringBootApplication` is a **combination of 3 annotations** in one:

```java
@SpringBootApplication
=
@Configuration // Marks it as a configuration class
+ @EnableAutoConfiguration // Enables Spring Boot's auto-configuration
+ @ComponentScan // Scans the current package for Spring components
```

It is placed on the **main class** of the Spring Boot application and tells Spring Boot to:
1. Load this class as a configuration class
2. Automatically configure the application based on dependencies in the classpath
3. Scan all classes in the package for Spring beans

**Example:**
```java
@SpringBootApplication
public class MyApplication {
 public static void main(String[] args) {
 SpringApplication.run(MyApplication.class, args); // Starts the application
 }
}
```

**Interview Answer:**
> "`@SpringBootApplication` is a convenience annotation that combines `@Configuration`, `@EnableAutoConfiguration`, and `@ComponentScan`. It's placed on the main class. When the application starts, it triggers auto-configuration, component scanning, and Spring context initialization — all with a single annotation."

---

### Q20. What is Auto-Configuration in Spring Boot?

**Answer:**

**Auto-configuration** is one of Spring Boot's most powerful features. It automatically configures your application based on the **libraries present in your classpath**.

**How it works:**
> If you add `spring-boot-starter-data-jpa` to your project, Spring Boot automatically:
> - Configures a DataSource
> - Sets up Hibernate (the JPA provider)
> - Creates a TransactionManager
> - All without you writing a single line of config code!

**Example:**
```
# Just add this in application.properties
spring.datasource.url=jdbc:mysql://localhost:3306/mydb
spring.datasource.username=root
spring.datasource.password=password

# Spring Boot auto-configures everything else!
```

**Interview Answer:**
> "Auto-configuration in Spring Boot automatically sets up beans and configuration based on the dependencies you've added. For example, if you include the web starter, Spring Boot auto-configures an embedded Tomcat server, DispatcherServlet, and error handling. You can override any auto-configured bean by providing your own configuration. This is what makes Spring Boot so much faster to set up than plain Spring."

**Interview Answer:**

### Q22. What is `application.properties` / `application.yml`?

**Answer:**

These files are used to **externalize configuration** in a Spring Boot application. Instead of hardcoding values in the code, you define them here.

**application.properties:**
```properties
# Server configuration
server.port=8081

# Database configuration
spring.datasource.url=jdbc:mysql://localhost:3306/school_db
spring.datasource.username=root
spring.datasource.password=secret123

# JPA configuration
spring.jpa.show-sql=true
spring.jpa.hibernate.ddl-auto=update

app.name=My Spring Boot App
app.version=1.0
```

**Accessing custom properties in code:**
```java
@RestController
public class AppController {

 @Value("${app.name}") // Injects value from properties file
 private String appName;

 @GetMapping("/info")
 public String getInfo() {
 return "App Name: " + appName;
 }
}
```

**Interview Answer:**
> "`application.properties` or `application.yml` is where we externalize configuration — like server port, database URL, passwords, and custom settings. This keeps configuration separate from code. We can access these values in our beans using the `@Value` annotation or `@ConfigurationProperties`. We can also have environment-specific files like `application-dev.properties` and `application-prod.properties`."

## 8. Spring Boot REST API

---

### Q24. What is a REST API? What are HTTP methods?

**Answer:**

**REST (Representational State Transfer)** is an architectural style for designing web APIs. A REST API uses **HTTP methods** to perform **CRUD operations** on resources.

**HTTP Methods Mapping to CRUD:**

| HTTP Method | CRUD Operation | Example URL | What it does |
|---|---|---|---|
| **GET** | Read | `GET /students` | Get all students |
| **POST** | Create | `POST /students` | Create a new student |
| **PUT** | Update (Full) | `PUT /students/1` | Update student with ID 1 |
| **PATCH** | Update (Partial) | `PATCH /students/1` | Update specific fields |
| **DELETE** | Delete | `DELETE /students/1` | Delete student with ID 1 |

**Interview Answer:**
> "REST is an architectural style for APIs that uses standard HTTP methods. GET retrieves data, POST creates new records, PUT updates existing records, and DELETE removes them. REST APIs are stateless — each request contains all information needed to process it, and no session is stored on the server."

---

### Q25. What is `@RestController` and how is it different from `@Controller`?

**Answer:**

- **`@Controller`** — Used for traditional MVC applications that return **views (HTML pages)**.
- **`@RestController`** — Used for REST APIs that return **data (JSON/XML)** directly.

`@RestController` = `@Controller` + `@ResponseBody`

```java
// @Controller — Returns a view name (HTML template)
@Controller
public class WebController {
 @GetMapping("/home")
 public String homePage() {
 return "home"; // Returns 'home.html' template
 }
}

// @RestController — Returns data directly as JSON
@RestController
@RequestMapping("/api/students")
public class StudentController {

 @GetMapping
 public List<Student> getAllStudents() {
 return List.of(new Student(1, "Alice"), new Student(2, "Bob"));
 // Returns: [{"id":1,"name":"Alice"}, {"id":2,"name":"Bob"}]
 }
}
```

**Interview Answer:**
> "`@Controller` is used in Spring MVC to return view names for rendering HTML pages. `@RestController` is a combination of `@Controller` and `@ResponseBody`, meaning every method automatically serializes the return value to JSON/XML and writes it directly to the HTTP response. We use `@RestController` for building REST APIs."

---

### Q26. Explain all important Spring Boot annotations for REST APIs

**Answer:**

```java
@RestController // Marks class as REST controller
@RequestMapping("/api/v1") // Base path for all endpoints in this controller
public class StudentController {

 @GetMapping("/students") // GET /api/v1/students
 public List<Student> getAll() { ... }

 @GetMapping("/students/{id}") // GET /api/v1/students/1
 public Student getById(@PathVariable Long id) { ... }
 // @PathVariable extracts 'id' from the URL path

 @GetMapping("/students/search") // GET /api/v1/students/search?name=Alice
 public Student search(@RequestParam String name) { ... }
 // @RequestParam extracts query parameters from URL

 @PostMapping("/students") // POST /api/v1/students
 public Student create(@RequestBody Student student) { ... }
 // @RequestBody reads JSON from request body and converts to object

 @PutMapping("/students/{id}") // PUT /api/v1/students/1
 public Student update(@PathVariable Long id, @RequestBody Student student) { ... }

 @DeleteMapping("/students/{id}") // DELETE /api/v1/students/1
 public void delete(@PathVariable Long id) { ... }
}
```

**Interview Answer:**
> "`@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping` map HTTP methods to controller methods. `@PathVariable` extracts values from the URL path (like `/students/{id}`). `@RequestParam` extracts query parameters (like `?name=Alice`). `@RequestBody` deserializes the request JSON body into a Java object."

### Q27. What is `ResponseEntity` in Spring Boot?

**Answer:**

`ResponseEntity` is a class in Spring that allows you to control the **full HTTP response** — including the status code, headers, and body.

```java
@RestController
@RequestMapping("/api/students")
public class StudentController {

 @GetMapping("/{id}")
 public ResponseEntity<Student> getStudent(@PathVariable Long id) {
 Student student = studentService.findById(id);

 if (student == null) {
 return ResponseEntity.status(HttpStatus.NOT_FOUND).body(null);
 // Returns: 404 Not Found
 }

 return ResponseEntity.ok(student);
 // Returns: 200 OK + student JSON
 }

 @PostMapping
 public ResponseEntity<Student> createStudent(@RequestBody Student student) {
 Student saved = studentService.save(student);
 return ResponseEntity.status(HttpStatus.CREATED).body(saved);
 // Returns: 201 Created + saved student JSON
 }
}
```

**Interview Answer:**
> "`ResponseEntity` gives us full control over the HTTP response. We can set the status code, headers, and body. For example, we return `200 OK` with data when a resource is found, `404 Not Found` when it's missing, and `201 Created` when a new resource is created. This is a best practice in REST API development."

## 9. Spring Data JPA & Database

---

### Q28. What is JPA and Hibernate?

**Answer:**

- **JPA (Java Persistence API):** A **specification** (a set of rules/interfaces) that defines how Java objects should be stored in a relational database. JPA itself doesn't do anything — it's a blueprint.

- **Hibernate:** The most popular **implementation** of JPA. It's the actual library that converts Java objects to database rows and back. When you use Spring Data JPA, Hibernate is the default JPA provider.

**Analogy:**
> JPA is like the **interface** (contract), and Hibernate is the **class** (implementation). Just like `List` is the interface and `ArrayList` is the implementation.

**Interview Answer:**
> "JPA is a Java specification that defines how to map Java objects to relational database tables using annotations. Hibernate is the most widely used implementation of JPA — it does the actual work of translating between Java objects and SQL. Spring Data JPA sits on top of Hibernate to make database operations even simpler."

---

### Q29. What is Spring Data JPA? What is a `JpaRepository`?

**Answer:**

**Spring Data JPA** is a Spring module that makes working with databases incredibly easy. You just **extend `JpaRepository`** and Spring automatically provides all CRUD operations — no SQL needed!

**Entity (Database Table Mapping):**
```java
@Entity
@Table(name = "students")
public class Student {

 @Id
 @GeneratedValue(strategy = GenerationType.IDENTITY)
 private Long id;

 @Column(name = "student_name")
 private String name;

 private String email;

 // Getters and Setters...
}
```

**Repository (Database Operations):**
```java
@Repository
public interface StudentRepository extends JpaRepository<Student, Long> {
 // Spring auto-generates: save(), findById(), findAll(), delete(), etc.

 // Custom query methods using method naming convention:
 List<Student> findByName(String name); // SELECT * FROM students WHERE name = ?
 List<Student> findByEmailContaining(String email); // SELECT * WHERE email LIKE %email%
}
```

**Service (Business Logic):**
```java
@Service
public class StudentService {

 @Autowired
 private StudentRepository studentRepository;

 public List<Student> getAllStudents() {
 return studentRepository.findAll();
 }

 public Student saveStudent(Student student) {
 return studentRepository.save(student);
 }

 public void deleteStudent(Long id) {
 studentRepository.deleteById(id);
 }
}
```

**Interview Answer:**
> "Spring Data JPA makes database operations very simple. We define an entity class with `@Entity` to map it to a database table, then create a repository interface that extends `JpaRepository`. Spring automatically provides implementations for all standard CRUD methods. We can also define custom queries just by naming the method correctly, like `findByName()` or `findByEmail()`."

### Q30. What are the common JPA annotations?

**Answer:**

| Annotation | Purpose | Example |
|---|---|---|
| `@Entity` | Marks class as a database table | `@Entity` |
| `@Table` | Specifies table name | `@Table(name = "students")` |
| `@Id` | Marks primary key | `@Id` |
| `@GeneratedValue` | Auto-generate primary key | `@GeneratedValue(strategy = GenerationType.IDENTITY)` |
| `@Column` | Maps field to a specific column | `@Column(name = "full_name", nullable = false)` |
| `@OneToOne` | One-to-one relationship | Student ↔ Profile |
| `@OneToMany` | One-to-many relationship | Department → many Employees |
| `@ManyToOne` | Many-to-one relationship | Many Employees → one Department |
| `@ManyToMany` | Many-to-many relationship | Students ↔ Courses |
