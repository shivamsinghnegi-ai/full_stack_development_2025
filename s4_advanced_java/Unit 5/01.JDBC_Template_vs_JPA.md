# JDBC Template vs JPA — Overview and MySQL Setup
---

## 1. Recap — How You Connected to Database Before

In your previous Java programs, you used plain JDBC to connect to MySQL:

```java
// What you wrote in every DAO class
Class.forName("com.mysql.cj.jdbc.Driver");
Connection con = DriverManager.getConnection(
    "jdbc:mysql://localhost:3306/studentdb", "root", "1234"
);
PreparedStatement ps = con.prepareStatement("SELECT * FROM students");
ResultSet rs = ps.executeQuery();

while (rs.next()) {
    int id       = rs.getInt("id");
    String name  = rs.getString("name");
    // manually read each column
}
rs.close();
ps.close();
con.close();
```

This worked. But you had to write this for every single operation — SELECT, INSERT, UPDATE, DELETE. Same boilerplate, every time, in every DAO.

Spring gives two better ways to handle database operations:

1. Spring JDBC Template
2. Spring Data JPA (with Hibernate)

---

## 2. What is Spring JDBC Template?

Spring JDBC Template is a class provided by Spring Framework that **wraps plain JDBC** and removes the repetitive boilerplate code.

You still write SQL queries, but Spring handles:
- Opening and closing connections automatically
- Handling exceptions
- Managing PreparedStatements
- Iterating ResultSets

### JDBC vs Spring JDBC Template — Comparison

**Plain JDBC (what you wrote before):**
```java
Connection con = DriverManager.getConnection(url, user, pass);
PreparedStatement ps = con.prepareStatement("SELECT * FROM students");
ResultSet rs = ps.executeQuery();
List<Student> list = new ArrayList<>();
while (rs.next()) {
    Student s = new Student();
    s.setId(rs.getInt("id"));
    s.setName(rs.getString("name"));
    list.add(s);
}
rs.close();
ps.close();
con.close();
```

**Spring JDBC Template (same result, much less code):**
```java
List<Student> list = jdbcTemplate.query(
    "SELECT * FROM students",
    (rs, rowNum) -> {
        Student s = new Student();
        s.setId(rs.getInt("id"));
        s.setName(rs.getString("name"));
        return s;
    }
);
```

### What Spring JDBC Template Removes

| What you had to write manually | Spring JDBC Template handles it |
|---|---|
| Open connection | Automatic |
| Create PreparedStatement | Automatic |
| Execute query | Automatic |
| Iterate ResultSet | Automatic |
| Close connection | Automatic |
| Handle SQLException | Automatic |
| SQL query | You still write this |
| Map ResultSet to object | You still write this |

### When to Use JDBC Template

- When you need full control over your SQL queries
- Complex queries with joins, subqueries, stored procedures
- When you are migrating a large existing JDBC project to Spring
- When the team is more comfortable writing SQL

## 3. What is JPA?

**JPA = Java Persistence API**

JPA is a **specification** (a set of rules and interfaces) that defines how Java objects should be stored in a relational database.

JPA itself does not do anything — it is just a specification. You need an **implementation** of JPA to actually work.

**Hibernate** is the most popular implementation of JPA. When you add `spring-boot-starter-data-jpa` to your project, Hibernate is included automatically.

### What JPA Does

JPA maps Java classes to database tables and Java objects to database rows. This is called **ORM — Object Relational Mapping**.

```
Java Class   →  Database Table
Java Object  →  Database Row
Java Field   →  Database Column
```

Example:

```java
// Java class
public class Student {
    int    id;
    String name;
    String email;
}

// JPA automatically maps this to:
// CREATE TABLE student (
//     id    INT PRIMARY KEY,
//     name  VARCHAR(255),
//     email VARCHAR(255)
// );
```

You do not write the CREATE TABLE SQL. JPA creates the table automatically based on your Java class.

### What is Spring Data JPA?

Spring Data JPA is a Spring project built on top of JPA and Hibernate. It goes one step further — it gives you **ready-made database methods** without writing any SQL or any implementation code.

```java
// Just extend JpaRepository — you get all of these free:
studentRepository.findAll();         // SELECT * FROM students
studentRepository.findById(1);       // SELECT * WHERE id = 1
studentRepository.save(student);     // INSERT or UPDATE
studentRepository.deleteById(1);     // DELETE WHERE id = 1
studentRepository.count();           // SELECT COUNT(*)
```

No SQL. No JDBC. No connection management. Everything is handled.

## 4. JDBC Template vs JPA — Full Comparison

This is the most important table for your exam.

| Feature | JDBC Template | Spring Data JPA |
|---|---|---|
| SQL writing | You write SQL | No SQL needed for basic operations |
| Boilerplate | Reduced (no connection management) | Almost zero |
| ResultSet mapping | You write it manually | Automatic (Hibernate maps it) |
| Table creation | You write CREATE TABLE | Automatic from Java class |
| Learning curve | Easy (you already know SQL) | Medium (need to learn annotations) |
| Control over queries | Full control | Less control (for complex queries) |
| Performance tuning | Easy — you write the SQL | Harder — Hibernate generates SQL |
| Complex queries | Better suited | Needs @Query annotation |
| Best for | SQL-heavy projects, migrations | Standard CRUD applications |
| Used in industry | Legacy projects | Modern Spring Boot projects |

### Simple Way to Remember

- JDBC Template = Spring + your SQL
- JPA = Spring + no SQL (for basic operations)

## 5. Why JPA is Preferred in Modern Projects

You already know JDBC. You know writing SQL for every operation is tedious. JPA solves this:

**Problem 1 — Changing database**
With plain JDBC, if you move from MySQL to PostgreSQL, your SQL queries may need changes. With JPA, Hibernate generates the SQL — you just change one property in `application.properties`. Your Java code stays the same.

**Problem 2 — Table creation**
With JDBC, you write CREATE TABLE manually. With JPA, the table is created automatically from your Java class using `@Entity`.

**Problem 3 — Boilerplate**
With JDBC, save operation needs INSERT query, UPDATE query, check if exists — separately. With JPA, `save()` handles everything — it decides INSERT or UPDATE automatically.

**Problem 4 — Relationships**
Managing foreign keys and joins in plain JDBC is very complex. JPA handles relationships between tables using annotations like `@OneToMany`, `@ManyToOne` (advanced topic).

## 8. pom.xml — Understanding the Dependencies

When you open `pom.xml`, you will see these three dependencies:

```xml
<!-- DEPENDENCY 1: Spring Web -->
<!-- Gives: @RestController, @GetMapping, embedded Tomcat, Jackson (JSON) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

<!-- DEPENDENCY 2: Spring Data JPA -->
<!-- Gives: JPA, Hibernate, JpaRepository, @Entity, @Id, @Column -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-jpa</artifactId>
</dependency>

<!-- DEPENDENCY 3: MySQL Driver -->
<!-- Gives: ability to connect to MySQL database -->
<!-- scope=runtime: only needed when app is running, not during compilation -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>
```

Previously you downloaded `mysql-connector.jar` manually. Now Maven downloads it automatically when you add this dependency. That is the benefit of a build tool.

## 9. application.properties — Database Configuration

This is the most important file for Class 1. Open:
`src/main/resources/application.properties`

```properties
# Application name
spring.application.name=studentcrud

# Server port (default is 8080 — no need to change)
server.port=8080

# -----------------------------------------------
# MySQL Connection Configuration
# -----------------------------------------------
# This replaces the 3 lines you wrote in every JDBC DAO:
#   Class.forName("com.mysql.cj.jdbc.Driver");
#   Connection con = DriverManager.getConnection(url, user, pass);
# Spring Boot reads these properties and creates the connection automatically.

spring.datasource.url=jdbc:mysql://localhost:3306/studentdb
spring.datasource.username=root
spring.datasource.password=1234
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

# -----------------------------------------------
# JPA and Hibernate Configuration
# -----------------------------------------------

# ddl-auto controls what Hibernate does with your database tables on startup.
#
# update  → If table does not exist, create it.
#           If table exists, update its structure. Data is NOT deleted.
#           Use this for development — safe option.
#
# create  → Drop the table and recreate it every time app starts.
#           All data is lost on restart. Use only for initial testing.
#
# none    → Do nothing. You manage tables manually.
#           Use this in production.
#
# validate → Check if table matches Entity class. Throw error if mismatch.
#            Use this in production to catch mistakes.

spring.jpa.hibernate.ddl-auto=update

# Show the SQL queries Hibernate generates in the console
# Useful for learning — you can see what SQL JPA is running
# Set to false in production
spring.jpa.show-sql=true

# Format the SQL output so it is readable
spring.jpa.properties.hibernate.format_sql=true

# Tell Hibernate which database dialect to use
# Dialect = the specific SQL flavour for MySQL
# Hibernate uses this to generate MySQL-compatible SQL
spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQLDialect
```

### What is a Dialect?

Different databases (MySQL, PostgreSQL, Oracle) have slightly different SQL syntax. Hibernate needs to know which database you are using so it generates the correct SQL.

For MySQL: `org.hibernate.dialect.MySQLDialect`
For PostgreSQL: `org.hibernate.dialect.PostgreSQLDialect`
For H2 (in-memory): `org.hibernate.dialect.H2Dialect`

---

## 10. ddl-auto Values — Detailed Explanation

This is commonly asked in exams.

| Value | What Happens | When to Use |
|---|---|---|
| `update` | Creates table if not exists. Alters if structure changed. Data is safe. | Development |
| `create` | Drops and recreates table every time. All data lost. | Initial testing only |
| `create-drop` | Creates on startup, drops on shutdown. | Unit testing |
| `validate` | Checks table matches Entity. Error if mismatch. No changes made. | Production |
| `none` | Does nothing. You manage tables. | Production |

## 11. How Spring Boot Connects to MySQL — Behind the Scenes

When `SpringApplication.run()` is called:

```
1. Spring Boot reads application.properties
          |
          v
2. Creates a Connection Pool (HikariCP — bundled with Spring Boot)
   HikariCP manages multiple database connections efficiently
   You do not need to open/close connections manually
          |
          v
3. Hibernate reads all @Entity classes in the project
          |
          v
4. Based on ddl-auto=update:
   - Checks if "students" table exists in MySQL
   - If not → runs CREATE TABLE automatically
   - If yes → checks if any column needs to be added
          |
          v
5. Application is ready to handle requests
```

---

## 12. What is HikariCP?

When you used JDBC, you created one connection per request. This is slow because opening a database connection takes time.

**Connection Pool** is a collection of pre-opened database connections that are reused across requests.

**HikariCP** is the connection pool library that Spring Boot uses by default. It comes bundled inside `spring-boot-starter-data-jpa`.

```
Without connection pool:
  Request 1 → open connection → query → close connection
  Request 2 → open connection → query → close connection   (slow)

With HikariCP:
  App starts → 10 connections opened and kept ready
  Request 1 → takes one connection from pool → query → returns to pool
  Request 2 → takes another connection → query → returns                (fast)
```

You do not need to configure HikariCP. Spring Boot handles it automatically.

## 13. Comparison — Old JDBC vs Spring Boot + JPA

| Task | Old JDBC Way | Spring Boot + JPA Way |
|---|---|---|
| Load driver | `Class.forName(...)` | Not needed |
| Get connection | `DriverManager.getConnection(...)` | Automatic (HikariCP) |
| Write SQL | Every operation needs SQL | Not needed for basic CRUD |
| Create table | Manual `CREATE TABLE` SQL | Automatic from `@Entity` |
| Map result | Manually read from ResultSet | Automatic (Hibernate) |
| Close connection | Manually in finally block | Automatic |
| Configuration | Hardcoded in Java file | `application.properties` |

## 14. Questions

**Q1. What is the difference between JDBC Template and Spring Data JPA?**

Ans: JDBC Template is a Spring class that simplifies plain JDBC by removing connection management and exception handling, but you still write SQL queries and map ResultSet to objects manually. Spring Data JPA is built on top of JPA and Hibernate — it removes the need to write SQL for basic operations and maps Java classes to database tables automatically. JDBC Template gives more control over SQL. Spring Data JPA is preferred for standard CRUD applications due to less boilerplate.

---

**Q2. What is JPA? What is the difference between JPA and Hibernate?**

Ans: JPA (Java Persistence API) is a specification that defines rules for how Java objects should be stored in relational databases. It is only a set of interfaces — it does not have its own implementation. Hibernate is an implementation of the JPA specification. When you use Spring Data JPA, Hibernate runs behind the scenes as the actual engine performing database operations.

---

**Q3. What is ORM? How does JPA implement it?**

Ans: ORM stands for Object Relational Mapping. It is a technique that maps Java objects to relational database tables so that developers do not need to write SQL for basic operations. JPA implements ORM using annotations:
- `@Entity` maps a Java class to a database table
- `@Id` marks the primary key field
- `@Column` maps a Java field to a database column
- `@GeneratedValue` enables auto-increment for the primary key

---

**Q4. Explain the ddl-auto property in application.properties.**

Ans: The `spring.jpa.hibernate.ddl-auto` property controls how Hibernate manages the database schema when the application starts.
- `update` — Creates table if not present, updates structure if changed, data is preserved. Used in development.
- `create` — Drops and recreates the table every time. All data is lost. Used for initial testing.
- `validate` — Checks if the table matches the Entity class. Throws error if mismatch. Used in production.
- `none` — Does nothing. Tables are managed manually. Used in production.

---

**Q5. What is HikariCP? Why is it used?**

Ans: HikariCP is a high-performance connection pool library that is bundled with Spring Boot. Instead of opening and closing a database connection for every request (which is slow), HikariCP maintains a pool of pre-opened connections that are reused. This significantly improves application performance. Spring Boot configures HikariCP automatically — no manual setup is required.

---

**Q6. What does spring.jpa.show-sql=true do?**

Ans: Setting `spring.jpa.show-sql=true` in `application.properties` tells Hibernate to print all the SQL queries it generates to the console. This is useful during development and learning because you can see the exact SQL (SELECT, INSERT, UPDATE, DELETE) that Hibernate runs behind the scenes. It should be set to false in production to avoid performance overhead and to keep logs clean.
