## 2. What is CRUD

CRUD stands for Create, Read, Update, Delete. These are the four fundamental operations performed on any data in a database. You already performed CRUD in plain JDBC using SQL queries. In Spring Boot with JPA, the same operations are performed through repository methods.

| CRUD Operation | HTTP Method | SQL Behind the Scenes | JPA Method |
|---|---|---|---|
| Create | POST | INSERT INTO students ... | save() |
| Read | GET | SELECT * FROM students | findAll(), findById() |
| Update | PUT | UPDATE students SET ... | save() with existing id |
| Delete | DELETE | DELETE FROM students WHERE id = ? | deleteById() |

## 5. CREATE — POST Request

### What Happens

When a POST request is received with a JSON body, Spring reads the JSON using `@RequestBody`, converts it to a `Student` object, passes it to the service, which calls `studentRepository.save(student)`. Since the `id` field is not set, Hibernate runs an INSERT query and MySQL assigns the `id` using AUTO_INCREMENT.

### Controller Method

```java
@PostMapping
public ResponseEntity<Student> addStudent(@RequestBody Student student) {
    Student saved = studentService.addStudent(student);
    return ResponseEntity.status(201).body(saved);
}
```

### Service Method

```java
public Student addStudent(Student student) {
    return studentRepository.save(student);
}
```

## 7. UPDATE — PUT Request

### What Happens

A PUT request carries two pieces of information:
- The `id` of the student to update — in the URL path (`/{id}`)
- The new data — in the JSON request body

The service fetches the existing student using `findById(id)`. If found, it updates the fields and calls `save()` again. Since the object already has an `id`, Hibernate runs an UPDATE query instead of INSERT.

### Controller Method

```java
@PutMapping("/{id}")
public ResponseEntity<Student> updateStudent(
        @PathVariable int id,
        @RequestBody Student updatedData) {
    Optional<Student> result = studentService.updateStudent(id, updatedData);
    if (result.isPresent()) {
        return ResponseEntity.ok(result.get());
    }
    return ResponseEntity.notFound().build();
}
```

### Service Method

```java
public Optional<Student> updateStudent(int id, Student updatedData) {
    Optional<Student> existing = studentRepository.findById(id);
    if (existing.isPresent()) {
        Student s = existing.get();
        s.setName(updatedData.getName());
        s.setEmail(updatedData.getEmail());
        s.setCourse(updatedData.getCourse());
        s.setPhone(updatedData.getPhone());
        return Optional.of(studentRepository.save(s));
    }
    return Optional.empty();
}
```

### SQL Generated by Hibernate

```sql
-- First: SELECT to fetch existing record
SELECT * FROM students WHERE id = 1

-- Then: UPDATE with new values
UPDATE students
SET name = 'Ravi Sharma Updated', email = 'ravi.new@gmail.com',
    course = 'Spring Boot', phone = '9111111111'
WHERE id = 1
```

## 8. DELETE — DELETE Request

### What Happens

A DELETE request carries only the `id` of the student to delete in the URL path. The service checks if the student exists using `existsById(id)`. If found, it calls `deleteById(id)`. Hibernate runs a DELETE query in MySQL.

### Controller Method

```java
@DeleteMapping("/{id}")
public ResponseEntity<String> deleteStudent(@PathVariable int id) {
    boolean deleted = studentService.deleteStudent(id);
    if (deleted) {
        return ResponseEntity.ok("Student with ID " + id + " deleted successfully.");
    }
    return ResponseEntity.notFound().build();
}
```

### Service Method

```java
public boolean deleteStudent(int id) {
    if (studentRepository.existsById(id)) {
        studentRepository.deleteById(id);
        return true;
    }
    return false;
}
```

### SQL Generated by Hibernate

-- First: SELECT to check existence

## 11. How save() Decides INSERT vs UPDATE

This is commonly asked in exams. The behaviour of `save()` depends entirely on the `id` field.

```java
// Case 1: New student — id not set — INSERT runs
Student s = new Student();
s.setName("Ravi");
// s.id = 0 (default int value)
studentRepository.save(s);
// Hibernate: INSERT INTO students (name, ...) VALUES (?, ...)

// Case 2: Existing student — id is set — UPDATE runs
Student s = studentRepository.findById(1).get();
// s.id = 1 (fetched from DB)
s.setName("Ravi Updated");
studentRepository.save(s);
// Hibernate: UPDATE students SET name=? WHERE id=1
```

The rule is:
- `id` is 0 or null → record does not exist → INSERT
- `id` has a value → record exists → UPDATE

## 12. @PathVariable vs @RequestParam vs @RequestBody — Final Comparison

All three are used in this project. Here is where each one appears:

| Annotation | Used In | Where Data Comes From | Example URL |
|---|---|---|---|
| `@PathVariable` | GET by id, PUT, DELETE | URL path segment | `/api/students/1` |
| `@RequestParam` | Search by course, by name | URL query string | `/api/students/search?course=Java` |
| `@RequestBody` | POST, PUT | JSON in request body | Body: `{"name":"Ravi",...}` |

```java
// @PathVariable — reads {id} from the URL path
@GetMapping("/{id}")
public ResponseEntity<Student> getById(@PathVariable int id) { }

// @RequestParam — reads ?course=Java from the URL
@GetMapping("/search")
public ResponseEntity<List<Student>> search(@RequestParam String course) { }

// @RequestBody — reads the JSON body of the request
@PostMapping
public ResponseEntity<Student> add(@RequestBody Student student) { }
```

## 13. Optional — Why It is Used

`Optional<Student>` is returned by `findById()` because the student may or may not exist in the database.

```java
Optional<Student> student = studentRepository.findById(id);

// Check if value is present
if (student.isPresent()) {
    Student s = student.get();   // get the actual value
    return ResponseEntity.ok(s);
}

// If not present
return ResponseEntity.notFound().build();
```

Using `Optional` prevents `NullPointerException`. If `findById` returned a plain `Student` object and the record did not exist, calling any method on a null object would crash the application.

| Optional Method | Returns |
|---|---|
| `isPresent()` | true if value exists |
| `isEmpty()` | true if no value |
| `get()` | the actual value (use only after isPresent check) |
| `orElse(default)` | value if present, otherwise the default |
| `orElseThrow(exception)` | value if present, otherwise throws exception |

## 14. Common Errors During Testing

**Error: 400 Bad Request on POST**

Cause: JSON body is malformed, or `Content-Type` header is missing in Postman.

Fix:
- In Postman, go to Body → raw → select JSON from the dropdown
- Check that the JSON has correct syntax — no missing commas, all strings in double quotes

---

**Error: 500 Internal Server Error on POST with duplicate email**

Cause: `@Column(unique = true)` is set on the email field. Sending the same email twice violates this constraint.

Fix: Use a different email address for each student.

Console shows:
```
could not execute statement [Duplicate entry 'ravi@gmail.com' for key 'students.email']
```

---

**Error: Empty JSON response { }**

Cause: Getters are missing in `Student.java`.

Fix: Add getters for all fields.

---

**Error: All fields null after POST**

Cause: Default constructor or setters are missing in `Student.java`.

Fix: Add `public Student() {}` and setters for all fields.

---

**Error: 404 on correct URL**

Cause: `StudentController.java` is not in the correct package. Spring's `@ComponentScan` did not find it.

Fix: The controller must be in `com.college.studentcrud.controller` or any sub-package of `com.college.studentcrud`.

---

**Error: Access denied for user 'root'**

Cause: Wrong password in `application.properties`.

Fix: Update `spring.datasource.password` with your actual MySQL password.

## 15.Questions 

**Q1. What HTTP method is used for each CRUD operation in REST?**

Ans:
- Create → POST. A new resource is created. Status 201 Created.
- Read → GET. Data is fetched. Status 200 OK.
- Update → PUT. An existing resource is updated. Status 200 OK.
- Delete → DELETE. A resource is removed. Status 200 OK or 204 No Content.

---

**Q2. How does JPA handle INSERT and UPDATE using the same save() method?**

Ans: `save()` inspects the primary key field of the entity object. If the `id` is 0 or null, the record does not exist in the database, so Hibernate executes an INSERT query. If the `id` has a value corresponding to an existing database record, Hibernate executes an UPDATE query. This is why when updating a student, the existing record must first be fetched using `findById()`, its fields updated, and then `save()` called on the same object — the `id` is retained, so UPDATE is triggered.

---

**Q3. What is the difference between @PathVariable and @RequestParam?**

Ans: `@PathVariable` reads a value embedded inside the URL path. For example, in the URL `/api/students/3`, the value `3` is extracted using `@PathVariable int id`. It is used when identifying a specific resource. `@RequestParam` reads a value from the query string after the `?` in the URL. For example, in `/api/students/search?course=Java`, the value `Java` is extracted using `@RequestParam String course`. It is used for filtering or searching.

---

**Q4. What is Optional? Why is it used with findById()?**

Ans: `Optional<T>` is a container class in Java that may or may not hold a non-null value. `findById()` returns `Optional<Student>` because the record may not exist in the database. Using Optional forces the developer to explicitly check whether the value is present before using it, which prevents NullPointerException. Methods like `isPresent()`, `get()`, and `orElseThrow()` are used to work with Optional values safely.

---

**Q5. What status code should be returned for each scenario?**

Ans:

| Scenario | Status Code |
|---|---|
| GET request — data found | 200 OK |
| POST request — new record created | 201 Created |
| PUT request — record updated | 200 OK |
| DELETE request — record removed | 200 OK |
| Any request — ID does not exist | 404 Not Found |
| POST/PUT request — invalid input | 400 Bad Request |
| Unexpected server error | 500 Internal Server Error |

---

**Q6. Explain the role of show-sql property. What is its use?**

Ans: `spring.jpa.show-sql=true` in `application.properties` instructs Hibernate to print every SQL query it generates to the application console. This is useful during development and learning because it makes the database activity visible — you can see the exact INSERT, SELECT, UPDATE, and DELETE statements that JPA runs for each repository method call. It helps in debugging query issues and understanding how method names are converted to SQL. It should be set to false in production.
