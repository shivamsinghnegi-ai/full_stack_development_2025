## 2. The Problem With the Current Code

In Class 3, when a student with a given ID does not exist, the controller returns:

```java
return ResponseEntity.notFound().build();
```

The client receives:

```
HTTP/1.1 404 Not Found
(empty body)
```

This tells the client that something went wrong, but gives no information about what went wrong or why. A real application should return something like:

```json
HTTP/1.1 404 Not Found

{
    "status"   : 404,
    "message"  : "Student with ID 99 does not exist.",
    "timestamp": "2026-04-10T10:30:00"
}
```

This is what every production REST API returns. The client — whether it is a mobile app, a frontend, or another service — can read the message and display it to the user or log it for debugging.

The same problem exists for other error scenarios:
- What if the client sends invalid JSON?
- What if a database constraint is violated (duplicate email)?
- What if an unexpected server error occurs?

All of these need proper responses. That is what this class covers.

## 3. Types of Exceptions in a Spring Boot Application

| Exception Type | When It Happens | Example |
|---|---|---|
| Resource not found | Requested ID does not exist in DB | GET /students/999 |
| Duplicate data | Unique constraint violated | POST with duplicate email |
| Invalid input | Required field is missing or blank | POST with empty name |
| Bad request | Malformed JSON sent in body | Missing quotes in JSON |
| Server error | Unexpected exception in code | NullPointerException |

Each of these should return a different HTTP status code and a meaningful message.

---

## 4. Two Ways to Handle Exceptions

Spring Boot provides two approaches:

| Approach | Annotation | Scope |
|---|---|---|
| Local handler | `@ExceptionHandler` | One controller only |
| Global handler | `@ControllerAdvice` | All controllers |

For any real project with more than one controller, the global approach using `@ControllerAdvice` is always preferred. It avoids repeating the same error handling code in every controller.

## 6. FILE 1 — ErrorResponse.java

This class defines the structure of the JSON that is returned when an error occurs.

Create this inside the `exception` package.

```java
package com.college.studentcrud.exception;

import java.time.LocalDateTime;

public class ErrorResponse {

    private int           status;
    private String        message;
    private LocalDateTime timestamp;

    public ErrorResponse(int status, String message) {
        this.status    = status;
        this.message   = message;
        this.timestamp = LocalDateTime.now();
    }

    // Getters — required for Jackson to convert this object to JSON
    public int           getStatus()    { return status; }
    public String        getMessage()   { return message; }
    public LocalDateTime getTimestamp() { return timestamp; }
}
```

When an exception occurs and this object is returned, Jackson converts it to:

```json
{
    "status"   : 404,
    "message"  : "Student with ID 99 does not exist.",
    "timestamp": "2026-04-10T10:30:00"
}
```

## 7. FILE 2 — StudentNotFoundException.java

This is a custom exception class for the specific case when a student ID is not found.

```java
package com.college.studentcrud.exception;

public class StudentNotFoundException extends RuntimeException {

    private int studentId;

    public StudentNotFoundException(int id) {
        super("Student with ID " + id + " does not exist.");
        this.studentId = id;
    }

    public int getStudentId() {
        return studentId;
    }
}
```

### Why RuntimeException?

Java exceptions are divided into two categories:

| Category | Base class | Must declare with throws? |
|---|---|---|
| Checked exception | Exception | Yes — must declare or catch |
| Unchecked exception | RuntimeException | No — no need to declare |

Using `RuntimeException` means the method that throws it does not need to declare `throws StudentNotFoundException` in its signature. This keeps the service and controller code clean.

### Why a Custom Exception Instead of Generic RuntimeException?

A custom exception class carries specific information — in this case, the `studentId` that was not found. It also makes the code more readable and allows the `@ExceptionHandler` to target it specifically.

## 9. FILE 4 — GlobalExceptionHandler.java

This is the central class that catches exceptions thrown from any controller in the application.

```java
package com.college.studentcrud.exception;

import org.springframework.http.ResponseEntity;
import org.springframework.http.converter.HttpMessageNotReadableException;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

@ControllerAdvice
public class GlobalExceptionHandler {

    // -------------------------------------------------------
    // Handler 1: StudentNotFoundException
    // -------------------------------------------------------
    // Triggered when: service throws StudentNotFoundException
    // Happens when: findById() returns empty, student not found
    // Returns: 404 Not Found + JSON error message
    // -------------------------------------------------------
    @ExceptionHandler(StudentNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleStudentNotFound(
            StudentNotFoundException ex) {

        ErrorResponse error = new ErrorResponse(404, ex.getMessage());
        return ResponseEntity.status(404).body(error);
    }

    // -------------------------------------------------------
    // Handler 2: DuplicateEmailException
    // -------------------------------------------------------
    // Triggered when: service throws DuplicateEmailException
    // Happens when: client tries to add a student with an
    //               email that already exists in the database
    // Returns: 409 Conflict + JSON error message
    // -------------------------------------------------------
    @ExceptionHandler(DuplicateEmailException.class)
    public ResponseEntity<ErrorResponse> handleDuplicateEmail(
            DuplicateEmailException ex) {

        ErrorResponse error = new ErrorResponse(409, ex.getMessage());
        return ResponseEntity.status(409).body(error);
    }

    // -------------------------------------------------------
    // Handler 3: HttpMessageNotReadableException
    // -------------------------------------------------------
    // Triggered when: client sends malformed JSON in request body
    // Happens when: missing comma, missing quote, wrong format
    // Returns: 400 Bad Request + JSON error message
    // -------------------------------------------------------
    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ErrorResponse> handleMalformedJson(
            HttpMessageNotReadableException ex) {

        ErrorResponse error = new ErrorResponse(
            400,
            "Invalid JSON format in request body. Please check your input."
        );
        return ResponseEntity.status(400).body(error);
    }

    // -------------------------------------------------------
    // Handler 4: General Exception (safety net)
    // -------------------------------------------------------
    // Triggered when: any exception not handled above is thrown
    // Happens when: unexpected errors, NullPointerException, etc.
    // Returns: 500 Internal Server Error + JSON error message
    // -------------------------------------------------------
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception ex) {

        ErrorResponse error = new ErrorResponse(
            500,
            "An unexpected error occurred on the server. Please try again."
        );
        return ResponseEntity.status(500).body(error);
    }
}
```

### How @ControllerAdvice Works

```
Any controller throws an exception
              |
              v
Spring intercepts the exception before it reaches the client
              |
              v
Spring looks for a matching @ExceptionHandler in @ControllerAdvice
              |
              v
If found → that handler method runs → returns error JSON
              |
              v
If not found → falls through to the general Exception handler
              |
              v
Client receives a proper JSON error response
```

Without `@ControllerAdvice`, the exception would reach the client as a Spring default error page or an empty body. With `@ControllerAdvice`, every exception is intercepted and converted into a meaningful JSON response.

## 10. FILE 5 — Updated StudentService.java

The service is updated to throw custom exceptions instead of returning `Optional`. This makes the controller much simpler.

```java
package com.college.studentcrud.service;

import com.college.studentcrud.exception.DuplicateEmailException;
import com.college.studentcrud.exception.StudentNotFoundException;
import com.college.studentcrud.model.Student;
import com.college.studentcrud.repository.StudentRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class StudentService {

    private final StudentRepository studentRepository;

    @Autowired
    public StudentService(StudentRepository studentRepository) {
        this.studentRepository = studentRepository;
    }

    // CREATE
    public Student addStudent(Student student) {

        // Check if email already exists before saving
        if (studentRepository.findByEmail(student.getEmail()).isPresent()) {
            throw new DuplicateEmailException(student.getEmail());
        }

        return studentRepository.save(student);
    }

    // READ — all
    public List<Student> getAllStudents() {
        return studentRepository.findAll();
    }

    // READ — by id
    // orElseThrow() returns the Student if found,
    // or throws StudentNotFoundException if not found.
    // No Optional.isPresent() check needed here.
    public Student getStudentById(int id) {
        return studentRepository.findById(id)
                .orElseThrow(() -> new StudentNotFoundException(id));
    }

    // READ — by course
    public List<Student> getStudentsByCourse(String course) {
        return studentRepository.findByCourse(course);
    }

    // READ — by name
    public List<Student> searchByName(String keyword) {
        return studentRepository.findByNameContaining(keyword);
    }

    // UPDATE
    public Student updateStudent(int id, Student updatedData) {

        // getStudentById() throws StudentNotFoundException if not found
        Student existing = getStudentById(id);

        // Check if new email is already taken by a different student
        studentRepository.findByEmail(updatedData.getEmail())
                .ifPresent(found -> {
                    if (found.getId() != id) {
                        throw new DuplicateEmailException(updatedData.getEmail());
                    }
                });

        existing.setName(updatedData.getName());
        existing.setEmail(updatedData.getEmail());
        existing.setCourse(updatedData.getCourse());
        existing.setPhone(updatedData.getPhone());

        return studentRepository.save(existing);
    }

    // DELETE
    public void deleteStudent(int id) {
        // getStudentById() throws StudentNotFoundException if not found
        Student student = getStudentById(id);
        studentRepository.delete(student);
    }

    // COUNT
    public long getTotalCount() {
        return studentRepository.count();
    }
}
```

### Key Change — orElseThrow()

Previously in Class 3:

```java
// Old approach — returns Optional, controller checks isPresent()
public Optional<Student> getStudentById(int id) {
    return studentRepository.findById(id);
}
```

Now:

```java
// New approach — returns Student directly, throws exception if not found
public Student getStudentById(int id) {
    return studentRepository.findById(id)
            .orElseThrow(() -> new StudentNotFoundException(id));
}
```

`orElseThrow()` is an `Optional` method. If the value is present, it returns it. If the value is absent, it executes the lambda and throws the provided exception.

This removes the need for `Optional.isPresent()` checks in both the service and the controller.

## 11. FILE 6 — Updated StudentController.java

With exceptions being thrown by the service and caught by `GlobalExceptionHandler`, the controller becomes much cleaner. There are no more `if (result.isPresent())` checks.

```java
package com.college.studentcrud.controller;

import com.college.studentcrud.model.Student;
import com.college.studentcrud.service.StudentService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.DeleteMapping;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.PutMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;
import java.util.List;

@RestController
@RequestMapping("/api/students")
public class StudentController {

    private final StudentService studentService;

    @Autowired
    public StudentController(StudentService studentService) {
        this.studentService = studentService;
    }

    @GetMapping
    public ResponseEntity<List<Student>> getAllStudents() {
        return ResponseEntity.ok(studentService.getAllStudents());
    }

    // No Optional check — if student not found, service throws
    // StudentNotFoundException, GlobalExceptionHandler catches it
    @GetMapping("/{id}")
    public ResponseEntity<Student> getStudentById(@PathVariable int id) {
        return ResponseEntity.ok(studentService.getStudentById(id));
    }

    @GetMapping("/search")
    public ResponseEntity<List<Student>> getByCourse(@RequestParam String course) {
        return ResponseEntity.ok(studentService.getStudentsByCourse(course));
    }

    @GetMapping("/searchByName")
    public ResponseEntity<List<Student>> searchByName(@RequestParam String name) {
        return ResponseEntity.ok(studentService.searchByName(name));
    }

    @GetMapping("/count")
    public ResponseEntity<String> getCount() {
        return ResponseEntity.ok("Total students: " + studentService.getTotalCount());
    }

    @PostMapping
    public ResponseEntity<Student> addStudent(@RequestBody Student student) {
        return ResponseEntity.status(201).body(studentService.addStudent(student));
    }

    // No Optional check — if student not found, service throws exception
    @PutMapping("/{id}")
    public ResponseEntity<Student> updateStudent(
            @PathVariable int id,
            @RequestBody Student updatedData) {
        return ResponseEntity.ok(studentService.updateStudent(id, updatedData));
    }

    // No boolean return — if student not found, service throws exception
    @DeleteMapping("/{id}")
    public ResponseEntity<String> deleteStudent(@PathVariable int id) {
        studentService.deleteStudent(id);
        return ResponseEntity.ok("Student with ID " + id + " deleted successfully.");
    }
}
```

### Controller Before vs After

| Aspect | Class 3 (before) | Class 4 (after) |
|---|---|---|
| getById() | Checks Optional.isPresent() | Just calls service, no check |
| updateStudent() | Checks Optional.isPresent() | Just calls service, no check |
| deleteStudent() | Checks boolean return | Just calls service, no check |
| Error handling | Spread across all methods | Centralized in GlobalExceptionHandler |
| Lines of code | More | Fewer and cleaner |
| Responsibility | Mixed — request + error | Only request and response |

The controller now has a single responsibility — receive the request, call the service, return the response. All error handling is in one place.

## 13. @ResponseStatus — Alternative Approach

Instead of returning `ResponseEntity`, you can annotate the exception class itself with `@ResponseStatus` to associate a status code with it permanently.

```java
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.annotation.ResponseStatus;

@ResponseStatus(HttpStatus.NOT_FOUND)
public class StudentNotFoundException extends RuntimeException {

    public StudentNotFoundException(int id) {
        super("Student with ID " + id + " does not exist.");
    }
}
```

When Spring Boot throws this exception and no `@ExceptionHandler` is present, it automatically uses the status code from `@ResponseStatus`.

However, `@ResponseStatus` alone does not return a custom JSON body. It only sets the HTTP status code. For a structured JSON error response (with message and timestamp), `@ControllerAdvice` with `@ExceptionHandler` is the proper approach.

| Feature | @ResponseStatus | @ControllerAdvice |
|---|---|---|
| Sets HTTP status | Yes | Yes |
| Returns custom JSON body | No | Yes |
| Centralized handling | No | Yes |
| Recommended for production | No | Yes |

| Code | Name | When to Use |
|---|---|---|
| 200 | OK | Successful GET, PUT, DELETE with response body |

## 16. Questions

**Q1. What is @ControllerAdvice? What is its purpose?**

Ans: `@ControllerAdvice` is a Spring annotation used to create a global exception handler class that applies to all controllers in the application. Without it, exception handling code would need to be duplicated in every controller. With `@ControllerAdvice`, all exceptions thrown from any controller are intercepted by this single class. It is used together with `@ExceptionHandler` methods to catch specific exception types and return structured JSON error responses with appropriate HTTP status codes.

---

**Q2. What is the difference between @ExceptionHandler and @ControllerAdvice?**

Ans: `@ExceptionHandler` is a method-level annotation that handles a specific exception. When placed inside a controller class, it handles exceptions only from that controller. When placed inside a class annotated with `@ControllerAdvice`, it handles that exception from all controllers in the application. For a real application, `@ExceptionHandler` methods should be placed in a `@ControllerAdvice` class so that error handling is centralized and not repeated in every controller.

---

**Q3. What is orElseThrow()? How is it used in exception handling?**

Ans: `orElseThrow()` is a method on `Optional`. If the Optional contains a value, it returns that value. If the Optional is empty, it executes the provided lambda expression and throws the specified exception. It is used in the service layer to eliminate verbose `Optional.isPresent()` checks:

```java
return studentRepository.findById(id)
        .orElseThrow(() -> new StudentNotFoundException(id));
```

This reads as: return the student if found, otherwise throw `StudentNotFoundException`. The thrown exception is then caught by `@ControllerAdvice`.

---

**Q4. Why should a custom exception extend RuntimeException?**

Ans: In Java, exceptions that extend `RuntimeException` are unchecked exceptions, meaning the method that throws them does not need to declare them with a `throws` clause in the method signature. This keeps the service and controller code clean. Checked exceptions (extending `Exception`) require every method in the call stack to either catch them or declare them, which adds unnecessary boilerplate. For custom application exceptions like `StudentNotFoundException`, `RuntimeException` is the correct base class.

---

**Q5. What is the difference between @ResponseStatus and @ControllerAdvice for exception handling?**

Ans: `@ResponseStatus` is placed on an exception class and associates a fixed HTTP status code with it. When that exception is thrown and not handled by any `@ExceptionHandler`, Spring uses the status code from `@ResponseStatus`. However, it does not return a custom JSON body — the client receives only the status code. `@ControllerAdvice` with `@ExceptionHandler` provides complete control — you can set the HTTP status code, construct a custom error response object, and return it as JSON. For production APIs where the client needs structured error information, `@ControllerAdvice` is the correct and recommended approach.

---

**Q6. What HTTP status code should be returned for each error scenario?**

Ans:

| Scenario | Status Code | Reason |
|---|---|---|
| Resource not found | 404 Not Found | The requested ID does not exist |
| Duplicate data | 409 Conflict | Data violates uniqueness constraint |
| Invalid JSON body | 400 Bad Request | Client sent malformed input |
| Unexpected server error | 500 Internal Server Error | Unhandled exception on server |
| Successful creation | 201 Created | New resource was inserted |
| Successful update | 200 OK | Resource was modified |
