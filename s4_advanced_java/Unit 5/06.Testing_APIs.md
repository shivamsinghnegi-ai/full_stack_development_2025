## 3. Setting Up Postman

### What is Postman

Postman is a tool used to send HTTP requests to an API and view the response. It is the standard tool used in industry for testing REST APIs. A browser can only send GET requests. Postman can send GET, POST, PUT, DELETE, and all other HTTP methods.

### Setting Up a Request in Postman

For every request:

| Setting | Where to Set | Value |
|---|---|---|
| Method | Dropdown next to URL | GET / POST / PUT / DELETE |
| URL | URL bar | http://localhost:8080/api/students |
| Header | Headers tab | Content-Type: application/json |
| Body | Body tab → raw → JSON | JSON data |

The `Content-Type: application/json` header is required for POST and PUT requests. Without it, Spring Boot does not know the body is JSON and returns 415 Unsupported Media Type.

## 6. REST API Naming Best Practices

These are standard conventions used in the industry. They are also asked in theory exams.

### Rule 1 — Use Nouns, Not Verbs in URLs

The HTTP method already expresses the action. The URL should only identify the resource.

```
Incorrect — verb in URL:
  GET  /getStudents
  POST /createStudent
  PUT  /updateStudent/1
  DELETE /deleteStudent/1

Correct — noun in URL:
  GET    /students
  POST   /students
  PUT    /students/1
  DELETE /students/1
```

### Rule 2 — Use Plural Nouns

```
Incorrect: /student
Correct:   /students
```

### Rule 3 — Use Lowercase Letters

```
Incorrect: /Students
Incorrect: /STUDENTS
Correct:   /students
```

### Rule 4 — Use Hyphens for Multi-Word Resources

```
Incorrect: /studentCourses       (camelCase)
Incorrect: /student_courses      (underscore)
Correct:   /student-courses      (hyphen)
```

### Rule 5 — Use Nested URLs for Relationships

When one resource belongs to another:

```
GET  /students/1/courses        All courses of student 1
GET  /students/1/courses/3      Course 3 of student 1
POST /students/1/courses        Add a course to student 1
```

### Rule 6 — Always Return Correct HTTP Status Codes

An API that always returns 200 even for errors is a badly designed API. Status codes tell the client what happened without needing to read the response body.

```
200 — operation succeeded, data returned
201 — new resource created
204 — operation succeeded, nothing to return
400 — client sent bad input
404 — resource does not exist
409 — conflict, duplicate data
500 — server-side error
```

### Summary Table

| Practice | Incorrect | Correct |
|---|---|---|
| Verbs in URL | /getStudents | /students |
| Plural nouns | /student | /students |
| Lowercase | /Students | /students |
| Hyphens | /studentList | /student-list |
| Status codes | Always 200 | Appropriate code |

## 9. Questions

**Q1. What is the difference between JDBC Template and Spring Data JPA?**

Ans: JDBC Template is a Spring class that simplifies plain JDBC by removing connection management and exception handling boilerplate. You still write SQL queries and map ResultSet to objects manually. Spring Data JPA is built on top of JPA and Hibernate. It removes the need to write SQL for basic CRUD operations by generating implementation from interface method names and mapping Java classes to database tables using annotations. JDBC Template gives more control over SQL. Spring Data JPA reduces boilerplate significantly and is preferred for standard applications.

---

**Q2. Explain @Entity and its related annotations.**

Ans: `@Entity` is placed on a Java class to indicate that it represents a database table. Related annotations:
- `@Table(name)` — sets the table name
- `@Id` — marks the primary key field
- `@GeneratedValue(strategy = GenerationType.IDENTITY)` — uses AUTO_INCREMENT for the primary key
- `@Column(nullable, unique, length)` — defines column constraints

When the application starts with `ddl-auto=update`, Hibernate reads all `@Entity` classes and creates the corresponding tables automatically.

---

**Q3. What is JpaRepository? List any five methods it provides.**

Ans: `JpaRepository` is an interface in Spring Data JPA. When a repository interface extends `JpaRepository<EntityType, PrimaryKeyType>`, Spring generates a complete implementation at runtime. Five methods provided:
1. `findAll()` — retrieves all records
2. `findById(id)` — retrieves one record by primary key, returns Optional
3. `save(entity)` — inserts if id is null/0, updates if id exists
4. `deleteById(id)` — deletes by primary key
5. `count()` — returns total number of records

---

**Q4. What is @ControllerAdvice? How does it work with @ExceptionHandler?**

Ans: `@ControllerAdvice` marks a class as a global exception handler that applies to all controllers in the application. Inside this class, `@ExceptionHandler(ExceptionClass.class)` methods are defined for each exception type that needs to be handled. When any controller or service throws an exception, Spring intercepts it, finds the matching `@ExceptionHandler` method in the `@ControllerAdvice` class, runs it, and returns the result as the HTTP response. This centralizes all error handling in one place instead of repeating it across multiple controllers.

---

**Q5. Explain the 3-layer architecture in Spring Boot with an example.**

Ans: Spring Boot applications follow a 3-layer architecture where each layer has one responsibility:

Controller layer (`@RestController`) — receives HTTP requests and sends responses. It calls the service and returns `ResponseEntity`. It does not contain business logic or database code.

Service layer (`@Service`) — contains business logic. It validates input, applies rules, and calls the repository. It throws exceptions when something goes wrong.

Repository layer (extends JpaRepository) — communicates with the database. It runs queries using JPA methods.

Example flow for GET /api/students/1: The controller receives the request and calls `studentService.getStudentById(1)`. The service calls `studentRepository.findById(1)`. The repository runs `SELECT * FROM students WHERE id = 1`. The result travels back to the controller, which returns `ResponseEntity.ok(student)`.

---

**Q7. Why is constructor injection preferred over field injection in Spring Boot?**

Ans: Constructor injection is preferred because:
1. Dependencies are declared as `final` — they cannot be accidentally changed after injection
2. The class cannot be created without its dependencies — prevents NullPointerException
3. Dependencies are visible in the constructor signature — improves code readability
4. Easier to write unit tests — dependencies can be passed directly without a Spring context

Field injection (using `@Autowired` directly on the field) works but makes testing harder and hides dependencies, which is why the Spring team recommends constructor injection.

---

**Q8. What is the purpose of application.properties in Spring Boot?**

Ans: `application.properties` is the central configuration file in a Spring Boot project. It replaces all the configuration that used to be scattered across XML files and hardcoded in Java classes. Key configurations it contains:
- Database connection URL, username, and password (replaces `DriverManager.getConnection()`)
- Server port
- JPA and Hibernate settings like `ddl-auto`, `show-sql`, and dialect
- Application name and custom properties

Spring Boot reads this file at startup and auto-configures components accordingly — database connection pool, Hibernate, embedded Tomcat — without any manual Java configuration code.
