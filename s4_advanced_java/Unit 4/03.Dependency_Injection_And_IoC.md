## 2. Let's Start With a Real Problem You Have Faced

You have written code like this in your Servlet days:

```java
public class StudentServlet extends HttpServlet {

    StudentDAO dao = new StudentDAO();   // creating object directly

    protected void doGet(HttpServletRequest req, HttpServletResponse res) {
        List<Student> list = dao.getAllStudents();
        // ...
    }
}
```

This looks fine. But there are **hidden problems** here.

### Problem 1 — Tight Coupling

`StudentServlet` is directly creating a `StudentDAO` object using `new`.

This means:
- `StudentServlet` is **tightly coupled** (strongly connected) to `StudentDAO`
- If tomorrow you create `StudentDAOv2` (better version), you must go into `StudentServlet` and change the code
- If you have 10 Servlets using `StudentDAO`, you must change all 10 files
- This violates a software design rule called **"Open/Closed Principle"** — a class should be open for extension but closed for modification

### Problem 2 — Hard to Test

When you write unit tests for `StudentServlet`, it automatically creates a real `StudentDAO` object, which tries to connect to a real database.

You can't test the Servlet logic alone — it always needs the database. This makes testing slow and complicated.

### Problem 3 — No Control Over Object Lifecycle

When you write `new StudentDAO()`, YOU are responsible for:
- When the object is created
- How many objects are created (every Servlet creates its own copy)
- When the object is destroyed

This leads to **memory waste** and **inconsistency**.

## 3. What is a Dependency?

Before understanding DI, you must understand what a **Dependency** means.

**Simple Definition:**
> When Class A uses Class B, we say "Class A is dependent on Class B" or "Class B is a dependency of Class A."

**Example:**
```java
public class StudentServlet {
    StudentDAO dao = new StudentDAO();  // StudentDAO is a dependency of StudentServlet
}
```

Here:
- `StudentServlet` = **dependent class**
- `StudentDAO` = **dependency**

The problem is that `StudentServlet` is creating the dependency itself (using `new`).

---

## 4. What is Inversion of Control (IoC)?

### The Old Way (You are in control)

In traditional Java, **YOU** (the developer) are responsible for:
- Creating objects (`new StudentDAO()`)
- Managing their lifecycle
- Passing them where needed

The **control of object creation** is with YOU.

### IoC — Flip the Control

**Inversion of Control** means:
> Instead of YOU creating and managing objects, you give that responsibility to a **framework (Spring)**.

The **control is inverted (flipped)** — from you to Spring.

**Simple Real-Life Analogy:**

| Without IoC | With IoC |
|---|---|
| You cook your own food | You order from a restaurant |
| You manage everything | Restaurant manages everything |
| You are in control | Restaurant (Spring) is in control |

You just say "I need food (StudentDAO)" — Spring prepares and delivers it to you.

### Technical Definition for Exam:
> Inversion of Control (IoC) is a design principle where the control of object creation and lifecycle management is transferred from the programmer to the Spring framework.

---

## 5. What is the IoC Container?

The IoC Container is the **heart of Spring Framework**.

**Definition:**
> The IoC Container is a component of Spring that is responsible for creating objects, managing their lifecycle, and injecting dependencies.

Think of the IoC Container as a **big factory** that:
1. Reads your code (annotations or XML)
2. Creates all the required objects
3. Stores them
4. Injects (provides) them wherever needed
5. Destroys them when no longer needed

### How does Spring know what to create?

You tell Spring by adding **annotations** on your classes:

| Annotation | Meaning |
|---|---|
| `@Component` | "Hey Spring, please manage this class" |
| `@Service` | Same as @Component — used for business logic classes |
| `@Repository` | Same as @Component — used for database classes |
| `@Controller` / `@RestController` | Same as @Component — used for web/API classes |

When Spring Boot starts, the IoC Container scans all these annotations and creates objects automatically.

### What is ApplicationContext?

`ApplicationContext` is the **interface** that represents the IoC Container in Spring.

```java
// You rarely write this yourself — Spring Boot does it internally
ApplicationContext context = SpringApplication.run(StudentappApplication.class, args);
```

When `SpringApplication.run()` is called, it creates the `ApplicationContext` (IoC Container) automatically.

---

## 6. What is a Bean?

**Definition:**
> A Bean is simply a Java object that is created and managed by the Spring IoC Container.

**Normal Java Object:**
```java
StudentDAO dao = new StudentDAO();  // YOU created this — it is NOT a bean
```

**Spring Bean:**
```java
@Component
public class StudentDAO {           // Spring created this — it IS a bean
    // ...
}
```

The difference is simple:
- If YOU create it with `new` → it is a normal object
- If **Spring** creates and manages it → it is a **Bean**

### Bean Naming Convention

By default, Spring names the bean after the class name but with a **lowercase first letter**.

| Class Name | Default Bean Name |
|---|---|
| `StudentDAO` | `studentDAO` |
| `StudentService` | `studentService` |
| `HelloController` | `helloController` |

## 7. What is Dependency Injection (DI)?

Now that you understand IoC and Beans, DI is easy.

**Definition:**
> Dependency Injection is a technique where the Spring IoC Container automatically provides (injects) the required dependencies into a class instead of the class creating them itself.

**Simple Analogy:**  
Instead of going to the shop to buy groceries yourself (creating with `new`), someone delivers the groceries to your door (Spring injects the dependency).

**Without DI (old way):**
```java
public class StudentService {
    // YOU are creating the dependency
    StudentDAO dao = new StudentDAO();
}
```

**With DI (Spring way):**
```java
@Service
public class StudentService {
    // Spring INJECTS the dependency for you
    @Autowired
    StudentDAO dao;   // no "new" keyword needed
}
```

You just declare what you need — Spring finds the right bean and gives it to you.

---

## 8. Types of Dependency Injection

There are **3 types** of DI in Spring. All 3 are important for exams.

---

### Type 1 — Field Injection (using @Autowired on field)

The dependency is injected directly into the **field (variable)**.

```java
@Service
public class StudentService {

    @Autowired                      // Spring injects StudentDAO here
    private StudentDAO studentDAO;

    public List<Student> getAllStudents() {
        return studentDAO.findAll();
    }
}
```

**How it works:**
- Spring sees `@Autowired` on the field
- It looks inside the IoC Container for a bean of type `StudentDAO`
- It finds it (because `StudentDAO` has `@Repository`)
- It sets the value of `studentDAO` automatically

**Pros:** Simple, less code  
**Cons:** Hard to test, not recommended for production code  
**When used:** Quick prototyping, small projects

---

### Type 2 — Constructor Injection (Recommended)

The dependency is injected through the **constructor**.

```java
@Service
public class StudentService {

    private final StudentDAO studentDAO;  // final — cannot be changed after injection

    @Autowired                            // Spring calls this constructor
    public StudentService(StudentDAO studentDAO) {
        this.studentDAO = studentDAO;     // Spring passes the bean here
    }

    public List<Student> getAllStudents() {
        return studentDAO.findAll();
    }
}
```

**How it works:**
- Spring sees `@Autowired` on the constructor
- When creating `StudentService` bean, it calls this constructor
- It automatically passes the `StudentDAO` bean as argument

**Pros:**
- Dependencies are clear and visible
- Field is `final` — cannot be accidentally changed
- Easy to test (can pass mock objects in tests)
- **Recommended by Spring team**

**Cons:** Slightly more code

> **Exam Tip:** If asked "Which type of DI is recommended?", answer is **Constructor Injection**.

> **Note:** In newer Spring versions, if a class has only ONE constructor, `@Autowired` is optional — Spring automatically uses it.

---

### Type 3 — Setter Injection

The dependency is injected through a **setter method**.

```java
@Service
public class StudentService {

    private StudentDAO studentDAO;

    @Autowired                            // Spring calls this setter
    public void setStudentDAO(StudentDAO studentDAO) {
        this.studentDAO = studentDAO;
    }

    public List<Student> getAllStudents() {
        return studentDAO.findAll();
    }
}
```

**How it works:**
- Spring creates `StudentService` bean (calls default constructor)
- Then calls the `setStudentDAO()` method and passes the `StudentDAO` bean

**Pros:** Dependency can be changed later (optional dependencies)  
**Cons:** Object can be used before dependency is set (risky)  
**When used:** Optional dependencies, legacy code

---

### Comparison of All 3 Types

| Feature | Field Injection | Constructor Injection | Setter Injection |
|---|---|---|---|
| Code | Least code | More code | Medium code |
| Recommended? | No | **Yes** | Sometimes |
| Can be final? | No | **Yes** | No |
| Testing | Difficult | **Easy** | Medium |
| Null safety | Risk | **Safe** | Risk |
| Spring default? | Works | **Preferred** | Works |

## 9. @Autowired — Explained in Detail

`@Autowired` is the annotation that tells Spring: **"Please inject the dependency here."**

Spring searches for a matching bean in the IoC Container by **type** (class name).

```java
@Autowired
private StudentDAO studentDAO;
```

Spring looks for a bean of type `StudentDAO` → finds the `@Repository` annotated class → injects it.

### What if Spring finds NO matching bean?

You get an error:
```
NoSuchBeanDefinitionException: No qualifying bean of type 'StudentDAO'
```

**Fix:** Make sure `StudentDAO` has `@Repository` or `@Component` annotation.

### What if Spring finds MULTIPLE matching beans?

You get an error:
```
NoUniqueBeanDefinitionException: expected single matching bean but found 2
```

**Fix:** Use `@Qualifier` to specify which bean to inject (covered below).

## 10. Stereotype Annotations — @Component, @Service, @Repository, @Controller

These 4 annotations all do the **same thing** — they tell Spring to create and manage a bean.

But they are used in **different layers** of the application to make code more readable and organized.

| Annotation | Used For | Layer |
|---|---|---|
| `@Component` | Generic Spring-managed class | Any layer |
| `@Service` | Business logic / service layer | Service layer |
| `@Repository` | Database operations (DAO) | Data layer |
| `@Controller` | Handle web requests (returns views) | Web layer |
| `@RestController` | Handle REST API requests (returns JSON) | Web layer |

### Example — Layered Architecture

```java
// Web Layer — handles HTTP requests
@RestController
public class StudentController {
    @Autowired
    StudentService service;
}

// Service Layer — contains business logic
@Service
public class StudentService {
    @Autowired
    StudentRepository repository;
}

// Data Layer — talks to the database
@Repository
public class StudentRepository {
    // database methods
}
```

This is called **3-Layer Architecture** and it is the standard way to build Spring Boot apps.

```
[ @RestController ]   ← receives HTTP request
        ↓
[ @Service ]          ← processes business logic
        ↓
[ @Repository ]       ← talks to database
        ↓
[ Database ]
```

> **Exam Tip:** This 3-layer architecture diagram is very commonly asked. Memorize it.

## 11. @Qualifier — When Multiple Beans Exist

Sometimes you have **multiple implementations** of the same interface.

```java
public interface PaymentService {
    void pay();
}

@Service("creditCardService")
public class CreditCardPaymentService implements PaymentService {
    public void pay() { System.out.println("Paid by Credit Card"); }
}

@Service("upiService")
public class UPIPaymentService implements PaymentService {
    public void pay() { System.out.println("Paid by UPI"); }
}
```

Now if you write:
```java
@Autowired
private PaymentService paymentService;  // Spring is confused — which one?
```

Spring doesn't know which one to inject. Use `@Qualifier` to specify:

```java
@Autowired
@Qualifier("upiService")               // tell Spring which one to use
private PaymentService paymentService;
```

## 12. Bean Scope — How Many Times is a Bean Created?

By default, Spring creates only **ONE instance** of each bean and reuses it everywhere. This is called **Singleton** scope.

But you can change this using `@Scope` annotation.

| Scope | Meaning | How many objects? |
|---|---|---|
| `singleton` | One instance for entire application | **1 object** (default) |
| `prototype` | New instance every time it is requested | New object each time |
| `request` | One instance per HTTP request (web only) | 1 per request |
| `session` | One instance per HTTP session (web only) | 1 per session |

### Example

```java
// Default — singleton (one object shared everywhere)
@Service
public class StudentService { }

// Prototype — new object every time
@Service
@Scope("prototype")
public class StudentService { }
```

> **Exam Tip:** Default scope of a Spring bean is **Singleton**. This is a very common MCQ.

## 13. Bean Lifecycle — Birth to Death of a Bean

Every Spring Bean goes through a lifecycle:

```
1. Bean Definition Read
   (Spring reads @Component/@Service/@Repository annotations)
          ↓
2. Bean Instantiation
   (Spring creates the object — calls constructor)
          ↓
3. Dependency Injection
   (Spring injects @Autowired dependencies)
          ↓
4. @PostConstruct Method
   (custom initialization code runs — optional)
          ↓
5. Bean is Ready
   (Bean is fully initialized and available for use)
          ↓
6. Application Running...
          ↓
7. @PreDestroy Method
   (cleanup code runs before bean is destroyed — optional)
          ↓
8. Bean Destroyed
   (IoC Container shuts down)
```

### @PostConstruct and @PreDestroy

```java
@Service
public class StudentService {

    @PostConstruct                          // runs AFTER bean is created and dependencies injected
    public void init() {
        System.out.println("StudentService is ready!");
    }

    @PreDestroy                             // runs BEFORE bean is destroyed
    public void cleanup() {
        System.out.println("StudentService is shutting down!");
    }
}
```

> **Exam Tip:** `@PostConstruct` runs after DI is complete. `@PreDestroy` runs before bean is destroyed.

## 14. Complete Working Example — Putting It All Together

Let's build a small Student app that uses all concepts from today:

### Step 1 — Create the Model (Student.java)

```java
package com.college.studentapp.model;

public class Student {
    private int id;
    private String name;
    private String email;

    // Constructor
    public Student(int id, String name, String email) {
        this.id = id;
        this.name = name;
        this.email = email;
    }

    // Getters
    public int getId() { return id; }
    public String getName() { return name; }
    public String getEmail() { return email; }
}
```

---

### Step 2 — Create the Repository (StudentRepository.java)

```java
package com.college.studentapp.repository;

import com.college.studentapp.model.Student;
import org.springframework.stereotype.Repository;
import java.util.Arrays;
import java.util.List;

@Repository                             // tells Spring: manage this as a bean
public class StudentRepository {

    // Dummy data (no DB needed for now)
    public List<Student> findAll() {
        return Arrays.asList(
            new Student(1, "Ravi", "ravi@gmail.com"),
            new Student(2, "Priya", "priya@gmail.com"),
            new Student(3, "Amit", "amit@gmail.com")
        );
    }
}
```

---

### Step 3 — Create the Service (StudentService.java)

```java
package com.college.studentapp.service;

import com.college.studentapp.model.Student;
import com.college.studentapp.repository.StudentRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.List;

@Service                                // tells Spring: manage this as a bean
public class StudentService {

    private final StudentRepository studentRepository;

    @Autowired                          // Constructor Injection (recommended)
    public StudentService(StudentRepository studentRepository) {
        this.studentRepository = studentRepository;
        // Spring automatically passes the StudentRepository bean here
    }

    public List<Student> getAllStudents() {
        return studentRepository.findAll();     // calls repository
    }
}
```

---

### Step 4 — Create the Controller (StudentController.java)

```java
package com.college.studentapp.controller;

import com.college.studentapp.model.Student;
import com.college.studentapp.service.StudentService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import java.util.List;

@RestController                         // tells Spring: manage this as a bean + handle REST requests
public class StudentController {

    private final StudentService studentService;

    @Autowired                          // Constructor Injection
    public StudentController(StudentService studentService) {
        this.studentService = studentService;
    }

    @GetMapping("/students")            // URL: http://localhost:8080/students
    public List<Student> getStudents() {
        return studentService.getAllStudents();
    }
}
```

---

### What Happens When You Run This App

```
SpringApplication.run() starts
        ↓
IoC Container is created
        ↓
Spring scans packages, finds:
  - StudentRepository (@Repository) → creates bean
  - StudentService (@Service)       → creates bean, injects StudentRepository
  - StudentController (@RestController) → creates bean, injects StudentService
        ↓
Embedded Tomcat starts on port 8080
        ↓
User opens: http://localhost:8080/students
        ↓
StudentController.getStudents() is called
        ↓
Calls studentService.getAllStudents()
        ↓
Calls studentRepository.findAll()
        ↓
Returns list of students as JSON
```

### Output in Browser

```json
[
  {"id": 1, "name": "Ravi", "email": "ravi@gmail.com"},
  {"id": 2, "name": "Priya", "email": "priya@gmail.com"},
  {"id": 3, "name": "Amit", "email": "amit@gmail.com"}
]
```

## 15. IoC vs DI — What's the Difference?

Students often confuse these two. Here's a clear distinction:

| | IoC | DI |
|---|---|---|
| **What is it?** | A **principle / concept** | A **technique** to implement IoC |
| **What it means** | Give control of object creation to Spring | Spring injects dependencies automatically |
| **Analogy** | "Don't call us, we'll call you" | "We'll bring what you need to you" |
| **Relationship** | IoC is the goal | DI is how IoC is achieved |

**Simple way to remember:**
> IoC is the **idea**. DI is the **implementation** of that idea.

---

## 16. @Component vs @Bean — Another Common Confusion

| | @Component | @Bean |
|---|---|---|
| **Written on** | Class | Method |
| **Used when** | You own the class (you wrote it) | You don't own the class (third-party library) |
| **Auto detection** | Yes (via @ComponentScan) | No (must be in @Configuration class) |

**@Component example (your own class):**
```java
@Component
public class EmailService {
    public void sendEmail() { ... }
}
```

**@Bean example (third-party class you can't modify):**
```java
@Configuration
public class AppConfig {

    @Bean
    public JavaMailSender mailSender() {
        // creating and configuring a third-party class
        JavaMailSenderImpl sender = new JavaMailSenderImpl();
        sender.setHost("smtp.gmail.com");
        return sender;
    }
}
```

## 17. Summary — All Key Concepts at a Glance

| Concept | Simple Meaning |
|---|---|
| **Dependency** | A class that another class needs to work |
| **Tight Coupling** | Class creates its own dependency using `new` |
| **Loose Coupling** | Class receives dependency from outside (Spring) |
| **IoC** | Give Spring the control of creating and managing objects |
| **IoC Container** | Spring's internal factory that creates and manages beans |
| **Bean** | A Java object created and managed by Spring |
| **DI** | Spring automatically injects required beans into your class |
| **@Component** | Generic annotation to mark a class as a Spring bean |
| **@Service** | Marks service/business logic class as a bean |
| **@Repository** | Marks DAO/database class as a bean |
| **@RestController** | Marks web controller class as a bean |
| **@Autowired** | Tells Spring to inject the dependency here |
| **@Qualifier** | Specifies which bean to inject when multiple exist |
| **@Scope** | Controls how many instances of a bean are created |
| **Singleton** | Default scope — one bean instance for whole application |
| **@PostConstruct** | Method runs after bean is fully initialized |
| **@PreDestroy** | Method runs before bean is destroyed |

## 18. Questions

**Q1. What is Inversion of Control (IoC)? Explain with example.**

**Ans:** Inversion of Control (IoC) is a design principle in Spring where the control of creating and managing objects is given to the Spring Framework instead of the developer.

Without IoC:
```java
StudentDAO dao = new StudentDAO();  // developer creates object
```
With IoC:
```java
@Autowired
StudentDAO dao;  // Spring creates and injects the object
```
The control is "inverted" — from developer to Spring.

---

**Q2. What is Dependency Injection? What are its types?**

**Ans:** Dependency Injection is a technique where the Spring IoC Container automatically provides the required dependencies to a class instead of the class creating them itself.

Types of DI:
1. **Field Injection** — `@Autowired` on the field directly
2. **Constructor Injection** — `@Autowired` on constructor (Recommended)
3. **Setter Injection** — `@Autowired` on setter method

Constructor Injection is recommended because it makes dependencies explicit, allows `final` fields, and is easy to test.

---

**Q3. What is a Spring Bean?**

**Ans:** A Spring Bean is a Java object that is created, configured, and managed by the Spring IoC Container. A class is marked as a bean using annotations like `@Component`, `@Service`, `@Repository`, or `@RestController`. Unlike normal Java objects created with the `new` keyword, beans are managed entirely by Spring — their creation, dependency injection, and destruction are all handled by the IoC Container.

---

**Q4. Explain @Component, @Service, @Repository, and @RestController annotations.**

**Ans:** All four annotations are used to declare a class as a Spring Bean (tell Spring to manage the class). The difference is the **layer** they are used in:

| Annotation | Layer | Purpose |
|---|---|---|
| @Component | Any | Generic bean |
| @Service | Service | Business logic |
| @Repository | Data | Database operations |
| @RestController | Web | REST API handling |

---

**Q5. What is the default scope of a Spring Bean? Name all scopes.**

**Ans:** The default scope of a Spring Bean is **Singleton** — meaning Spring creates only one instance of the bean and shares it throughout the application.

Other scopes:
- `prototype` — New instance every time the bean is requested
- `request` — One instance per HTTP request
- `session` — One instance per HTTP session

---

**Q6. Explain the Bean Lifecycle in Spring.**

**Ans:** The Spring Bean goes through the following lifecycle:
1. Bean definition is read (via annotations or XML)
2. Bean is instantiated (constructor is called)
3. Dependencies are injected (@Autowired)
4. `@PostConstruct` method runs (custom initialization)
5. Bean is ready to use
6. Application runs
7. `@PreDestroy` method runs (cleanup)
8. Bean is destroyed (container shuts down)

---

**Q7. What is the difference between IoC and DI?**

**Ans:**
- **IoC (Inversion of Control)** is a **principle** that says object creation and lifecycle management should be handled by the framework, not the developer.
- **DI (Dependency Injection)** is a **technique** that implements IoC by having the Spring container inject required dependencies into classes automatically.
- IoC is the concept; DI is how that concept is achieved in Spring.
