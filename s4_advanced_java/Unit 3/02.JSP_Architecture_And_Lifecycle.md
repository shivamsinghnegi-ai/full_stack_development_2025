# JSP Architecture & Lifecycle

---

## What is JSP?

JSP stands for **Java Server Pages**.

It is a technology that allows you to write **Java code inside HTML** to create dynamic web pages.

**Simple meaning:**
- Normal HTML = shows same content to everyone (static)
- JSP = shows different content based on user, time, data, etc. (dynamic)

**Example:**
```jsp
<h1>Hello, <%= username %>!</h1>   ← shows different name for every user
```

---

## Why JSP? (JSP vs Servlet)

| | Servlet | JSP |
|--|---------|-----|
| Writing style | Java code with HTML inside | HTML with Java code inside |
| Easy for | Backend logic | Designing pages |
| Used for | Controller / Logic | View / Display |
| Code looks like | Difficult to read | Easy to read |

> In real projects, **Servlet handles logic** and **JSP handles display**.
> This is called the **MVC pattern** (covered in Part 5).

# JSP Architecture

---

## How JSP Works — Big Picture

```
User (Browser)
     |
     |  1. Sends Request
     ↓
  Web Server (Tomcat)
     |
     |  2. Finds the .jsp file
     ↓
  JSP Engine (inside Tomcat)
     |
     |  3. Converts JSP → Servlet (.java file)
     |  4. Compiles Servlet → (.class file)
     |  5. Executes the class file
     ↓
  Response (HTML page sent back)
     |
     ↓
User (Browser) sees the final HTML page
```

---

## Key Components of JSP Architecture

### 1. Client (Browser)
- The user who opens the website
- Sends a **HTTP Request** to the server
- Receives a **HTML Response** back

### 2. Web Server (Tomcat)
- Receives the request from browser
- Finds the correct `.jsp` file
- Passes it to the JSP Engine

### 3. JSP Engine
- Lives inside Tomcat
- Responsible for converting JSP into Servlet
- Only converts the **first time** the JSP is requested
- From second request onwards — directly executes compiled class

### 4. Servlet (Generated automatically)
- The JSP engine converts your `.jsp` file into a `.java` (Servlet) file
- You never write or see this file — Tomcat handles it automatically
- This servlet is then compiled into a `.class` file and executed

### 5. Response
- After execution, the output is pure **HTML**
- This HTML is sent back to the browser

---

## Where are the converted files stored?

On Mac (Homebrew Tomcat 9):
```
/opt/homebrew/opt/tomcat@9/libexec/work/
```

On Windows:
```
C:\tomcat9\work\
```

> You can go look inside this folder after running a JSP — you will find the auto-generated `.java` servlet file Tomcat created from your JSP!

## Simple Real Life Analogy

Think of JSP like a **Pizza Order Form**:

```
Customer fills form (Request)
       ↓
Staff reads form (JSP Engine reads .jsp)
       ↓
Kitchen prepares pizza (Servlet executes)
       ↓
Pizza delivered to customer (HTML Response)
```

The customer never sees the kitchen work — they just get the final pizza (HTML page).

# JSP Lifecycle

---

## What is Lifecycle?

Lifecycle means the **stages a JSP goes through** from the moment it is first requested to the moment the server shuts down.

---

## JSP Lifecycle — 5 Stages

```
Stage 1:  Translation       .jsp file → .java (Servlet) file
Stage 2:  Compilation       .java file → .class file
Stage 3:  jspInit()         JSP is initialized (runs once)
Stage 4:  _jspService()     JSP handles every request (runs every time)
Stage 5:  jspDestroy()      JSP is destroyed when server stops (runs once)
```

---

## Stage 1 — Translation

- Tomcat reads your `.jsp` file
- Converts it into a **Servlet `.java` file**
- Happens only **once** — the first time the JSP is requested
- If you change the `.jsp` file → translation happens again

**What gets translated?**

```jsp
<h1>Hello</h1>
<% out.println("World"); %>
```
Gets converted into something like:
```java
out.write("<h1>Hello</h1>\n");
out.println("World");
```

---

## Stage 2 — Compilation

- The generated `.java` servlet file is **compiled** into a `.class` file
- This is done by the Java compiler automatically
- Happens only **once** (same time as translation)
- If translation happens again → compilation happens again

---

## Stage 3 — jspInit()

```java
public void jspInit() {
    // runs once when JSP is first loaded
}
```

- Called **only once** in the entire lifecycle
- Used to set up resources — like database connections, reading config files
- You can **override** this method if you need custom initialization
- If not overridden, Tomcat uses the default empty implementation

---

## Stage 4 — _jspService()

```java
public void _jspService(HttpServletRequest request, 
                         HttpServletResponse response) {
    // runs EVERY TIME a user requests this JSP
}
```

- Called **every time** a user opens the JSP page
- This is where all your JSP code actually runs
- Handles the request and writes the HTML response
- You **cannot override** this method — Tomcat generates it automatically from your JSP content

---

## Stage 5 — jspDestroy()

```java
public void jspDestroy() {
    // runs once when server is shutting down
}
```

- Called **only once** at the end
- Used to release resources — close database connections, etc.
- Runs when Tomcat is stopped or the JSP is removed

---

## Lifecycle Visual Summary

```
First Request
─────────────
.jsp file
   ↓  [Translation - once]
.java file
   ↓  [Compilation - once]
.class file
   ↓  [jspInit() - once]
JSP Ready


Every Request After That
────────────────────────
User opens page
   ↓  [_jspService() - every time]
HTML sent to browser


Server Shutdown
───────────────
   ↓  [jspDestroy() - once]
Resources released
```

---

## Lifecycle Methods — Quick Comparison

| Method | When it runs | How many times | Can you override? |
|--------|-------------|----------------|-------------------|
| `jspInit()` | JSP first loaded | Once | Yes |
| `_jspService()` | Every request | Every time | No |
| `jspDestroy()` | Server stops | Once | Yes |

# Code Examples

---

## Example 1 — Basic JSP Page

Create `src/main/webapp/index.jsp`:

```jsp
<%@ page language="java" contentType="text/html; charset=UTF-8" %>
<!DOCTYPE html>
<html>
<head>
    <title>My First JSP</title>
</head>
<body>
    <h1>Hello from JSP!</h1>
    <p>Current Time: <%= new java.util.Date() %></p>
    <p>Server: <%= application.getServerInfo() %></p>
</body>
</html>
```

**Output in browser:**
```
Hello from JSP!
Current Time: Fri Mar 06 10:30:00 IST 2026
Server: Apache Tomcat/9.0.x
```

---

## Example 2 — Overriding jspInit() and jspDestroy()

```jsp
<%@ page language="java" contentType="text/html; charset=UTF-8" %>

<%!
    // jspInit - runs once when JSP loads
    public void jspInit() {
        System.out.println("JSP Initialized! Resources loaded.");
    }

    // jspDestroy - runs once when server stops
    public void jspDestroy() {
        System.out.println("JSP Destroyed! Resources released.");
    }
%>

<!DOCTYPE html>
<html>
<body>
    <h2>Lifecycle Demo</h2>
    <p>Check your Tomcat terminal for jspInit message!</p>
    <p>Page loaded at: <%= new java.util.Date() %></p>
</body>
</html>
```

> The `System.out.println` messages appear in the **Tomcat terminal** (where you ran `tomcat-start`), not in the browser.

---

## Example 3 — Seeing the Lifecycle in Action

Create `src/main/webapp/lifecycle.jsp`:

```jsp
<%@ page language="java" contentType="text/html; charset=UTF-8" %>

<%!
    int visitCount = 0;   // declaration - shared across all requests

    public void jspInit() {
        System.out.println("=== JSP INIT called - happens only ONCE ===");
    }

    public void jspDestroy() {
        System.out.println("=== JSP DESTROY called - happens only ONCE ===");
    }
%>

<%
    visitCount++;   // scriptlet - runs every request
    System.out.println("_jspService called - visit number: " + visitCount);
%>

<!DOCTYPE html>
<html>
<body>
    <h2>JSP Lifecycle Demo</h2>
    <p>This page has been visited: <b><%= visitCount %></b> times</p>
    <p>Refresh the page and watch the count go up!</p>
    <p>Check terminal to see _jspService running every time.</p>
</body>
</html>
```

**What to observe:**
- Refresh the page → `visitCount` increases each time → `_jspService` runs every time 
- Stop and restart Tomcat → count resets → `jspInit` runs again

# Important Points for Exam

1. JSP is internally converted to a **Servlet** by the container (Tomcat)
2. Translation and Compilation happen only **once** — on first request
3. From second request onwards — Tomcat uses the already compiled `.class` file (faster)
4. `jspInit()` → runs **once** (can override)
5. `_jspService()` → runs **every request** (cannot override)
6. `jspDestroy()` → runs **once** on shutdown (can override)
7. The final output of any JSP is always **pure HTML** — browser never sees Java code
8. JSP Engine is part of the **Web Container** (Tomcat)

---

## Difference: First Request vs Subsequent Requests

| | First Request | Subsequent Requests |
|--|--------------|-------------------|
| Translation | Happens | Skipped |
| Compilation | Happens | Skipped |
| jspInit() | Runs | Skipped |
| _jspService() | Runs | Always runs |
| Speed | Slower | Faster |
