# Development Server, Key Settings, and Chapter Revision


## 1. Django Development Server

**Definition:** The Django development server is a lightweight, built-in web server, started using the runserver command, that allows a developer to run and test a Django project locally without configuring a production-grade web server.

### 1.1 Starting the Server

```
python manage.py runserver
```

By default, this starts the server at `http://127.0.0.1:8000/`.

### 1.2 Specifying a Custom Port or IP

| Command | Effect |
|---|---|
| `python manage.py runserver 8080` | Runs on port 8080 instead of default 8000 |
| `python manage.py runserver 0.0.0.0:8000` | Makes the server accessible from other devices on the same network |
| `python manage.py runserver 127.0.0.1:8000` | Explicitly binds to localhost only (default behavior) |

**Important Note for Exam:** `127.0.0.1` (also called localhost) refers to the local machine itself, meaning the server is only accessible from the same computer. Using `0.0.0.0` binds the server to all available network interfaces, making it reachable from other devices on the same local network — useful for testing on a mobile device during development.

### 1.3 Characteristics of the Development Server

- Automatically reloads when Python source code changes are detected (auto-reload feature).
- Displays detailed error tracebacks in the browser when DEBUG is True.
- Single-threaded by default and not optimized for handling concurrent production traffic.
- Not intended for production deployment under any circumstances.

**Important Definition (Quotable):** The Django documentation explicitly states that runserver is intended only for local development and testing; it has not gone through extensive security review and should never be used in a production environment.

### 1.4 Common Errors While Running the Server

| Error | Likely Cause |
|---|---|
| Port already in use | Another process is already using the specified port; use a different port or stop the other process |
| ModuleNotFoundError | Virtual environment not activated, or Django not installed in the current environment |
| TemplateDoesNotExist | Incorrect template path, or TEMPLATES/APP_DIRS misconfigured |
| You have unapplied migrations warning | Migrations exist but have not been applied using migrate (relevant from Chapter 3 onward) |

## 2. DEBUG Setting

**Definition:** DEBUG is a Boolean setting in settings.py that controls whether Django displays detailed, technical error pages (including full tracebacks and configuration details) when an exception occurs.

### 2.1 Behavior

| DEBUG Value | Behavior on Error |
|---|---|
| True | Full technical error page shown, including traceback, request/settings details |
| False | Generic error page shown (e.g., a plain 500 or 404 page), unless custom error templates are defined |

### 2.2 Why DEBUG=True Is Dangerous in Production

- Exposes internal file paths, installed apps, and configuration values.
- Reveals parts of the source code through tracebacks.
- Can expose environment variables and settings if not carefully handled.

**Important Note for Exam:** DEBUG=True is meant strictly for local development. A frequently tested exam point is: "What security risk arises from leaving DEBUG=True in production?" — answer should mention exposure of sensitive configuration and source code details to any user who triggers an error.

## 3. ALLOWED_HOSTS Setting

**Definition:** ALLOWED_HOSTS is a list in settings.py that specifies which host/domain names the Django application is permitted to serve; it is a security measure against HTTP Host header attacks.

### 3.1 Behavior

- If DEBUG is True and ALLOWED_HOSTS is empty, Django allows requests from 'localhost', '127.0.0.1', and '[::1]' automatically.
- If DEBUG is False, ALLOWED_HOSTS must be explicitly set, or Django will refuse all requests and raise an error.

### 3.2 Example

```python
ALLOWED_HOSTS = ['127.0.0.1', 'localhost', 'example.com']
```

**Important Definition (Quotable):** ALLOWED_HOSTS protects against HTTP Host header attacks by ensuring that Django only responds to requests that declare one of the explicitly permitted hostnames; without this check, an attacker could manipulate the Host header to trick the application into generating malicious links or cache poisoning.

## 4. TEMPLATES Setting 

this section briefly reconnects the TEMPLATES setting to the server/settings context.

| Key | Recap |
|---|---|
| BACKEND | Defines which template engine Django should use |
| DIRS | Additional project-level directories Django should search for templates |
| APP_DIRS | If True, enables Django to look inside each app's own templates/ folder |

**Exam Tip:** A question may combine settings.py concepts - for example, asking to explain DEBUG, ALLOWED_HOSTS, and TEMPLATES together as "three important settings.py configurations." Be ready to answer all three concisely in one response.

## 5. End-to-End Mini Practice: URL to View to Template

### 5.1 Scenario

Build a simple "Student Dashboard" page that displays a welcome message, a list of enrolled subjects, and the total subject count.

### 5.2 Step 1: Project and App Setup

```
django-admin startproject campus
cd campus
python manage.py startapp dashboard
```

Add the app to INSTALLED_APPS in settings.py:
```python
INSTALLED_APPS = [
    ...
    'dashboard',
]
```

### 5.3 Step 2: View (dashboard/views.py)

```python
from django.shortcuts import render

def student_dashboard(request):
    context = {
        'student_name': 'Vikas',
        'subjects': ['Computer Networks', 'DAA', 'Python with Django'],
        'total_subjects': 3,
    }
    return render(request, 'dashboard/student_dashboard.html', context)
```

### 5.4 Step 3: App-Level URLs (dashboard/urls.py)

```python
from django.urls import path
from . import views

urlpatterns = [
    path('dashboard/', views.student_dashboard, name='student_dashboard'),
]
```

### 5.5 Step 4: Project-Level URLs (campus/urls.py)

```python
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('', include('dashboard.urls')),
]
```

### 5.6 Step 5: Template (dashboard/templates/dashboard/student_dashboard.html)

```html
<h1>Welcome, {{ student_name }}</h1>
<p>Total Subjects Enrolled: {{ total_subjects }}</p>

<ul>
{% for subject in subjects %}
    <li>{{ subject }}</li>
{% endfor %}
</ul>
```

### 5.7 Step 6: Run the Server

```
python manage.py runserver
```

Visit `http://127.0.0.1:8000/dashboard/` to view the result.

### 5.8 Tracing the Complete Flow

1. Browser sends GET request to `/dashboard/`.
2. Django checks ROOT_URLCONF (campus/urls.py), matches empty prefix, delegates to dashboard.urls via include().
3. dashboard/urls.py matches `dashboard/` to the student_dashboard view.
4. View builds a context dictionary and calls render().
5. Template engine locates student_dashboard.html (found via APP_DIRS since it is inside dashboard/templates/dashboard/).
6. Template tags ({% for %}) and variables ({{ }}) are processed using the context.
7. Final HTML is wrapped in an HttpResponse and returned to the browser.

**This exact flow — URL to view to template — is the single most important integrated concept of Chapter 2 and should be practiced until it can be written from memory.**
