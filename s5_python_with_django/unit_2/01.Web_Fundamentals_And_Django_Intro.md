## 1. Client-Server Architecture

Client-server architecture is the foundational model on which the entire web operates. Every website, web application, and API call follows this pattern.

**Definition:** Client-server architecture is a distributed computing model in which tasks are divided between service providers, called servers, and service requesters, called clients, who communicate over a network.

### 1.1 Components

**Client**
- The client is the requesting device or program (browser, mobile app, or any software that sends a request).
- The client initiates communication.
- Examples: Chrome browser, Postman, a mobile app.

**Server**
- The server is a program or system that listens for requests and provides responses.
- The server processes the request and returns the appropriate resource (HTML page, JSON data, file, etc.).
- Examples: Apache, Nginx, Django development server.

**Network**
- The medium (usually the Internet) through which client and server communicate.
- Communication happens using protocols, primarily HTTP/HTTPS.

### 1.2 How It Works

1. Client sends a request to the server (typically by entering a URL or clicking a link).
2. Request travels over the network to the server.
3. Server processes the request (may involve reading a database, running logic, rendering a template).
4. Server sends back a response (HTML, JSON, file, or error).
5. Client (browser) renders or uses the response.

### 1.3 Why Client-Server Model Is Used

- Centralized data management: all data resides on the server, making updates and maintenance easier.
- Scalability: multiple clients can be served by a single server infrastructure.
- Security: sensitive logic and data stay on the server, not exposed on the client side.
- Separation of concerns: client handles presentation, server handles processing and storage.

**Important Note for Exam:** Client-server architecture is different from peer-to-peer architecture, where each node acts as both client and server. This distinction is a common short-answer question.

## 2. HTTP Protocol

**Definition:** HTTP (HyperText Transfer Protocol) is an application-layer protocol used for transmitting hypermedia documents, such as HTML, between a client and a server. It is the foundation of data communication on the World Wide Web.

### 2.1 Key Characteristics of HTTP

- Stateless: each request is independent; the server does not retain information about previous requests by default.
- Text-based: HTTP messages are human-readable (plain text format), which makes debugging easier.
- Uses TCP/IP: HTTP relies on TCP as the underlying transport protocol to ensure reliable delivery.
- Client-initiated: only the client can initiate a request; the server cannot push data on its own under standard HTTP.

### 2.2 HTTP Methods

HTTP methods (also called HTTP verbs) define the type of action the client wants the server to perform.

| Method | Purpose | Idempotent | Typical Use Case |
|---|---|---|---|
| GET | Retrieve data from server | Yes | Loading a webpage, fetching a list of records |
| POST | Send data to server to create a resource | No | Submitting a form, creating a new user |
| PUT | Update/replace an existing resource completely | Yes | Updating a full user profile |
| PATCH | Partially update an existing resource | No | Updating only one field of a record |
| DELETE | Remove a resource | Yes | Deleting a record |

**Important Definition (Quotable):** Idempotent means that making the same request multiple times produces the same result as making it once, without additional side effects.

**Exam Tip:** GET and POST are the two most frequently tested methods for Semester 5 level. Be ready to explain the difference: GET appends data to the URL (visible, limited size, cacheable), while POST sends data in the request body (not visible in URL, larger size, not cached by default).

### 2.3 Request-Response Cycle

The request-response cycle describes the complete round trip of communication between client and server.

**Steps:**

1. **DNS Resolution:** Browser converts the domain name into an IP address using DNS (Domain Name System).
2. **TCP Connection:** A TCP connection is established between client and server (three-way handshake).
3. **HTTP Request Sent:** Client sends an HTTP request containing method, URL, headers, and optionally a body.
4. **Server Processing:** Server receives the request, processes it (may query a database, run business logic).
5. **HTTP Response Sent:** Server sends back a response containing a status code, headers, and body (HTML/JSON/file).
6. **Rendering:** Browser receives the response and renders it, or the client application uses the data.

**Structure of an HTTP Request:**
- Request Line (Method, URL, HTTP version)
- Headers (metadata such as Content-Type, User-Agent)
- Body (optional, used in POST/PUT/PATCH)

**Structure of an HTTP Response:**
- Status Line (HTTP version, status code, status message)
- Headers (metadata such as Content-Type, Content-Length)
- Body (the actual content: HTML, JSON, etc.)

### 2.4 HTTP Status Codes

Status codes are three-digit numbers returned by the server indicating the outcome of the request.

| Range | Category | Meaning | Common Examples |
|---|---|---|---|
| 1xx | Informational | Request received, processing continues | 100 Continue |
| 2xx | Success | Request successfully processed | 200 OK, 201 Created |
| 3xx | Redirection | Further action needed to complete request | 301 Moved Permanently, 302 Found |
| 4xx | Client Error | Request has an error caused by the client | 400 Bad Request, 404 Not Found |
| 5xx | Server Error | Server failed to process a valid request | 500 Internal Server Error |

**Important Codes to Memorize for Exam:**
- 200 OK — request succeeded
- 201 Created — resource successfully created (common after POST)
- 301 — permanent redirect
- 400 Bad Request — malformed request syntax
- 401 Unauthorized — authentication required
- 403 Forbidden — server understood but refuses to authorize
- 404 Not Found — resource does not exist
- 500 Internal Server Error — generic server-side failure

## 3. URL Structure

**Definition:** URL (Uniform Resource Locator) is the address used to identify and locate a resource on the Internet.

### 3.1 Components of a URL

Consider: `https://www.example.com:443/products/view?id=25#reviews`

| Component | Example | Description |
|---|---|---|
| Scheme/Protocol | https | Defines the protocol used (http, https, ftp) |
| Domain/Host | www.example.com | Identifies the server hosting the resource |
| Port | 443 | Network port (optional, default 80 for HTTP, 443 for HTTPS) |
| Path | /products/view | Specific resource location on the server |
| Query String | ?id=25 | Key-value pairs sending additional parameters |
| Fragment | #reviews | Points to a specific section within the resource (client-side only) |

**Important Note for Exam:** The fragment identifier (after #) is never sent to the server; it is used only by the browser to scroll to a section of the page.

## 4. Web Frameworks

**Definition:** A web framework is a software framework that provides a standard way to build and deploy web applications, offering pre-built components, libraries, and tools so developers do not need to write repetitive low-level code from scratch.

### 4.1 Why Use a Web Framework

- Faster development through reusable components.
- Built-in security features (protection against SQL injection, CSRF, XSS).
- Standardized project structure, easing collaboration and maintenance.
- Reduces boilerplate code for common tasks (routing, database access, form handling).

### 4.2 Why Django

- Django is a high-level Python web framework that follows the "batteries-included" philosophy, meaning it ships with most tools needed out of the box (ORM, admin panel, authentication, forms).
- Encourages rapid development and clean, pragmatic design.
- Follows the DRY (Don't Repeat Yourself) principle.

## 5. MVT Architecture

**Definition:** MVT (Model-View-Template) is the architectural pattern used by Django to separate application logic, data handling, and presentation.

### 5.1 Components of MVT

| Component | Responsibility |
|---|---|
| Model | Defines the data structure; interacts with the database (via ORM) |
| View | Contains the business logic; processes requests and decides what data to send |
| Template | Handles the presentation layer; defines how data is displayed (HTML with dynamic content) |

### 5.2 How MVT Differs from MVC

- In traditional MVC (Model-View-Controller), the Controller manages the flow between Model and View.
- In Django's MVT, Django itself acts as the controller internally; the developer writes Models, Views, and Templates, while Django's core framework handles URL routing and request dispatching.

**Important Definition (Quotable):** In MVT architecture, the View in Django is not the same as the View in traditional MVC; Django's View performs the role that a Controller performs in MVC, while the Template performs the role of the View in MVC.

### 5.3 MVT Flow (Step by Step)

1. User requests a URL through the browser.
2. Django's URL dispatcher matches the URL to a corresponding View.
3. View interacts with the Model if data is needed (queries the database).
4. View selects a Template and passes data (context) to it.
5. Template renders the final HTML using the context data.
6. Response is sent back to the browser.

## 6. Virtual Environment

**Definition:** A virtual environment is an isolated Python environment that allows a project to have its own dependencies, independent of other projects and the system-wide Python installation.

### 6.1 Why Virtual Environments Are Needed

- Prevents version conflicts between projects using different versions of the same package.
- Keeps the global Python installation clean.
- Makes projects portable and reproducible (dependencies can be listed and shared via requirements.txt).

### 6.2 Common Commands

| Task | Command |
|---|---|
| Create virtual environment | `python -m venv env` |
| Activate (Windows) | `env\Scripts\activate` |
| Activate (macOS/Linux) | `source env/bin/activate` |
| Deactivate | `deactivate` |

## 7. Django Installation

### 7.1 Steps

1. Ensure Python is installed (Django requires Python 3.x).
2. Create and activate a virtual environment.
3. Install Django using pip:
   ```
   pip install django
   ```
4. Verify installation:
   ```
   django-admin --version
   ```

### 7.2 Creating a Django Project

**Command:**
```
django-admin startproject projectname
```

**Effect:** This command generates a new Django project directory containing the initial project structure, including configuration files and a manage.py utility script.

**Important Note for Exam:** `django-admin` is the general Django command-line utility available system-wide (or within the environment) once Django is installed, whereas `manage.py` is a project-specific version of the same utility, automatically generated inside the project, that is aware of the project's settings.
