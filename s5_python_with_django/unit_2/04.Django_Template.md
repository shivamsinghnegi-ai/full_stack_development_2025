# Django Templates

---

## 1. Introduction to the Template Engine

**Definition:** The Django template engine is a component of Django that processes template files, which are typically HTML files embedded with special Django syntax, and combines them with dynamic data to produce the final HTML sent to the client.

### 1.1 Role in MVT Architecture

- The Template is the presentation layer of Django's MVT architecture.
- It separates the visual structure (HTML) from the business logic (Python code in views).
- Templates do not contain complex logic; they use a restricted template language designed to keep presentation and logic separate.

**Important Definition (Quotable):** The Django Template Language (DTL) is intentionally limited in what it allows, so that template designers cannot write arbitrary Python code inside a template; this enforces separation of concerns between logic and presentation.

### 1.2 Why Templates Are Needed

- Allow dynamic content (data from the database, computed values) to be inserted into static HTML structure.
- Enable reuse of common HTML structures (headers, footers, navigation) across multiple pages through template inheritance.
- Keep Python logic in views.py and presentation markup in .html files, improving maintainability.

## 2. Template Folder Setup

### 2.1 Two Approaches

Django templates can be organized in two ways:

**Approach 1: App-Level Templates Directory**
```
appname/
    templates/
        appname/
            home.html
```

**Approach 2: Project-Level Templates Directory**
```
projectname/
    templates/
        home.html
        about.html
```

### 2.2 Configuring the TEMPLATES Setting

In settings.py, the TEMPLATES list defines how Django locates and processes template files.

```python
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [BASE_DIR / 'templates'],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]
```

| Key | Purpose |
|---|---|
| BACKEND | Specifies which template engine to use (Django's own engine by default) |
| DIRS | List of additional directories to search for templates (project-level) |
| APP_DIRS | If True, Django also searches for a templates/ folder inside each installed app |
| OPTIONS | Additional settings, including context processors |

**Important Note for Exam:** When APP_DIRS is True, it is best practice to namespace app-level templates inside a subfolder matching the app name (e.g., appname/templates/appname/home.html), to avoid naming conflicts when multiple apps have templates with the same filename.

## 3. The render() Function

**Definition:** render() is a Django shortcut function that combines a given template with a context dictionary and returns an HttpResponse object containing the rendered HTML.

### 3.1 Syntax

```python
render(request, template_name, context=None, status=None)
```

| Argument | Purpose |
|---|---|
| request | The HttpRequest object (mandatory, required for context processors) |
| template_name | Path to the template file, relative to a templates directory |
| context | Optional dictionary of data to be passed into the template |
| status | Optional HTTP status code for the response (default 200) |

### 3.2 Example

```python
from django.shortcuts import render

def home(request):
    return render(request, 'home.html')
```

**Important Note for Exam:** render() is a shortcut that internally loads the template, renders it with the given context, and wraps the result in an HttpResponse — this means a view using render() does not need to manually import or use HttpResponse for that return statement.

### 3.3 render() vs Manual Approach

| Aspect | Using render() | Manual (loader + HttpResponse) |
|---|---|---|
| Lines of Code | Shorter, single call | Longer, multiple steps |
| Readability | Cleaner and preferred | More verbose |
| Usage in Practice | Standard, recommended approach | Rarely used directly |

## 4. Passing Context Data

**Definition:** Context refers to a Python dictionary passed from a view to a template, where each key becomes a variable accessible inside that template using Django template syntax.

### 4.1 Example

**View:**
```python
def profile(request):
    context = {
        'name': 'Rahul',
        'course': 'B.Tech CSE',
        'semester': 5,
    }
    return render(request, 'profile.html', context)
```

**Template (profile.html):**
```html
<h1>Name: {{ name }}</h1>
<p>Course: {{ course }}</p>
<p>Semester: {{ semester }}</p>
```

### 4.2 Variable Syntax

- Variables are accessed using double curly braces: `{{ variable_name }}`.
- Dictionary keys, object attributes, and list indices can all be accessed using dot notation inside templates: `{{ user.name }}`, `{{ items.0 }}`.

**Important Definition (Quotable):** Django template variable syntax uses dot notation uniformly for dictionary lookups, attribute lookups, and index lookups; the template engine automatically tries each type of lookup in sequence until one succeeds.

## 5. Template Tags

**Definition:** Template tags are special constructs enclosed in `{% %}` that provide logic-related functionality inside a template, such as loops, conditionals, and template inheritance.

### 5.1 Common Template Tags

| Tag | Purpose |
|---|---|
| `{% if %} ... {% endif %}` | Conditional rendering |
| `{% for %} ... {% endfor %}` | Looping over an iterable |
| `{% block %} ... {% endblock %}` | Defines a block for template inheritance |
| `{% extends %}` | Inherits structure from a parent template |
| `{% include %}` | Inserts another template into the current one |
| `{% url %}` | Generates a URL based on a named URL pattern |
| `{% csrf_token %}` | Inserts a CSRF protection token (used in forms) |

### 5.2 Example: Conditional and Loop

```html
{% if user.is_authenticated %}
    <p>Welcome, {{ user.username }}</p>
{% else %}
    <p>Please log in</p>
{% endif %}

<ul>
{% for item in item_list %}
    <li>{{ item }}</li>
{% endfor %}
</ul>
```

### 5.3 Template Inheritance

**Definition:** Template inheritance is a mechanism that allows a base (parent) template to define a common structure, which child templates can then extend and override specific sections of, using block tags.

**Base template (base.html):**
```html
<html>
<body>
    {% block content %}{% endblock %}
</body>
</html>
```

**Child template:**
```html
{% extends 'base.html' %}

{% block content %}
    <h1>This is the homepage</h1>
{% endblock %}
```

**Important Note for Exam:** {% extends %} must always be the first tag in a child template; if anything (even whitespace-sensitive tags) appears before it, Django will raise a template syntax error.

## 6. Template Filters

**Definition:** Template filters are used to modify the display of a variable inside a template; they are applied using the pipe symbol and can transform, format, or manipulate the variable's value before rendering.

### 6.1 Syntax

```html
{{ variable|filter_name }}
{{ variable|filter_name:argument }}
```

### 6.2 Commonly Used Filters

| Filter | Purpose | Example |
|---|---|---|
| `upper` | Converts text to uppercase | `{{ name|upper }}` |
| `lower` | Converts text to lowercase | `{{ name|lower }}` |
| `length` | Returns the length of a string/list | `{{ item_list|length }}` |
| `default` | Provides a fallback value if the variable is empty/false | `{{ value|default:"N/A" }}` |
| `date` | Formats a date value | `{{ created_at|date:"d-m-Y" }}` |
| `truncatewords` | Truncates text to a given number of words | `{{ content|truncatewords:20 }}` |

**Important Note for Exam:** Multiple filters can be chained together, applied left to right, for example: `{{ name|lower|title }}`.

## 7. Connecting Views to Templates (End-to-End Flow)

This ties together everything learned so far in Chapter 2.

### 7.1 Complete Flow

1. Browser sends a request to a URL.
2. Django's URL dispatcher matches the URL to a view function.
3. The view function executes its logic and prepares a context dictionary.
4. The view calls render(request, template_name, context).
5. The template engine locates the template file, processes template tags and filters, and substitutes variables from the context.
6. The final rendered HTML is wrapped in an HttpResponse and sent back to the browser.

### 7.2 Complete Example

**views.py:**
```python
from django.shortcuts import render

def student_list(request):
    students = ['Amit', 'Priya', 'Rahul']
    context = {'students': students, 'total': len(students)}
    return render(request, 'students/student_list.html', context)
```

**urls.py:**
```python
from django.urls import path
from . import views

urlpatterns = [
    path('students/', views.student_list, name='student_list'),
]
```

**student_list.html:**
```html
<h2>Total Students: {{ total }}</h2>
<ul>
{% for student in students %}
    <li>{{ student }}</li>
{% endfor %}
</ul>
```
