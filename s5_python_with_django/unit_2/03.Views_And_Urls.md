# Views and URLs

---

## 1. Introduction to Views

**Definition:** A view, in Django, is a Python function or class that receives a web request and returns a web response. It contains the business logic that determines what content is sent back to the user.

### 1.1 Role of a View in MVT

- A view acts as the bridge between the Model (data) and the Template (presentation).
- It receives an `HttpRequest` object as input.
- It must return an `HttpResponse` object (or a subclass of it) as output.
- This request-in, response-out contract is mandatory; a view that does not return an HttpResponse-compatible object will raise an error.

**Important Definition (Quotable):** In Django, every view function must take at least one parameter, conventionally named request, which is an HttpRequest object, and must return an HttpResponse object.

---

## 2. Function-Based Views (FBV)

**Definition:** A function-based view is a Python function that takes an HttpRequest object as its first argument and returns an HttpResponse object; it is the simplest and most direct way to define view logic in Django.

### 2.1 Basic Syntax

```python
from django.http import HttpResponse

def home(request):
    return HttpResponse("Welcome to the homepage")
```

### 2.2 Characteristics of Function-Based Views

- Straightforward and easy to read for simple logic.
- Each HTTP method (GET, POST, etc.) must be handled explicitly using conditional statements inside the same function.
- Preferred for views with simple, one-off logic that does not need to be reused or extended.

### 2.3 Handling Different HTTP Methods in an FBV

```python
def contact(request):
    if request.method == 'POST':
        # process submitted form data
        return HttpResponse("Form submitted")
    else:
        return HttpResponse("Contact page")
```

**Important Note for Exam:** `request.method` is a string attribute (such as 'GET' or 'POST') that tells the view which HTTP method was used to make the request; this is the standard way to differentiate behavior for the same URL.

---

## 3. HttpResponse

**Definition:** HttpResponse is a Django class, part of `django.http`, used to construct and return an HTTP response with a given content body, status code, and content type.

### 3.1 Common Usage

```python
from django.http import HttpResponse

def greet(request):
    return HttpResponse("Hello, World!")
```

### 3.2 Passing Additional Parameters

```python
return HttpResponse("Not allowed", status=403)
return HttpResponse("<h1>Data</h1>", content_type="text/html")
```

| Parameter | Purpose |
|---|---|
| content | The response body (string or bytes) |
| status | HTTP status code to return (default is 200) |
| content_type | MIME type of the response (default is 'text/html') |

### 3.3 Related Response Classes

| Class | Purpose |
|---|---|
| `HttpResponse` | Basic response with arbitrary content |
| `HttpResponseRedirect` | Redirects the client to a different URL |
| `HttpResponseNotFound` | Returns a response with 404 status code |
| `JsonResponse` | Returns a properly formatted JSON response |

**Exam Tip:** JsonResponse is especially relevant later when building APIs (Chapter 5), but it is useful to introduce here as it is technically a subclass of HttpResponse.

---

## 4. Basic View Logic

View logic refers to the processing steps a view performs before returning a response. At this stage (before models are introduced in Chapter 3), view logic is limited to request handling and simple computation.

### 4.1 Typical Elements of View Logic

1. **Reading request data:** accessing data sent by the client, such as `request.GET` (query parameters) or `request.POST` (form data).
2. **Conditional processing:** using `if` statements to branch behavior based on method type or input values.
3. **Returning a response:** constructing and returning the appropriate HttpResponse.

### 4.2 Example: Using Query Parameters

```python
def search(request):
    query = request.GET.get('q', '')
    return HttpResponse(f"You searched for: {query}")
```

**Important Note for Exam:** `request.GET` and `request.POST` are QueryDict objects (dictionary-like), and using `.get('key', default)` is the safe way to access values without raising a KeyError if the key is missing.

## 5. URL Routing

**Definition:** URL routing is the mechanism by which Django matches an incoming request's URL path to a specific view function, based on a set of defined patterns called urlpatterns.

### 5.1 urlpatterns

**Definition:** urlpatterns is a list defined in a urls.py file, consisting of `path()` (or `re_path()`) entries, that together form the URLconf for that file.

**Basic Example:**
```python
from django.urls import path
from . import views

urlpatterns = [
    path('', views.home, name='home'),
    path('about/', views.about, name='about'),
    path('contact/', views.contact, name='contact'),
]
```

### 5.2 Structure of path()

```python
path(route, view, kwargs=None, name=None)
```

| Argument | Purpose |
|---|---|
| route | A string representing the URL pattern to match |
| view | The view function (or class) to call when the pattern matches |
| kwargs | Optional dictionary of extra keyword arguments passed to the view |
| name | A unique name for the URL, used for reverse URL lookups |

**Important Definition (Quotable):** The name argument in path() allows a URL to be referenced by a fixed identifier rather than a hardcoded path, which means the actual URL string can change without breaking references to it elsewhere in the code (in templates, using the `{% url %}` tag, or in views, using `reverse()`).

---

## 6. Path Converters

**Definition:** Path converters are type specifiers used inside a URL pattern to capture a portion of the URL as a variable and convert it to a specific Python data type before passing it to the view.

### 6.1 Syntax

```python
path('article/<int:id>/', views.article_detail, name='article_detail')
```

Here, `<int:id>` captures a segment of the URL, converts it to an integer, and passes it to the view as a keyword argument named `id`.

### 6.2 Built-in Path Converters

| Converter | Matches | Example |
|---|---|---|
| `str` | Any non-empty string, excluding the slash character (default if unspecified) | `<str:username>` |
| `int` | Any positive integer, including zero | `<int:id>` |
| `slug` | Slug strings (letters, numbers, hyphens, underscores) | `<slug:post_slug>` |
| `uuid` | A formatted UUID string | `<uuid:id>` |
| `path` | Any non-empty string, including the slash character | `<path:filepath>` |

### 6.3 Example View Using a Path Converter

```python
def article_detail(request, id):
    return HttpResponse(f"Displaying article number {id}")
```

**Important Note for Exam:** The name used inside the converter (such as `id` in `<int:id>`) must exactly match the parameter name in the corresponding view function, since Django passes it as a keyword argument.

## 7. App-Level vs Project-Level URLs

As a Django project grows to include multiple apps, it is standard practice to keep each app's URL patterns separate and then include them into the main project-level urls.py.

### 7.1 Project-Level urls.py

- Located inside the project package (the inner folder created during `startproject`).
- Referenced by `ROOT_URLCONF` in settings.py.
- Acts as the entry point for all URL routing in the project.

**Example:**
```python
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('blog/', include('blog.urls')),
]
```

### 7.2 App-Level urls.py

- Created manually inside each app folder (Django does not generate this automatically with startapp).
- Contains urlpatterns specific to that app only.

**Example (inside blog/urls.py):**
```python
from django.urls import path
from . import views

urlpatterns = [
    path('', views.blog_home, name='blog_home'),
    path('<int:post_id>/', views.blog_detail, name='blog_detail'),
]
```

### 7.3 The include() Function

**Definition:** include() is a Django function used in a project-level urls.py to reference another URLconf module (typically from an app), allowing that app's URL patterns to be plugged into the project under a specific path prefix.

**Important Note for Exam:** Using include() keeps each app self-contained and modular; it means an app's URLs can be reused in other projects simply by including its urls.py, without modifying the app's internal code.

### 7.4 Comparison Table

| Aspect | Project-Level urls.py | App-Level urls.py |
|---|---|---|
| Location | Inside the project package | Inside each individual app |
| Created By | Automatically, during startproject | Manually, by the developer |
| Scope | Root URLconf for the entire project | URLs specific to one app |
| Referenced In | ROOT_URLCONF in settings.py | Included into project-level urls.py via include() |
| Purpose | Delegates requests to the correct app | Maps specific paths to that app's views |
