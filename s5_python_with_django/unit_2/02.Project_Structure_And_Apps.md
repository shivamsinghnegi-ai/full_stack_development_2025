# Django Project Structure and Apps

---

## 1. Django Project Structure

When a Django project is created using `django-admin startproject projectname`, Django auto-generates a specific directory structure. Understanding this structure is essential before writing any application logic.

**Typical structure:**
```
projectname/
    manage.py
    projectname/
        __init__.py
        settings.py
        urls.py
        wsgi.py
        asgi.py
```

**The Django project structure refers to the standardized arrangement of files and folders that Django generates automatically, providing the configuration backbone for the entire web application.**

**Important Note for Exam:** Notice that the project name appears twice — once as the outer folder (the container for the whole project, including manage.py) and once as the inner folder (the actual Python package holding configuration files). This inner folder is often called the "project package."

---

## 2. manage.py

**manage.py is a command-line utility, automatically generated in every Django project, that lets developers interact with the project for administrative tasks such as running the server, creating apps, and applying migrations.**

### 2.1 Purpose

- Acts as a project-specific version of `django-admin`.
- Automatically configures the `DJANGO_SETTINGS_MODULE` environment variable so Django knows which settings file to use.
- Serves as the single entry point for nearly all command-line operations on the project.

### 2.2 Commonly Used manage.py Commands

| Command | Purpose |
|---|---|
| `python manage.py runserver` | Starts the development server |
| `python manage.py startapp appname` | Creates a new app inside the project |
| `python manage.py makemigrations` | Generates migration files based on model changes |
| `python manage.py migrate` | Applies migrations to the database |
| `python manage.py createsuperuser` | Creates an admin user for the Django admin panel |
| `python manage.py shell` | Opens an interactive Python shell with Django context loaded |

**Important:** manage.py is functionally identical to django-admin, except that it also sets the DJANGO_SETTINGS_MODULE environment variable to point to the current project's settings, and it puts the project's package on sys.path.

## 3. settings.py

**settings.py is the central configuration file of a Django project that defines all the settings required for the project to run, including database configuration, installed applications, middleware, templates, and security options.**

### 3.1 Key Configuration Variables

| Variable | Purpose |
|---|---|
| `DEBUG` | Boolean; if True, shows detailed error pages (used only in development) |
| `ALLOWED_HOSTS` | List of host/domain names the Django site can serve |
| `INSTALLED_APPS` | List of all applications enabled in this Django project |
| `MIDDLEWARE` | List of middleware components processed on every request/response |
| `ROOT_URLCONF` | Points to the Python module defining the root URL configuration |
| `TEMPLATES` | Configuration for the template engine, including template directories |
| `DATABASES` | Configuration dictionary defining database engine and connection details |
| `STATIC_URL` | URL prefix for serving static files (CSS, JS, images) |
| `SECRET_KEY` | A unique, secret value used for cryptographic signing; must be kept confidential |

**Important Note for Exam:** DEBUG must always be set to False in a production environment. Leaving DEBUG as True in production is a major security vulnerability, since it exposes sensitive configuration details in error pages.

## 4. urls.py

**urls.py is the file responsible for URL routing in Django; it maps incoming URL patterns to their corresponding views, forming what is called the URLconf (URL configuration).**

### 4.1 Structure

- The project-level urls.py (inside the project package) acts as the root URLconf, referenced by `ROOT_URLCONF` in settings.py.
- Each app can have its own urls.py, which is then included into the project-level urls.py using the `include()` function.

**Basic example structure:**
```
from django.contrib import admin
from django.urls import path

urlpatterns = [
    path('admin/', admin.site.urls),
]
```

**Important:** URLconf is the mapping between URL path patterns and the Python functions or classes (views) that should be called to handle each pattern.

## 5. wsgi.py and asgi.py

**WSGI:** WSGI (Web Server Gateway Interface) is a specification that defines how a web server communicates with a Python web application; wsgi.py exposes the WSGI-compatible application object used for deploying Django on traditional synchronous web servers.

**ASGI:** ASGI (Asynchronous Server Gateway Interface) is the successor to WSGI that supports asynchronous communication, allowing Django to handle asynchronous protocols such as WebSockets in addition to standard HTTP.

### 5.1 Comparison

| Aspect | wsgi.py | asgi.py |
|---|---|---|
| Full Form | Web Server Gateway Interface | Asynchronous Server Gateway Interface |
| Communication Type | Synchronous only | Supports both synchronous and asynchronous |
| Use Case | Traditional HTTP request-response deployment | Real-time features (WebSockets, long-lived connections) |
| Default Usage | Used by default for standard deployment | Used when async capabilities are required |

**Important Note for Exam:** Both files exist by default in every Django project since Django introduced async support, but for a typical Semester 5 web application (CRUD-based, no real-time features), wsgi.py is the one actually used for deployment.

## 6. Creating a Django App

**A Django app is a self-contained module that implements a specific piece of functionality within a Django project, such as a blog, authentication system, or shopping cart, and can be reused across multiple projects.**

### 6.1 Command to Create an App

```
python manage.py startapp appname
```

### 6.2 Structure Generated

```
appname/
    __init__.py
    admin.py
    apps.py
    migrations/
        __init__.py
    models.py
    tests.py
    views.py
```

| File | Purpose |
|---|---|
| `admin.py` | Registers models to appear in the Django admin interface |
| `apps.py` | Contains the app's configuration class |
| `migrations/` | Stores database migration files tracking model changes |
| `models.py` | Defines the data models (database schema) for this app |
| `tests.py` | Contains test cases for the app |
| `views.py` | Contains view functions/classes handling request logic for this app |

## 7. Project vs App

This distinction is one of the most frequently asked conceptual questions in Django exams.

| Aspect | Project | App |
|---|---|---|
| Definition | The entire web application, including configuration and settings | A modular component performing one specific function |
| Scope | Contains one or more apps | Exists within a project, focused on a single feature |
| Example | An e-commerce website | Apps within it: "products", "cart", "accounts" |
| Configuration | Has settings.py, urls.py at root level | Has its own models.py, views.py, admin.py |
| Reusability | Not reusable across other projects | Can be reused across multiple projects |

**Important Definition (Quotable):** A Django project is a collection of configuration and apps for a particular website, while a Django app is a self-contained module that can exist by itself and be plugged into different projects.

**Exam Tip:** A common trick question asks "can a project contain multiple apps?" — Yes. It can also ask "can an app exist without a project?" — Technically an app's code can exist independently, but it needs to be registered inside a project's settings to actually run.

## 8. INSTALLED_APPS

**INSTALLED_APPS is a list defined in settings.py that specifies all Django applications that are activated for the current project, including both built-in Django apps and custom apps created by the developer.**

### 8.1 Default Built-in Apps

```
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
]
```

| Built-in App | Purpose |
|---|---|
| `django.contrib.admin` | Provides the admin interface |
| `django.contrib.auth` | Provides authentication and permission system |
| `django.contrib.contenttypes` | Framework for tracking content types installed in the project |
| `django.contrib.sessions` | Manages user session data across requests |
| `django.contrib.messages` | Provides temporary/flash messaging framework |
| `django.contrib.staticfiles` | Manages static files (CSS, JS, images) |

### 8.2 Registering a Custom App

Once an app is created using `startapp`, it must be manually added to INSTALLED_APPS, otherwise Django will not recognize its models, templates, or admin registrations.

```
INSTALLED_APPS = [
    ...
    'appname',
]
```

**Important Note for Exam:** Simply running `startapp` does not activate the app. A very commonly tested/practical point is that the app must be explicitly added to INSTALLED_APPS before its models can be migrated or its templates recognized.

### Medium Answer (5 Marks)
1. Explain the complete folder structure generated when a Django project is created.
2. Explain any five important configuration variables found in settings.py.
3. Explain the role of urls.py at both the project level and app level.
4. Differentiate between wsgi.py and asgi.py with respect to their use cases.
5. Explain the difference between a Django project and a Django app with examples.
6. Explain the structure generated when a new app is created using startapp, describing the purpose of each file.

1. A student creates a new Django project named "college_portal" and wants to add two features: student records and library management. Explain, with proper justification, how the student should organize this using Django's project and app structure, and describe the steps involved from project creation to app registration.
2. Explain in detail the purpose and content of settings.py, covering at least six configuration variables, and explain the security implications of misconfiguring DEBUG and ALLOWED_HOSTS.
3. Describe the complete lifecycle of a Django project from creation to running the development server: include the commands used, files generated, and the purpose of manage.py at each step.
4. Compare and contrast WSGI and ASGI in detail, explaining why Django introduced ASGI support and in what real-world scenarios ASGI would be preferred over WSGI.
5. Explain the concept of reusability in Django apps. Using an example of an e-commerce website, describe how the project would be broken down into multiple apps and how INSTALLED_APPS would be configured.
