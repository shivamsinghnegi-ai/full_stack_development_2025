# **28: VM Lifecycle — Provisioning, Management & Migration**

*(How virtual machines are created, operated, and moved in real systems)*

---

## 1. Why VM Lifecycle Matters

So far, we’ve learned:

* virtualization creates VMs
* hypervisors manage them
* machine images define how they start

Now we connect everything into one continuous flow:

> **How does a virtual machine live from creation to termination?**

This is called the **VM lifecycle**.

Understanding this is critical because:

* cloud systems constantly create and destroy VMs
* scaling depends on this lifecycle
* failures are handled through lifecycle transitions

---

## 2. What Is VM Lifecycle?

The **VM lifecycle** describes all stages a virtual machine goes through:

1. Provisioning (creation)
2. Running (execution)
3. Management (monitoring and control)
4. Migration (movement across hosts)
5. Termination (deletion)

Important idea:

> VMs are **not permanent machines** — they are **temporary resources**.

---

## 3. Stage 1: Provisioning (Creation of VM)

Provisioning is the process of:

* creating a new VM from a machine image
* allocating CPU, memory, storage
* attaching network configuration

### Steps in Provisioning

1. Select machine image
2. Allocate resources
3. Configure network
4. Boot the VM
5. Start application

This process is:

* automated in cloud
* fast (seconds to minutes)
* repeatable

---

### Key Insight

> Provisioning is **not just creation**, it is **standardized system setup**.

---

## 4. Stage 2: Running State

Once provisioned, the VM enters the **running state**.

In this phase:

* applications execute
* users interact with services
* resources are consumed

Responsibilities during this phase:

* handle requests
* maintain performance
* ensure availability

---

## 5. Stage 3: Management (Monitoring & Control)

Running VMs need continuous management.

This includes:

### Monitoring

* CPU usage
* memory usage
* disk activity
* network traffic

---

### Health Checks

* is the VM responding?
* is the application healthy?

---

### Control Actions

* restart
* stop
* resize
* replace

---

### Key Idea

> Management ensures the VM continues to behave correctly under load and failure.

---

## 6. Stage 4: Migration (Moving VMs)

Migration is one of the most powerful features of virtualization.

It allows:

> **Moving a VM from one physical machine to another**

without affecting users (in many cases).

---

### Why Migration Is Needed

* hardware maintenance
* load balancing across hosts
* failure prevention
* resource optimization

---

### Types of Migration

#### Cold Migration

* VM is stopped
* moved to another host
* restarted

Pros:

* simple

Cons:

* downtime

---

#### Live Migration

* VM continues running
* memory state is transferred
* execution shifts to new host

Pros:

* no downtime

Cons:

* more complex

---

### Key Insight

> Migration allows cloud systems to **maintain availability even during infrastructure changes**.

---

## 7. Stage 5: Termination (Deletion)

When a VM is no longer needed:

* it is stopped
* resources are released
* billing stops

In cloud systems:

* termination happens frequently
* unused resources are removed automatically

---

### Important Concept

> Idle VMs cost money — termination is part of optimization.

---

## 8. VM Lifecycle in Real Cloud Systems

Let’s connect all stages:

1. Traffic increases → new VMs provisioned
2. Load distributed → VMs run applications
3. Monitoring checks performance
4. Some VMs migrated for optimization
5. Traffic drops → extra VMs terminated

This continuous cycle creates:

> **Dynamic, self-adjusting infrastructure**

---

## 9. Why VM Lifecycle Enables Cloud Scalability

The VM lifecycle enables:

### Elastic Scaling

* create and remove VMs based on demand

---

### Fault Recovery

* failed VMs replaced automatically

---

### Resource Optimization

* workloads moved to efficient hosts

---

### Cost Efficiency

* unused VMs removed

---

## 10. Common Mistakes in Understanding VM Lifecycle

### Mistake 1: Treating VMs as permanent

→ leads to poor design

---

### Mistake 2: Ignoring termination

→ increases cost

---

### Mistake 3: Not handling restarts

→ application failures

---

### Mistake 4: Storing critical data inside VM

→ data loss during lifecycle events

---
