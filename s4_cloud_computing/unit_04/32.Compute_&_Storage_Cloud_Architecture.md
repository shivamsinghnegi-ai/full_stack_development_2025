# **32: Compute & Storage Cloud Architecture**

*(Foundational design of cloud-based systems)*

---

## 1. Introduction

Cloud computing systems are designed to process large volumes of data and serve millions of users efficiently. At the core of these systems lie two fundamental components:

* **Compute**, which performs processing and executes application logic
* **Storage**, which is responsible for retaining, organizing, and managing data

The architectural design of cloud environments focuses on how these components are structured and interact to achieve:

* scalability
* reliability
* performance
* cost efficiency

A clear understanding of compute and storage architecture is essential for designing robust and scalable cloud-based applications.

---

## 2. Compute in Cloud Architecture

**Compute** refers to the resources responsible for executing application logic and processing user requests.

Common forms of compute include:

* virtual machines
* container-based environments
* serverless execution units

### Characteristics of Compute

* **Dynamic**: resources can be created or removed based on demand
* **Scalable**: supports horizontal scaling by adding multiple instances
* **Replaceable**: instances can be restarted or replaced without affecting the system design

A key architectural principle is:

> Compute resources are treated as temporary and disposable.

---

## 3. Storage in Cloud Architecture

**Storage** refers to the systems responsible for persisting data over time.

Types of storage used in cloud systems include:

* block storage
* object storage
* database systems

### Characteristics of Storage

* **Persistent**: data must remain available even after failures
* **Reliable**: ensures durability through replication and backups
* **Consistent**: maintains correctness of stored data

Unlike compute, storage is designed to be:

> Stable, durable, and long-lived.

---

## 4. Separation of Compute and Storage

A fundamental principle in modern cloud architecture is:

> **Separation of compute and storage**

### Purpose of Separation

* enables independent scaling of compute and storage
* improves system flexibility
* reduces failure impact
* simplifies system design

### Without Separation

* applications become tightly coupled with data
* scaling becomes complex
* failures can lead to data loss

### With Separation

* compute instances can scale independently
* storage remains stable and consistent
* system becomes modular and resilient

---

## 5. Types of Storage in Cloud Systems

Different storage mechanisms serve different purposes:

---

### 5.1 Block Storage

* provides low-level storage for virtual machines
* behaves like a physical disk
* used for operating systems and databases

---

### 5.2 Object Storage

* designed for unstructured data such as images, videos, and backups
* highly scalable
* accessed through APIs

---

### 5.3 Database Storage

* structured storage for applications
* supports querying and indexing
* ensures data consistency

---

### Key Observation

Selecting an appropriate storage type is critical for:

* performance
* cost efficiency
* scalability

---

## 6. Interaction Between Compute and Storage

In a typical cloud system, the interaction follows a structured flow:

1. A request is received by the system
2. Compute resources process the request
3. Storage systems are accessed to retrieve or update data
4. The processed response is returned

### Important Design Principle

> Compute should not maintain permanent data internally.

This ensures:

* scalability
* fault tolerance
* consistency

---

## 7. Stateless and Stateful Design

Cloud architecture distinguishes between two types of system behavior:

---

### Stateless Compute

* does not store data locally
* relies on external storage systems
* easily scalable
* fault tolerant

---

### Stateful Systems

* store data within the system
* difficult to scale
* harder to recover after failure

---

### Preferred Approach

Modern cloud systems follow:

> **Stateless compute with external stateful storage**

---

## 8. Scaling Considerations

### Compute Scaling

* fast and flexible
* achieved by adding or removing instances
* supports horizontal scaling

---

### Storage Scaling

* more complex
* requires planning and consistency management
* must ensure data durability and correctness

---

### Key Insight

> Compute scales rapidly, whereas storage requires controlled and structured scaling.

---

## 9. Fault Tolerance in Cloud Architecture

Cloud systems are designed with the assumption that failures will occur.

---

### Compute Failures

* handled by restarting or replacing instances
* minimal impact if stateless design is followed

---

### Storage Failures

* handled through replication and backups
* requires strong consistency mechanisms

---

### Architectural Principle

> Compute is disposable, while storage must be durable.

---

## 10. Common Design Issues

* coupling compute and storage tightly
* storing critical data within compute instances
* selecting inappropriate storage types
* ignoring differences in scaling behavior

These issues lead to:

* reduced scalability
* increased failure impact
* inefficient resource usage

---

## 11. Conceptual Summary

Cloud architecture is fundamentally based on:

* separation of responsibilities
* modular system design
* independent scaling of components

The distinction between compute and storage forms the basis of:

* modern distributed systems
* scalable cloud applications
* reliable infrastructure design

---

## Mental Model to Remember

> **Compute executes operations.
> Storage preserves data.
> Effective cloud architecture separates and coordinates both.**

---

## Quick Recap

* Cloud systems are built on compute and storage components
* Compute is dynamic, scalable, and replaceable
* Storage is persistent, reliable, and consistent
* Separation of compute and storage enables scalability
* Stateless compute is essential for modern architectures
