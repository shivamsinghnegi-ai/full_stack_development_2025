# 2: Roots of Cloud Computing

*(How Today’s Cloud Was Born)*

---

## 1. Cloud Computing Did Not Appear Overnight

Cloud computing looks modern, but it is actually the result of **decades of evolution** in how computers were used and shared.

To understand cloud properly, we must trace **four foundational ideas**:

* Centralized computing
* Distributed systems
* Grid computing
* Virtualization

Each solved a real-world problem—and cloud is the **combined outcome** of all four.

---

## 2. Centralized Computing: One Powerful System, Many Users

In the early days, computers were:

* Extremely expensive
* Very limited in number
* Shared by many users

A **single powerful machine** performed all processing.

### Education Example

Universities had **one central computer lab system**.
Students booked time slots to run programs.

### Business Analogy

Think of one large **college server** where:

* Student records
* Fees
* Exam results
  all lived on one machine.

If that system failed → **everything stopped**.

This model created **dependency on a single resource**, which later systems tried to fix.

---

## 3. Distributed Systems: Many Computers Working Together

Instead of relying on one powerful machine, systems evolved to:

* Use **multiple computers**
* Distribute work across them
* Continue functioning even if one fails

![Image](https://media.geeksforgeeks.org/wp-content/uploads/20230331120006/Distributed-System.png?utm_source=codinggita.com)

![Image](https://upload.wikimedia.org/wikipedia/commons/thumb/c/c9/Client-server-model.svg/1200px-Client-server-model.svg.png?utm_source=codinggita.com)

### Real-World Analogy (Cricket)

Imagine a cricket team:

* One player getting out doesn’t end the match
* Others continue contributing

A distributed system works the same way.

### OTT Example

When you watch a match on **JioHotstar**:

* Video does not come from one computer
* Multiple servers handle streaming, ads, analytics, and recommendations

If one server fails, **you don’t notice**.

---

## 4. Why Distributed Systems Were Still Not Enough

Distributed systems helped with:

* Reliability
* Performance
* Fault tolerance

But they still had problems:

* Machines were often underused
* Resource sharing was inefficient
* Scaling required manual coordination

For example:

* Admission portal traffic spikes only for 2 months
* Servers sit idle for the remaining 10 months

This led to the next idea.

---

## 5. Grid Computing: Sharing Resources Across Organizations

Grid computing aimed to **pool unused resources**.

Multiple organizations agreed to:

* Share computing power
* Share storage
* Share processing time

![Image](https://www.ionos.co.uk/digitalguide/fileadmin/DigitalGuide/Schaubilder/grid_vs_cloud_en.png?utm_source=codinggita.com)

![Image](https://zd-brightspot.s3.us-east-1.amazonaws.com/wp-content/uploads/2022/02/01132102/39.png?utm_source=codinggita.com)

### Education Example

Imagine 5 colleges:

* Each has servers idle at night
* All connect them into a shared “grid”
* Heavy computation tasks run using combined power

### Scientific Use Case

Weather forecasting, research simulations, exam result processing.

### Limitation

* Very complex to manage
* Not real-time friendly
* Not suitable for dynamic web apps like Swiggy or OTT platforms

Grid computing showed that **resource pooling works**, but it lacked flexibility.

---

## 6. Virtualization: The Game Changer

Virtualization introduced a **massive shift**.

Instead of:

* One OS → one machine

We got:

* Multiple virtual machines → one physical machine

![Image](https://eu-images.contentstack.com/v3/assets/blt8eb3cdfc1fce5194/bltef4f4e5acd5c9af3/662120d02b1c9d903c2d134b/Hypervisor101-Type2-470.jpg?utm_source=codinggita.com)

![Image](https://media2.dev.to/dynamic/image/width%3D800%2Cheight%3D%2Cfit%3Dscale-down%2Cgravity%3Dauto%2Cformat%3Dauto/https%3A%2F%2Fdev-to-uploads.s3.amazonaws.com%2Fuploads%2Farticles%2Fuqmfx0wvlvzcykijdcx0.png?utm_source=codinggita.com)

### Simple Explanation

A powerful server can be split into:

* 10 virtual servers
* Each behaving like a real computer
* Each isolated from the others

### Education Example

One physical server running:

* Student portal
* Learning management system
* Attendance system
  all independently.

### Business Example

**Swiggy** does not use:

* One server for orders
* One server for payments
* One server for tracking

Instead:

* Hundreds of virtual machines
* Created and destroyed dynamically

---

## 7. Virtualization + Automation = Cloud Foundation

Virtualization alone was not enough.

Cloud was born when virtualization was combined with:

* Automation
* Self-service provisioning
* Metered billing
* Centralized management

Now users could:

* Create servers in minutes
* Scale automatically
* Pay only for usage

This is exactly how modern cloud platforms work.

---

## 8. Real-World Mapping: From Roots to Cloud

Let’s connect everything with familiar platforms:

### OTT Platforms

**Netflix**

* Distributed systems → global streaming
* Virtualization → thousands of isolated services
* Cloud → instant scaling during new releases

### Cricket Streaming (IPL Finals)

* Millions join at once
* Servers scale instantly
* No manual intervention

### Fintech Example

**Groww**

* Market opens → traffic spikes
* Cloud scales backend systems
* After hours → resources scale down

### Education Platforms

* Exam days → heavy load
* Normal days → low usage
* Cloud prevents over-investment in hardware

---

## 9. Why Understanding These Roots Matters

If you skip these foundations, cloud feels like “magic”.

But when you understand the roots:

* You design better systems
* You avoid overengineering
* You understand cost vs performance trade-offs
* You grow from **developer to system thinker**

Senior engineers don’t just *use* cloud—they understand **why it behaves the way it does**.

---

## Quick Recap

* Centralized systems → single point of failure
* Distributed systems → reliability and scale
* Grid computing → shared resource idea
* Virtualization → efficient hardware usage
* Automation + virtualization → cloud computing
