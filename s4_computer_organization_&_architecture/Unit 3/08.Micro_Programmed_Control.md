# Micro-Programmed Control
---

## 1. What is Micro-Programmed Control?

### Definition

In micro-programmed control, the **control unit is implemented using a special memory** (called **control memory** or **control store**) that stores a set of microinstructions. Each machine-level instruction is broken down into a **sequence of microinstructions** called a **microprogram**.

Instead of using fixed hardware logic to generate control signals, the control unit **reads microinstructions from control memory** and uses them to generate the required control signals.

> **Introduced by:** Maurice Wilkes in **1951**

---

### Why Micro-Programmed Control? (Need / Motivation)

In a **hardwired control unit**, control signals are generated by fixed combinational logic. This works well for simple instruction sets, but:

- Adding new instructions requires **hardware redesign**
- Complex instruction sets make the logic **very complicated**
- **Difficult to debug or modify**

Micro-programmed control solves this by storing control logic in **memory** instead of hardware:
- Adding new instructions = just **add new microprograms** to control memory
- Easy to **modify and debug**
- Supports **complex instruction sets (CISC)**

---

### Core Idea — Simple Analogy

> Think of a music box:
> - **Hardwired** = Fixed metal pins on a rotating drum — cannot be changed
> - **Micro-programmed** = A music player reading from a replaceable cassette tape (control memory)
>
> Micro-programmed control is flexible because you can change the "tape" (reprogramme the control memory).

---

## 2. Key Terminology

| Term | Definition |
|------|-----------|
| **Microinstruction** | A single step stored in control memory — contains control signals for one clock cycle |
| **Microprogram** | A complete sequence of microinstructions that implements one machine instruction |
| **Control Memory** | Special ROM/RAM inside the control unit that stores all microprograms |
| **Control Address Register (CAR)** | Holds the address of the **current microinstruction** being executed (like a PC for control memory) |
| **Control Data Register (CDR)** | Holds the microinstruction **fetched from control memory** — ready to be decoded |
| **Subroutine Register (SBR)** | Saves the **return address** when a microprogram calls a subroutine |
| **Mapping Logic** | Translates the IR opcode into the **starting address** of the corresponding microprogram |
| **Opcode** | The part of a machine instruction that identifies the operation (e.g., ADD, SUB, LOAD) |

### How Each Block Works

**IR (Instruction Register):**
Holds the current machine instruction fetched from memory. Its opcode is sent to the mapping logic.

**Mapping Logic:**
Converts the opcode from IR into the **starting address** of the microprogram that implements that instruction. This is essentially a lookup table.

**CAR (Control Address Register):**
Works exactly like the Program Counter (PC) but for control memory. Always points to the address of the **next microinstruction** to be executed.

**Control Memory:**
A ROM (or sometimes writable RAM) that stores all microprograms. Each location holds one microinstruction.

**CDR (Control Data Register):**
Holds the microinstruction that was just **fetched from control memory**. It is then split into:
- Operations Field → generates control signals
- Sequencing Field → tells CAR where to go next

**SBR (Subroutine Register):**
When a microprogram needs to call a common subroutine (e.g., fetch cycle is a subroutine used by all instructions), SBR saves the return address so execution can resume after the subroutine ends.

## 4. Address Sequencing

Address sequencing determines **which microinstruction executes next** — i.e., what value goes into CAR after the current microinstruction completes.

There are **4 methods** of address sequencing:

---

### Method 1: Incrementing CAR (Sequential Execution)

The simplest method — next microinstruction is at **CAR + 1**.

```
CAR = 100  →  execute microinstruction at address 100
CAR = 101  →  execute microinstruction at address 101
CAR = 102  →  execute microinstruction at address 102
...
```

Used when microinstructions need to execute **one after another** in order.

---

### Method 2: Unconditional Branch

A microinstruction contains a **branch address** — CAR jumps to that address regardless of any condition.

```
CAR = 105  →  execute microinstruction (contains: go to address 200)
CAR = 200  →  execution jumps here unconditionally
```

Used to **loop back** or jump to a different part of a microprogram.

---

### Method 3: Conditional Branch

CAR jumps to a branch address **only if a condition is true**. The condition is checked using **status flags (C, Z, S, V)** from the ALU.

```
CAR = 110  →  microinstruction: IF Z=1 THEN go to 300, ELSE go to 111

  If Z=1: CAR = 300  (branch taken)
  If Z=0: CAR = 111  (branch not taken, sequential)
```

Used to implement **if-else** and **loop** logic at the micro-level.

---

### Method 4: Mapping from IR

When a new machine instruction is fetched and loaded into IR, the **opcode is mapped** to the starting address of its microprogram. This is used to **start executing** a new instruction's microprogram.

```
IR opcode = 0101  (e.g., ADD instruction)
Mapping Logic → starting address = 050

CAR = 050  →  microprogram for ADD begins executing
```

---

### Address Sequencing — Summary

| Method | When Used | How CAR is Updated |
|--------|-----------|-------------------|
| Increment (CAR + 1) | Sequential microinstructions | CAR = CAR + 1 |
| Unconditional Branch | Always jump to a fixed address | CAR = branch address |
| Conditional Branch | Jump only if flag condition is true | CAR = branch address (if condition true) else CAR + 1 |
| Mapping from IR | New machine instruction fetched | CAR = mapped start address from opcode |

## 5. Microinstruction Format

Every microinstruction stored in control memory has **two main fields**:

```
┌────────────────────────────────┬──────────────────────────┐
│       Operations Field         │     Sequencing Field     │
│  (WHAT to do in this step)     │  (WHERE to go next)      │
│                                │                          │
│  • ALU operation select        │  • Next address bits     │
│  • Register load signals       │  • Branch address        │
│  • Bus enable signals          │  • Condition select      │
│  • Memory read/write signals   │    (which flag to check) │
└────────────────────────────────┴──────────────────────────┘
```

### Operations Field
Contains control bits that directly **activate hardware components** in the datapath:

| Control Bit | Action |
|-------------|--------|
| ALU select bits | Tell ALU which operation to perform |
| Register Load (LD) | Load result into a specific register |
| Bus Enable (EN) | Put a register's value onto the bus |
| Memory Read (RD) | Read data from memory into DR |
| Memory Write (WR) | Write data from DR into memory |
| PC Increment | Increment the Program Counter |

### Sequencing Field
Contains information about **where to go next**:

| Sub-field | Purpose |
|-----------|---------|
| Next Address | Address of next microinstruction |
| Branch Condition | Which flag (C, Z, S, V) to check |
| Branch Type | Unconditional / Conditional / Map from IR |

## 6. Types of Microinstructions

### 6.1 Horizontal Microinstruction

**Concept:** Each bit in the operations field corresponds to **exactly one control signal**. No encoding is used.

```
Example (simplified 12 control signals):

Bit:  [ALU_ADD][ALU_SUB][LD_R1][LD_R2][EN_R1][EN_R2][RD][WR][BUS][PC_INC][...]
       1        0        1      0      0      1      0   0   1    0
```

**Characteristics:**
- Word is **very wide** (one bit per signal)
- **No decoder needed** — bits directly drive hardware
- **Multiple operations** can happen in one microinstruction (high parallelism)
- **Fast** — no decoding delay
- Uses **more control memory width**

---

### 6.2 Vertical Microinstruction

**Concept:** Control signals are **encoded** into a smaller number of bits. A decoder converts the encoded bits back into actual control signals.

```
Example: Instead of 8 bits for 8 ALU functions:
  Horizontal: [1][0][0][0][0][0][0][0]  → ADD
  Vertical:   [000]  → ADD  (3 bits encode 8 functions using a decoder)
```

**Characteristics:**
- Word is **narrow** (encoded bits)
- **Decoder required** to generate actual control signals
- **Less parallelism** — fewer simultaneous operations possible
- **Slower** — decoding takes extra time
- Uses **less control memory width**

---

### Horizontal vs Vertical — Comparison Table

| Feature | Horizontal | Vertical |
|---------|-----------|---------|
| Instruction width | Very **wide** | **Narrow** |
| Encoding | None (1 bit = 1 signal) | Encoded (n bits → 2ⁿ signals) |
| Decoder needed |  No |  Yes |
| Parallelism | **High** | Low |
| Speed | **Fast** | Slower |
| Control memory width | **More** | Less |
| Complexity of design | Simple | More complex |
| Flexibility | Less flexible | More flexible |

> **Memory trick:**
> - **Horizontal** = Wide format, like a horizontal banner — all signals spread out, nothing hidden
> - **Vertical** = Compact format, like a compressed zip file — decoded when needed

## 7. Fetch Cycle as a Microprogram

The fetch cycle (getting an instruction from memory) is itself implemented as a microprogram stored in control memory. It runs before every machine instruction.

```
Microinstruction 1:  AR ← PC
                     (Load address register with program counter value)

Microinstruction 2:  DR ← M[AR],  PC ← PC + 1
                     (Read memory at that address into DR,
                      increment PC to point to next instruction)

Microinstruction 3:  IR ← DR
                     (Load instruction register with fetched instruction)

Microinstruction 4:  Decode IR → CAR ← mapped address
                     (Map IR opcode to start of its microprogram,
                      jump there to execute the instruction)
```

This shows that even the **most basic CPU operation** (fetching an instruction) is just a sequence of microinstructions.

---

### Example — ADD Instruction Microprogram

Suppose machine instruction `ADD R1, R2` (add R2 to R1, store in R1):

```
After fetch cycle, IR = ADD opcode
Mapping Logic → CAR jumps to ADD microprogram start address

Microinstruction 1:  A ← R1
                     (Load operand A from R1)

Microinstruction 2:  B ← R2
                     (Load operand B from R2)

Microinstruction 3:  F ← A + B
                     (ALU performs addition, S1=0 S0=1 Cin=0)

Microinstruction 4:  R1 ← F
                     (Store result back into R1)

Microinstruction 5:  Go to Fetch cycle
                     (Unconditional branch back to start of fetch)
```

## 8. Micro-Programmed vs Hardwired Control — Comparison

| Feature | Hardwired Control | Micro-Programmed Control |
|---------|------------------|--------------------------|
| Implementation | Combinational logic (gates, FFs) | Control memory (ROM/RAM) |
| Speed | **Fast** | Slower (memory access needed) |
| Flexibility | **Rigid** — hardware must be redesigned | **Flexible** — reprogramme memory |
| Complexity | Very complex for large ISA | Easier to design and extend |
| Modification | Requires hardware change | Just update control memory |
| Used in | **RISC** processors | **CISC** processors (Intel x86) |
| Cost | Higher for complex instruction sets | Lower design cost |
| Debugging | Difficult | Easier |
| New instructions | Cannot add without redesign | Add new microprogram |

> **One-line exam answer:** RISC → Hardwired (few simple instructions, speed priority). CISC → Micro-programmed (many complex instructions, flexibility priority).

## 9. Quick Revision Summary

```
Micro-Programmed Control:
  → Control signals stored in CONTROL MEMORY as microinstructions
  → Each machine instruction = one microprogram (sequence of microinstructions)
  → Introduced by Maurice Wilkes, 1951

Key Registers:
  CAR  → Address of current microinstruction (like PC for control memory)
  CDR  → Holds fetched microinstruction
  SBR  → Saves return address for subroutine calls

Address Sequencing (4 methods):
  1. CAR + 1         → Sequential
  2. Unconditional   → Always branch
  3. Conditional     → Branch if flag true
  4. Map from IR     → Start new instruction's microprogram

Microinstruction Format:
  Operations Field  → Control signals (what to do)
  Sequencing Field  → Next address (where to go)

Horizontal MI → Wide, no decoder, fast, parallel
Vertical MI   → Narrow, decoder needed, slower, compact
```
