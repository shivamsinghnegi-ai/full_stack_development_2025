# Cache Performance
 
### 1. AMAT Formula — Most Important Formula
 
**AMAT = Average Memory Access Time**
 
```
AMAT = Hit Time + (Miss Rate × Miss Penalty)
```
 
| Term | Meaning |
|---|---|
| **Hit Time** | Time to access cache on a hit (tc) |
| **Miss Rate** | Fraction of accesses that miss = 1 − Hit Rate |
| **Miss Penalty** | Extra time needed to go to main memory on a miss (tm) |
 
---
 
### 2. AMAT — Solved Numericals
 
#### Numerical 1 (Basic)
 
**Given:**
- Cache access time (tc) = 10 ns
- Main memory access time (tm) = 100 ns
- Hit rate (h) = 0.9
 
**Solution:**
```
Miss Rate = 1 − 0.9 = 0.1
AMAT = 10 + (0.1 × 100)
AMAT = 10 + 10
AMAT = 20 ns
```
 
---
 
#### Numerical 2 (Alternative Formula)
 
Some questions give this form:
```
AMAT = (h × tc) + ((1 − h) × tm)
```
*(Where tm includes the cache time — i.e., tm is total time on miss)*
 
**Given:**
- Cache access time (tc) = 5 ns
- Main memory access time (tm) = 50 ns
- Hit rate (h) = 0.95
 
**Solution:**
```
AMAT = (0.95 × 5) + (0.05 × 50)
AMAT = 4.75 + 2.5
AMAT = 7.25 ns
```
 
---
 
#### Numerical 3 (Two-level cache)
 
**Given:**
- L1 cache time = 2 ns, Hit rate = 0.85
- L2 cache time = 10 ns, Hit rate = 0.95 (of L1 misses)
- Main memory time = 100 ns
 
**Solution:**
```
L2 Miss Rate  = 1 − 0.95 = 0.05
L1 Miss Rate  = 1 − 0.85 = 0.15
 
AMAT = L1 Hit Time + L1 Miss Rate × (L2 Hit Time + L2 Miss Rate × MM Time)
AMAT = 2 + 0.15 × (10 + 0.05 × 100)
AMAT = 2 + 0.15 × (10 + 5)
AMAT = 2 + 0.15 × 15
AMAT = 2 + 2.25
AMAT = 4.25 ns
```

### 3. Write Policies 
 
When CPU **writes data**, the cache is updated. But what about main memory? Two policies handle this:
 
#### Write-Through
 
**Rule:** Every write to cache is **simultaneously written to main memory** as well.
 
```
CPU writes → Update Cache → Also update RAM immediately
```
 
| Advantage | Disadvantage |
|---|---|
| Main memory always up-to-date | **Slow** — every write goes to RAM |
| Simple to implement | High memory bus traffic |
| No data loss if cache fails | Defeats the purpose of cache for writes |
 
---
 
#### Write-Back (Also called Copy-Back)
 
**Rule:** Write only to cache. Main memory is updated **only when the cache block is replaced** (evicted).
 
- A **dirty bit** is maintained per cache block:
  - `Dirty bit = 1` → block was modified, must write back to RAM when replaced
  - `Dirty bit = 0` → block not modified, can simply discard on replacement
 
```
CPU writes → Update Cache only → Set dirty bit = 1
On eviction → If dirty bit = 1 → Write back to RAM
```
 
| Advantage | Disadvantage |
|---|---|
| **Fast** — writes stay in cache | Complex — need dirty bit logic |
| Less memory bus traffic | Main memory may be stale temporarily |
| Efficient for multiple writes to same block | Data loss risk if cache fails before writeback |
 
---
 
#### Write-Through vs Write-Back — Comparison
 
| Feature | Write-Through | Write-Back |
|---|---|---|
| When RAM is updated | **Every write** | **Only on eviction** |
| Speed | Slower | **Faster** |
| Memory bus traffic | High | Low |
| Dirty bit needed? | No | **Yes** |
| Main memory consistency | Always consistent | May be stale |
| Used when | Simplicity needed | **Performance needed** |
 
---
 
### 4. Write Miss Policies
 
When CPU writes to an address **not in cache** (write miss):
 
| Policy | What happens |
|---|---|
| **Write Allocate** | Load block into cache, then write to it (used with Write-Back) |
| **No Write Allocate** | Write directly to RAM, don't load into cache (used with Write-Through) |
 
> **Common pairing:**
> - Write-Back + Write Allocate
> - Write-Through + No Write Allocate
