# Multiprocessor Characteristics + Flynn's Taxonomy + Interconnection Structures

---

## 1. Introduction — Why Multiprocessors?

### The Single Processor Limitation

A single CPU can only execute one instruction at a time (in a basic model). No matter how fast it gets, there is a physical limit — clock speed cannot increase indefinitely due to:
- Heat generation
- Power consumption
- Signal propagation delays

### The Solution — Multiple Processors

Instead of making one processor faster, use **multiple processors working together** to solve problems faster.

**Goals of multiprocessor systems:**
1. **Higher throughput** — more work done per unit time
2. **Fault tolerance** — if one processor fails, others continue
3. **Better price-performance ratio** — multiple cheap processors vs one expensive fast processor
4. **Scalability** — add more processors to increase performance

## 2. Characteristics of Multiprocessor Systems

### 2.1 Definition

A **multiprocessor system** is a computer system with **two or more processors** that:
- Share a common physical memory
- Share I/O devices and peripherals
- Are controlled by a single operating system
- Communicate and cooperate to solve problems

### 2.2 Key Characteristics

#### Characteristic 1 — Multiple CPUs

More than one CPU in the same system. Each CPU can independently fetch and execute instructions.

#### Characteristic 2 — Shared Memory

All processors can access the same main memory (RAM). This allows them to communicate by reading and writing shared data.

#### Characteristic 3 — Single Operating System

One OS manages all processors, allocates work, and coordinates access to shared resources.

#### Characteristic 4 — Communication via Shared Memory or Message Passing

Processors coordinate work either by:
- Writing to and reading from shared memory locations
- Sending messages to each other through a communication network

#### Characteristic 5 — Parallelism

Multiple processors execute different parts of a program simultaneously — true parallel execution.

---

### 2.3 Tightly Coupled vs Loosely Coupled Systems

This is an important classification of multiprocessor systems.

#### Tightly Coupled Systems

- All processors share a **single, centralized main memory**
- Processors communicate through **shared memory**
- Connected via a **common high-speed bus or switch**
- Also called **Shared Memory Multiprocessors**
- OS has full control and visibility over all processors

```
     CPU 1   CPU 2   CPU 3   CPU 4
       |       |       |       |
       +-------+-------+-------+
                   |
             Shared Memory
                (Single RAM)
                   |
             Shared I/O Devices
```

**Advantages:**
- Fast communication (through shared memory — no network needed)
- Simpler programming model
- Single OS manages everything

**Disadvantages:**
- Memory becomes a bottleneck — all processors competing for one memory
- Does not scale well — adding more processors increases contention
- Expensive to build with high-speed interconnects

**Examples:** Symmetric Multiprocessing (SMP) systems — most modern multi-core desktop computers

---

#### Loosely Coupled Systems

- Each processor has its **own private local memory**
- Processors communicate by **sending messages** over a network
- Also called **Distributed Memory Multiprocessors** or **Message Passing Systems**
- Each node is essentially a complete computer

```
  CPU1+Mem1    CPU2+Mem2    CPU3+Mem3
      |            |            |
      +------------+------------+
         Interconnection Network
```

**Advantages:**
- Highly scalable — add more nodes easily
- No shared memory bottleneck
- Can span large geographic distances (cluster, grid)

**Disadvantages:**
- Communication is slower (network vs memory access)
- More complex programming — explicit message passing
- No shared address space

**Examples:** Compute clusters, cloud computing nodes, distributed systems

---

#### Tightly Coupled vs Loosely Coupled — Comparison Table

| Feature | Tightly Coupled | Loosely Coupled |
|---|---|---|
| Memory | Shared (single) | Private per processor |
| Communication | Shared memory reads/writes | Message passing over network |
| Speed | Fast communication | Slower communication |
| Scalability | Limited | High |
| OS | Single OS for all | Each node can have own OS |
| Example | SMP (multi-core desktop) | Cluster, grid computing |
| Bottleneck | Memory bus | Network bandwidth |

## 3. Flynn's Taxonomy

Flynn's taxonomy is a classification of computer architectures based on two dimensions:

1. **Number of instruction streams** — how many instruction sequences are being executed
2. **Number of data streams** — how many data sequences are being processed

This gives four combinations:

```
                    Single Data     Multiple Data
Single Instruction:    SISD             SIMD
Multiple Instruction:  MISD             MIMD
```

---

### 3.1 SISD — Single Instruction Single Data

**One instruction stream, one data stream.**

This is the classic **von Neumann single processor** model.

- One processor fetches and executes one instruction at a time
- Each instruction operates on one data element at a time
- No parallelism

```
Instruction Stream: I1 → I2 → I3 → I4 (one at a time)
Data Stream:        D1 → D2 → D3 → D4 (one at a time)

    [Control Unit]
          |
    [Processing Unit] ← [Memory]
```

**Examples:**
- Traditional single-core processors
- Intel 8086, early microprocessors

**Characteristics:**
- Sequential execution
- Simple architecture
- No parallel processing

---

### 3.2 SIMD — Single Instruction Multiple Data

**One instruction stream, multiple data streams.**

A single instruction is broadcast to multiple processing units simultaneously, each of which operates on different data.

- One control unit issues one instruction
- Multiple ALUs execute the same instruction on different data in parallel
- Best for tasks that apply the same operation to large arrays of data

```
                [Control Unit]
                      |
         One instruction broadcast to all
              |        |        |
           [PE1]    [PE2]    [PE3]     (PE = Processing Element)
             |        |        |
            D1       D2       D3       (Different data)
```

**Example operation:** Add 5 to every element of an array of 1000 numbers.
- SISD: 1000 separate ADD instructions
- SIMD: 1 ADD instruction applied to all 1000 elements simultaneously

**Examples:**
- Vector processors
- GPU (Graphics Processing Unit) — thousands of cores doing same operation on pixels
- Intel SSE, AVX instructions (multimedia extensions)
- Array processors

**Characteristics:**
- High throughput for data-parallel tasks
- Efficient for scientific computing, image processing, graphics
- All processing elements must execute same instruction (limited flexibility)

---

### 3.3 MISD — Multiple Instruction Single Data

**Multiple instruction streams, single data stream.**

Multiple processors execute different instructions on the same data stream simultaneously.

```
Instruction Stream 1 → [PE1] ─┐
Instruction Stream 2 → [PE2] ─┼─ Same Data Stream D1
Instruction Stream 3 → [PE3] ─┘
```

**Practical use:** Very rare and theoretical in most textbooks.

Some consider **pipeline processors** as a loose example — different stages process the same data in different ways simultaneously.

**Examples:**
- Fault-tolerant systems — same data processed by multiple units for verification (space shuttle flight control)
- Systolic arrays (specialized signal processing)

**Characteristics:**
- Rarely used in practice
- Mainly theoretical
- Useful for redundancy and fault detection

---

### 3.4 MIMD — Multiple Instruction Multiple Data

**Multiple instruction streams, multiple data streams.**

Multiple processors each execute different instructions on different data simultaneously. This is true parallelism.

```
[CPU 1] → Instruction Stream 1 on Data Stream 1
[CPU 2] → Instruction Stream 2 on Data Stream 2
[CPU 3] → Instruction Stream 3 on Data Stream 3
[CPU 4] → Instruction Stream 4 on Data Stream 4

All executing independently and in parallel
```

**Examples:**
- Multi-core processors (Intel Core i7, AMD Ryzen)
- Symmetric Multiprocessing (SMP) servers
- Distributed computing clusters
- Most modern multiprocessor systems

**Sub-categories of MIMD:**
- **Shared Memory MIMD** — tightly coupled, processors share RAM
- **Distributed Memory MIMD** — loosely coupled, each processor has private memory

**Characteristics:**
- Most general and most powerful model
- Each processor has full independence
- Most real-world multiprocessors fall here
- Complex synchronization and communication needed

---

### 3.5 Flynn's Taxonomy — Master Summary Table

| Category | Instruction Streams | Data Streams | Parallelism | Example |
|---|---|---|---|---|
| **SISD** | 1 | 1 | None | Traditional single CPU |
| **SIMD** | 1 | Multiple | Data parallel | GPU, vector processor |
| **MISD** | Multiple | 1 | Instruction parallel | Fault-tolerant systems |
| **MIMD** | Multiple | Multiple | Full parallel | Multi-core, clusters |

## 4. Interconnection Structures

In a multiprocessor system, processors must communicate with each other and with shared memory and I/O. The **interconnection structure** is the hardware that enables this communication.

**The key requirements of an interconnection structure:**
- Allow any processor to access any memory module
- Handle simultaneous requests from multiple processors
- Minimize delay (latency)
- Maximize bandwidth (data throughput)
- Be cost-effective and scalable

Four major interconnection structures are covered:

1. Time Shared Bus
2. Multiport Memory
3. Crossbar Switch
4. Multistage Interconnection Network

---

### 4.1 Time Shared Bus (Common Bus)

#### Structure

All processors, memory modules, and I/O devices are connected to a **single shared bus**.

```
CPU1  CPU2  CPU3  CPU4  MEM1  MEM2  I/O1  I/O2
 |     |     |     |     |     |     |     |
 +-----+-----+-----+-----+-----+-----+-----+
                  Single Shared Bus
```

#### How it Works

- Only **one device** can use the bus at a time
- If a processor wants to access memory, it first checks if bus is free
- If bus is busy — processor must **wait** (bus arbitration decides who gets it)
- Bus arbitration schemes: daisy chain, polling, independent requesting

#### Advantages

- **Simplest and cheapest** design
- Easy to add new processors (just connect to bus)
- Easy to implement cache coherence (all processors see all bus transactions)

#### Disadvantages

- **Bus is a bottleneck** — one transfer at a time
- Performance degrades quickly as more processors are added
- **Not scalable** — beyond 4-8 processors, bus becomes saturated
- Any bus failure brings down the entire system

#### Use Case

- Small multiprocessor systems (2–8 processors)
- Low-cost systems where performance is not critical

---

### 4.2 Multiport Memory

#### Structure

Each **memory module** has **multiple ports** — one dedicated port for each processor (and possibly I/O).

```
        CPU1    CPU2    CPU3    CPU4
         |       |       |       |
         |       |       |       |
      [Port1] [Port2] [Port3] [Port4]
      +--------------------------------+
      |       Memory Module 1          |
      +--------------------------------+

      [Port1] [Port2] [Port3] [Port4]
      +--------------------------------+
      |       Memory Module 2          |
      +--------------------------------+
```

Each processor has a dedicated path to each memory module. No sharing of the path.

#### How it Works

- Each processor has its own dedicated connection to each memory module
- Multiple processors can access **different memory modules** simultaneously
- If two processors want the same memory module at the same time — one must wait (handled by arbitration logic inside the memory module)

#### Advantages

- **Multiple simultaneous accesses** to different memory modules
- Higher bandwidth than shared bus
- No single bus bottleneck

#### Disadvantages

- **Very expensive** — number of ports = number of processors × number of memory modules
- **Complex hardware** — each memory module needs multiple port controllers
- **Wiring complexity** increases rapidly with more processors
- Does not scale well for large systems

#### Use Case

- Small to medium multiprocessor systems where higher bandwidth is needed

---

### 4.3 Crossbar Switch

#### Structure

A **crossbar switch** creates a grid of switches that can connect **any processor to any memory module** at the same time, as long as no two processors are accessing the same memory module simultaneously.

```
         MEM1   MEM2   MEM3   MEM4
          |      |      |      |
CPU1 ----[X]---[X]---[X]---[X]----
          |      |      |      |
CPU2 ----[X]---[X]---[X]---[X]----
          |      |      |      |
CPU3 ----[X]---[X]---[X]---[X]----
          |      |      |      |
CPU4 ----[X]---[X]---[X]---[X]----

[X] = Switch point — close to connect, open to disconnect
```

If CPU1 wants MEM2 and CPU3 wants MEM4 — both switches close simultaneously. Both accesses happen in parallel with no conflict.

#### How it Works

- N processors × M memory modules requires N × M switch points
- Switch points are set by a control unit
- Any non-conflicting set of accesses can happen simultaneously
- If conflict (two CPUs want same memory) — one waits

#### Advantages

- **Multiple simultaneous transfers** — maximum bandwidth
- **No bottleneck** for non-conflicting accesses
- **Fast** — direct connection without intermediate nodes

#### Disadvantages

- **Very expensive** — N processors × M memory modules = N×M switches
- For 16 processors and 16 memories = 256 switches
- **Complexity grows as N²** — not scalable for large N
- Physical wiring becomes impractical for large systems

#### Use Case

- Medium-sized high-performance systems (8–64 processors)
- When maximum bandwidth is needed and cost is acceptable

---

### 4.4 Multistage Interconnection Network (MIN)

#### Structure

Instead of a full crossbar (which needs N² switches), a **multistage network** uses multiple stages of smaller switches arranged in levels to connect processors to memory.

The most common example is the **Omega Network** (also called Butterfly Network or Banyan Network).

```
For 4 processors and 4 memory modules — 2 stages:

CPU1 ─┬─[SW]─┬─[SW]─ MEM1
      │      ×      
CPU2 ─┘ [SW]─┘ [SW]─ MEM2
      
CPU3 ─┬─[SW]─┬─[SW]─ MEM3
      │      ×      
CPU4 ─┘ [SW]─┘ [SW]─ MEM4

Each [SW] is a 2×2 switch (2 inputs, 2 outputs)
Number of stages = log₂(N)
Total switches   = (N/2) × log₂(N)
```

For N = 8:
- Stages = log₂(8) = 3
- Switches = (8/2) × 3 = 12 (vs 64 for full crossbar)

#### How it Works

- A packet (request) travels through multiple stages
- At each stage, the switch routes it left or right based on destination address bits
- Path is determined by the binary address of the destination memory module

#### Blocking Problem

A key limitation of multistage networks is **blocking** — even if the source and destination are free, the path through intermediate switches may be occupied by another transfer, preventing the connection.

(The crossbar is non-blocking — multistage networks can be blocking.)

#### Advantages

- **Much cheaper** than crossbar — O(N log N) switches vs O(N²)
- Scalable for large N
- Reasonable bandwidth
- Good balance of cost and performance

#### Disadvantages

- **Blocking possible** — path conflicts through intermediate switches
- **Higher latency** — data must pass through multiple stages
- More complex routing logic

#### Use Case

- Large multiprocessor systems where crossbar is too expensive
- Supercomputers, large servers

---

### 4.5 Interconnection Structures — Comparison Table

| Feature | Time Shared Bus | Multiport Memory | Crossbar Switch | Multistage Network |
|---|---|---|---|---|
| Structure | Single shared bus | Multiple ports per memory | N×M switch grid | Multiple switch stages |
| Simultaneous transfers | **1 only** | Multiple (diff. modules) | **N simultaneous** | Multiple (may block) |
| Cost | **Cheapest** | Expensive | **Most expensive** | Moderate |
| Scalability | **Poor** | Poor | Poor (N²) | **Good** |
| Performance | **Lowest** | Medium | **Highest** | Good |
| Blocking | Yes (bus busy) | Possible (same module) | **No** (non-blocking) | Yes (path conflicts) |
| Complexity | **Simplest** | Moderate | Complex | Moderate |
| Used for | Small systems | Small-medium | Medium systems | **Large systems** |
| Switches needed | — | — | N × M | (N/2) × log₂N |

---


## 5. Important Formulas — Multistage Networks

```
Number of stages   = log₂(N)         where N = number of processors
Switches per stage = N/2
Total switches     = (N/2) × log₂(N)

Example: N = 8 processors
Stages         = log₂(8) = 3
Switches/stage = 8/2 = 4
Total switches = 4 × 3 = 12

Crossbar for N=8: N × N = 8 × 8 = 64 switches
Multistage N=8:   (N/2) × log₂N = 12 switches   (much cheaper)
```
