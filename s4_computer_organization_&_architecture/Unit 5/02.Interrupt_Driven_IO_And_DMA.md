# Interrupt Driven I/O + DMA + Full I/O Comparison

---

## 1. Interrupt Driven I/O 

### 1.1 Core Idea

> **Instead of CPU continuously checking the device, the device itself notifies the CPU when it is ready by sending an INTERRUPT signal.**

CPU issues an I/O command, then goes back to doing other useful work. When the device is ready, it raises an interrupt. CPU stops current work, handles the I/O, then resumes.

**Key difference from Programmed I/O:**

| | Programmed I/O | Interrupt Driven I/O |
|---|---|---|
| Who checks status | **CPU** (polls repeatedly) | **Device** (notifies CPU) |
| CPU during wait | **Wasted** (busy-waiting) | **Free** (does other work) |
| Efficiency | Low | Much better |

---

### 1.2 How Interrupt Driven I/O Works — Step by Step

**Example: CPU wants to read from keyboard**

```
Step 1: CPU writes command to I/O Control Register
        → "Keyboard, get ready to send a character"
        → CPU also enables interrupts for this device

Step 2: CPU goes back to executing its own program
        → CPU is FREE — does useful work
        → Device is working in background

Step 3: When device has data ready:
        → Device raises INTERRUPT REQUEST (IRQ) signal on the bus

Step 4: CPU finishes its current instruction
        → CPU acknowledges the interrupt
        → CPU saves its current state (PC, registers) onto the stack

Step 5: CPU jumps to ISR — Interrupt Service Routine
        → ISR is a special function written to handle this specific interrupt
        → ISR reads data from Data Register
        → ISR processes/stores the data

Step 6: ISR completes
        → CPU restores saved state from stack
        → CPU resumes the program from where it left off
```

---

### 1.3 Interrupt Flow Diagram 

```
CPU running Program A
        ↓
CPU issues I/O command → continues Program A
        ↓
[Device working in background...]
        ↓
Device ready → raises INTERRUPT signal
        ↓
CPU finishes current instruction
        ↓
CPU saves state (PC, registers) → Stack
        ↓
CPU jumps to ISR (Interrupt Service Routine)
        ↓
ISR reads data from I/O Data Register
        ↓
ISR completes
        ↓
CPU restores state from Stack
        ↓
CPU resumes Program A from where it stopped
```

---

### 1.4 ISR — Interrupt Service Routine

**ISR** (also called **Interrupt Handler**) is a special function/subroutine that:
- Is written specifically to handle a particular interrupt
- Stored at a known memory address
- CPU automatically jumps here when the interrupt occurs
- After ISR completes → CPU returns to interrupted program

**Interrupt Vector Table (IVT):**
- A table stored in memory that maps each interrupt number to the address of its ISR
- When interrupt occurs → CPU looks up interrupt number in IVT → jumps to that ISR address

```
Interrupt Vector Table:
┌──────────────┬────────────────────┐
│ Interrupt No.│ ISR Address        │
├──────────────┼────────────────────┤
│      0       │ 0x1000 (Divide by 0)│
│      1       │ 0x2000 (Keyboard)  │
│      2       │ 0x3000 (Timer)     │
│      3       │ 0x4000 (Disk)      │
└──────────────┴────────────────────┘
```

---

### 1.5 Types of Interrupts 

#### By Source:

**Hardware Interrupts:**
- Generated by external hardware devices
- Examples: keyboard press, mouse click, disk transfer complete, timer tick
- These are **asynchronous** — happen at any time, unpredictable

**Software Interrupts (Traps):**
- Generated by a program executing a special instruction (`INT n`)
- Used to request OS services (system calls)
- Examples: `INT 21h` in DOS for OS services
- These are **synchronous** — happen at a specific point in program

**Exception (Internal Interrupt):**
- Generated by the CPU itself when an error occurs during execution
- Examples: divide by zero, invalid memory access, illegal instruction
- Happens synchronously at the instruction that caused the error

---

#### By Maskability:

**Maskable Interrupts (MI):**
- Can be **disabled (masked)** by the CPU using a flag bit (Interrupt Flag — IF)
- When IF = 0 → CPU ignores these interrupts
- When IF = 1 → CPU responds to these interrupts
- Used for regular device interrupts (keyboard, disk, etc.)
- CPU uses `CLI` instruction to clear IF (disable) and `STI` to set IF (enable)

**Non-Maskable Interrupts (NMI):**
- **Cannot be disabled** — CPU must always respond
- Used for critical errors that must be handled immediately
- Examples: hardware failure, memory parity error, power failure warning
- Higher priority than maskable interrupts

| Feature | Maskable | Non-Maskable |
|---|---|---|
| Can be disabled? | **Yes** (using IF flag) | **No** |
| Priority | Normal | **Highest** |
| Used for | Regular I/O devices | Critical hardware errors |
| Example | Keyboard, disk | Power failure, memory error |

---

#### By Timing:

**Synchronous Interrupts:**
- Occur at a **specific, predictable point** in program execution
- Examples: software interrupts, exceptions

**Asynchronous Interrupts:**
- Occur at **any time** — not tied to any specific instruction
- Examples: hardware interrupts from devices

---

### 1.6 Interrupt Priority

When multiple devices raise interrupts simultaneously, CPU must decide which to handle first.

**Priority schemes:**
1. **Fixed priority** — higher priority device always wins (NMI > hardware IRQ > software)
2. **Daisy chaining** — devices connected in a chain; device closer to CPU has higher priority
3. **Priority encoder** — hardware circuit selects highest priority interrupt

---

### 1.7 Advantages and Disadvantages of Interrupt Driven I/O

**Advantages:**
1. **CPU is free** during I/O — does useful work instead of polling
2. **Much more efficient** than programmed I/O
3. **Multiple devices** can be handled — each raises interrupt when ready
4. **No wasted cycles** on polling

**Disadvantages:**
1. **Context switching overhead** — saving and restoring CPU state takes time
2. **Complex to implement** — ISR must be carefully written
3. **Interrupt latency** — small delay between interrupt raised and ISR starts
4. **For high-speed devices** (disk, network) — too many interrupts → CPU overloaded
   → This is where DMA is needed

## 2. DMA — Direct Memory Access

### 2.1 The Problem with Interrupt Driven I/O for Fast Devices

Interrupt Driven I/O works well for slow devices like keyboard. But for fast devices like hard disk — large blocks of data (megabytes) must be transferred.

**Without DMA:**
- Disk raises interrupt → CPU reads 1 byte → CPU stores it in memory → repeat for next byte
- For 1 MB transfer → 1,048,576 interrupts!
- CPU is completely occupied with data transfer — cannot do anything else
- This defeats the purpose of interrupts

**Solution → DMA**

---

### 2.2 What is DMA?

> **DMA (Direct Memory Access) allows I/O devices to transfer data directly to/from main memory WITHOUT CPU involvement.**

A special hardware unit called the **DMA Controller (DMAC)** manages this transfer.

CPU only:
1. Initiates the transfer (tells DMA Controller what to do)
2. Is notified when transfer is complete (via interrupt)

CPU does NOT:
- Move any actual data bytes
- Get involved during the transfer

---

### 2.3 DMA Controller

The **DMA Controller** is a dedicated hardware chip/circuit that can:
- Access the system bus independently (like a CPU)
- Read from I/O device and write to memory directly
- Work in parallel with CPU

**DMA Controller has these registers:**

| Register | Purpose |
|---|---|
| **Memory Address Register** | Starting address in RAM where data is to be stored |
| **Count Register** | Number of bytes/words to transfer |
| **Control Register** | Direction (read/write), mode of transfer |
| **Status Register** | Transfer status — busy, done, error |

---

### 2.4 DMA Working — Step by Step 

**Example: Disk → Memory transfer of 4 KB**

```
Step 1: CPU programs the DMA Controller
        → Memory address register = 0x5000 (where to store data in RAM)
        → Count register = 4096 (4 KB = 4096 bytes)
        → Control register = READ from disk

Step 2: CPU gives GO signal to DMA Controller
        → CPU returns to its own program — CPU is FREE

Step 3: DMA Controller requests bus from CPU
        → DMA sends BUS REQUEST signal to CPU

Step 4: CPU grants bus to DMA
        → CPU sends BUS GRANT signal
        → CPU temporarily stops using bus (for one cycle)

Step 5: DMA Controller reads data from Disk (I/O device)
        → Writes it directly into RAM at address 0x5000
        → Increments address register (0x5001, 0x5002...)
        → Decrements count register (4095, 4094...)

Step 6: DMA repeats Steps 3–5 until count = 0

Step 7: Transfer complete (count = 0)
        → DMA raises INTERRUPT to CPU
        → CPU now processes the transferred data

Step 8: CPU handles DMA completion interrupt
        → Data is already in RAM — CPU just works with it
```

---

### 2.5 DMA Working Diagram 

```
┌─────┐    ① Program DMA    ┌──────────────┐
│ CPU │ ─────────────────→  │ DMA Controller│
│     │ ←─────────────────  │  (DMAC)      │  ⑦ Interrupt when done
└─────┘    ⑦ Interrupt      └──────┬───────┘
   │                               │
   │ ② CPU does other work         │ ③④ Bus Request/Grant
   │                               ↕
   │                         System Bus
   │                          ↙       ↘
   │                   ┌─────┐       ┌──────────┐
   │                   │ RAM │       │ I/O Device│
   │                   │     │ ←─────│  (Disk)  │
   │                   │Data │  ⑤⑥   └──────────┘
   │                   │here │  Direct Transfer
   └──────────────────→└─────┘
      ⑧ CPU processes data
        already in RAM
```

### 2.6 DMA Transfer Modes 

#### Mode 1 — Cycle Stealing

> DMA **steals one bus cycle** at a time from CPU, transfers one word, returns bus to CPU.

```
CPU: |──use──|──use──|──STOLEN──|──use──|──use──|──STOLEN──|
DMA:                 |──transfer─|                |──transfer─|
```

- CPU and DMA **share the bus** — alternate use
- CPU slowed down slightly (stolen cycles)
- Good for **continuous, slow stream** of data
- Data transferred **one word at a time**

---

#### Mode 2 — Burst Mode (Block Transfer)

> DMA **takes control of the bus** for the entire duration of the transfer, then releases it.

```
CPU: |──use──|──────waiting──────────────────|──use──|
DMA:         |═══════ entire block transfer ══|
```

- DMA holds bus until ALL data is transferred
- CPU completely **blocked** during transfer
- Fast — no overhead of repeated bus request/grant
- Good for transferring **large blocks** (disk sector, network packet)
- CPU can do nothing during burst

---

#### Mode 3 — Transparent Mode (Hidden DMA)

> DMA transfers data **only when CPU is not using the bus** (e.g., during internal operations like instruction execution).

```
CPU: |─ executing (no bus needed)─|─ bus access ─|─ executing ─|
DMA:  |────── transfer here ───────|              |── transfer ──|
```

- Most efficient — **no interference with CPU at all**
- DMA "steals" cycles that CPU is not using anyway
- Slowest of the three modes (DMA must wait for CPU to free bus)
- Complex to implement (must detect when bus is free)

---

### 2.7 DMA Transfer Modes — Comparison Table

| Feature | Cycle Stealing | Burst Mode | Transparent Mode |
|---|---|---|---|
| Bus control | **Alternate** (one cycle at a time) | **DMA holds entire block** | **Only when CPU idle** |
| CPU impact | Slightly slowed | **Completely blocked** | **Not affected at all** |
| Transfer speed | Moderate | **Fastest** | Slowest |
| Complexity | Medium | Simple | Most complex |
| Best for | Continuous data stream | Large block transfer | No CPU disruption needed |

---

### 2.8 Advantages and Disadvantages of DMA

**Advantages:**
1. **CPU completely free** during data transfer — maximum parallelism
2. **Very fast** for large data transfers (disk, network, video)
3. **Reduces interrupt overhead** — only one interrupt at end of entire block
4. **Efficient bus use** — dedicated hardware for data movement

**Disadvantages:**
1. **Extra hardware cost** — DMA controller chip required
2. **Bus contention** — DMA and CPU compete for bus
3. **Cache coherence issue** — DMA writes to RAM directly; CPU cache may have stale data
4. **Complex design** — requires arbitration between CPU and DMA

## 3. Full I/O Technique Comparison

### 3.1 Complete Comparison Table — Must Memorize

| Feature | Programmed I/O | Interrupt Driven I/O | DMA |
|---|---|---|---|
| Also called | Polling / Busy-Wait | Interrupt I/O | Direct Memory Access |
| CPU role | **Controls everything** | **Handles ISR** | **Only initiates + gets notified** |
| CPU during transfer | **Busy polling** | **Does other work** | **Completely free** |
| Who transfers data | **CPU** | **CPU (in ISR)** | **DMA Controller** |
| Data path | Device → CPU → Memory | Device → CPU → Memory | Device → **Memory directly** |
| CPU efficiency | **Lowest** | Medium | **Highest** |
| Hardware needed | **None extra** | Interrupt controller | **DMA controller** |
| Complexity | **Simplest** | Moderate | Most complex |
| Suitable for | Very slow devices | Slow-medium devices | **Fast, bulk transfers** |
| Interrupt used? | No | **Yes — per transfer** | **Yes — end of block only** |
| Transfer unit | 1 byte/word at a time | 1 byte/word at a time | **Entire block at once** |
| Speed | **Slowest** | Medium | **Fastest** |
| Example use | Simple embedded | Keyboard, mouse | Disk, network, graphics |

---

### 3.2 CPU Activity Comparison — Timeline

```
PROGRAMMED I/O:
CPU: |─poll─poll─poll─poll─poll─DATA─poll─poll─poll─poll─DATA─|
     ← CPU mostly wasted in polling →

INTERRUPT DRIVEN I/O:
CPU: |─useful work─────────────|ISR|─useful work──────────|ISR|─|
     ← CPU does real work → interrupt → quick ISR → back to work

DMA:
CPU: |─program DMA─|──────────────── useful work ──────────────|INT|
DMA:               |════════ entire block transferred ══════════|
     ← CPU completely free during transfer → one interrupt at end
```

---

### 3.3 Data Path Comparison

```
PROGRAMMED I/O and INTERRUPT DRIVEN I/O:
Device → I/O Bus → CPU Register → Memory Bus → RAM
         (data goes THROUGH CPU)

DMA:
Device → DMA Controller → Memory Bus → RAM
         (data goes AROUND CPU — directly to RAM)
```

## 4. Interrupt vs DMA — Key Difference

This is a very common confusion. Clarify it clearly:

| | Interrupt Driven I/O | DMA |
|---|---|---|
| CPU handles data? | **Yes** — CPU reads each byte in ISR | **No** — DMA moves data, CPU not involved |
| Number of interrupts | **One per byte/word** | **One per entire block** |
| CPU free during transfer? | **Partially** (free between interrupts) | **Completely free during transfer** |
| Best for | Small, infrequent transfers | Large, bulk transfers |
| Example | Keyboard (1 char at a time) | Disk (4 KB sector at a time) |
