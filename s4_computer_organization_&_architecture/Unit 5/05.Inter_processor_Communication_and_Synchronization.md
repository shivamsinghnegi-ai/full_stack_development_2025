# Inter-processor Communication
 
Once arbitration is resolved and a processor has access, processors need to exchange data and coordinate work. There are two main methods.
 
### 1. Shared Memory Communication
 
#### Concept
 
All processors can **read from and write to a common shared memory**. Communication happens by one processor writing data to a memory location and another reading from it.
 
```
CPU1 writes message to address 0x2000
            |
        Shared RAM
            |
CPU2 reads message from address 0x2000
```
 
#### How it Works
 
```
Producer-Consumer Example:
 
CPU1 (Producer):
  1. Write data to shared_buffer[i]
  2. Set flag = 1  (signals data is ready)
 
CPU2 (Consumer):
  1. Check if flag == 1
  2. If yes → read shared_buffer[i]
  3. Set flag = 0  (signals buffer consumed)
```
 
#### Advantages
 
- **Fast** — memory access speed (no network)
- **Simple programming model** — read and write like normal variables
- **Low overhead** — no message formatting or routing
 
#### Disadvantages
 
- **Requires tight coupling** — only works when processors share memory
- **Synchronization needed** — without proper coordination, data races occur
- **Does not scale** — shared memory bandwidth limits performance
- **Cache coherence problem** — each processor may have stale copy in its cache
 
---
 
### 2. Message Passing Communication
 
#### Concept
 
Processors communicate by **explicitly sending and receiving messages** through a network. Each processor has its own private memory. No shared address space.
 
```
CPU1:  send(message, destination=CPU3)
              |
         Network / Interconnect
              |
CPU3:  receive(message, source=CPU1)
```
 
#### How it Works
 
- CPU1 packages data into a message with a header (destination, size, type)
- Message is sent into the interconnection network
- Network routes message to CPU3
- CPU3 receives the message and extracts the data
 
#### Advantages
 
- **Scalable** — works for loosely coupled distributed systems
- **No shared memory needed** — each node is independent
- **Explicit communication** — cleaner programming model for distributed systems
- **No cache coherence issues** — no shared memory to keep consistent
 
#### Disadvantages
 
- **Slower** — network latency added to every communication
- **Overhead** — message packing, routing, unpacking takes time
- **Complex programming** — programmer must explicitly manage all communication
- **Bandwidth limited** by network
 
---
 
### 3. Shared Memory vs Message Passing — Comparison Table
 
| Feature | Shared Memory | Message Passing |
|---|---|---|
| Memory | Shared (common RAM) | Private per processor |
| Communication method | Read/Write to shared address | Send/Receive messages |
| Speed | Fast | Slower (network latency) |
| Scalability | Limited | High |
| Programming complexity | Simpler | More complex |
| Synchronization | Required (race conditions) | Built into send/receive |
| System type | Tightly coupled | Loosely coupled |
| Cache coherence | Problem exists | No issue |

# Synchronization
 
### 1. Why is Synchronization Needed?
 
In a multiprocessor system, multiple processors can access and modify shared data simultaneously. Without proper coordination, this leads to errors.
 
**The fundamental problem:** Operations that appear atomic (indivisible) to a programmer are actually composed of multiple steps at the hardware level.
 
---
 
### 2. Race Condition
 
#### Definition
 
A **race condition** occurs when the result of a computation depends on the **relative timing of execution** of two or more processes — and the result changes depending on which process runs first.
 
#### Classic Example — Shared Counter
 
Two processors both want to increment a shared counter variable.
 
```
Initial value: counter = 5
 
CPU1 wants to do: counter = counter + 1
CPU2 wants to do: counter = counter + 1
 
Expected final value: counter = 7
 
At hardware level, counter = counter + 1 requires 3 steps:
  Step A: LOAD  register ← counter    (read from memory)
  Step B: ADD   register = register + 1
  Step C: STORE counter ← register    (write back to memory)
```
 
**What can go wrong — interleaved execution:**
 
```
CPU1: A → reads counter = 5, stores 5 in R1
CPU2: A → reads counter = 5, stores 5 in R2     (CPU2 also read 5, not 6!)
CPU1: B → R1 = 5 + 1 = 6
CPU2: B → R2 = 5 + 1 = 6
CPU1: C → writes counter = 6
CPU2: C → writes counter = 6
 
Final value: counter = 6   (WRONG — should be 7!)
```
 
One increment was lost. This is a race condition.
 
**The order of execution matters.** The result "races" on which CPU finishes first.
 
---
 
### 3. Critical Section
 
#### Definition
 
A **critical section** is a portion of code that accesses shared resources (shared variables, files, hardware) and **must not be executed by more than one processor at the same time**.
 
```
Code structure:
 
[Entry section]   ← Request permission to enter
  [Critical Section]  ← Access shared resource here
[Exit section]    ← Release permission
[Remainder section]  ← Rest of the program
```
 
**Rules for a correct critical section solution:**
 
1. **Mutual Exclusion** — Only one process in critical section at a time
2. **Progress** — If no process is in critical section, a waiting process should eventually get in
3. **Bounded Waiting** — A process should not wait forever to enter critical section (no starvation)
 
---
 
### 4. Mutual Exclusion
 
**Mutual exclusion** is the guarantee that **only one processor can execute in the critical section at any given time**.
 
Methods to achieve mutual exclusion:
1. Disable interrupts
2. Test-and-Set lock (hardware)
3. Spin lock
4. Semaphore
 
---
 
### 5. Disabling Interrupts
 
Simplest approach on a single processor — disable interrupts before entering critical section, re-enable after.
 
```
CLI          ← Disable interrupts
[Critical Section]
STI          ← Re-enable interrupts
```
 
**Problem on multiprocessors:**
- Disabling interrupts on one CPU does not stop other CPUs
- Other CPUs can still access shared memory simultaneously
- **Does not work for multiprocessors** — only works on single CPU systems
 
---
 
### 6. Test-and-Set Instruction (Hardware Solution)
 
A hardware instruction called **Test-and-Set (TAS)** atomically reads a memory location and sets it to 1 in a single indivisible operation.
 
```
Test-and-Set(lock):
  temp = lock          ← Read current value
  lock = 1             ← Set to 1 (locked)
  return temp          ← Return old value
```
 
This entire operation is **atomic** — no other CPU can interrupt between read and write.
 
**Using Test-and-Set for mutual exclusion:**
 
```
while (Test-and-Set(lock) == 1) {
    // spin — wait until lock is free
}
// Critical Section
lock = 0    // Release lock
```
 
If `lock` was 0 (free) — TAS sets it to 1 and returns 0 → CPU enters critical section.
If `lock` was 1 (busy) — TAS sets it to 1 (no change) and returns 1 → CPU keeps spinning.
 
---
 
### 7. Spin Lock (Busy-Wait Lock)
 
A **spin lock** is a locking mechanism where a processor **continuously loops (spins)** checking the lock variable until it becomes free.
 
```
Acquire lock:
  while (lock == 1) { }  ← spin until lock is 0
  lock = 1               ← take the lock
 
[Critical Section]
 
Release lock:
  lock = 0               ← release the lock
```
 
#### Advantages of Spin Locks
 
- Very **fast** when lock is held for a very short time
- No context switch overhead — processor stays active
- Simple implementation
 
#### Disadvantages of Spin Locks
 
- **Wastes CPU cycles** — processor does nothing useful while spinning
- Bad for **long critical sections** — better to sleep and wake up
- Can cause **starvation** — no guarantee of fairness
 
#### When to Use Spin Locks
 
- When the lock is expected to be held for a **very short time** (microseconds)
- When context switch overhead would be larger than the wait time
- Common in OS kernel code and device drivers
 
---
 
### 8. Semaphore
 
#### Definition
 
A **semaphore** is a synchronization variable that contains an **integer value** and supports two atomic operations:
 
**Wait (P operation / Down):**
```
wait(S):
  while (S <= 0) { }  ← Wait until S > 0
  S = S - 1           ← Decrement (take one unit)
```
 
**Signal (V operation / Up):**
```
signal(S):
  S = S + 1           ← Increment (release one unit)
```
 
Both wait and signal are **atomic** — they cannot be interrupted midway.
 
---
 
#### Binary Semaphore (Mutex)
 
A semaphore with value **0 or 1 only** — used for mutual exclusion.
 
```
Semaphore mutex = 1   ← Initially 1 (unlocked)
 
Process 1:              Process 2:
wait(mutex)             wait(mutex)
[Critical Section]      [Critical Section]
signal(mutex)           signal(mutex)
```
 
**Execution:**
```
mutex = 1 (initially)
 
CPU1 calls wait(mutex):
  mutex = 1 > 0 → proceed
  mutex = 1 - 1 = 0
  CPU1 enters critical section
 
CPU2 calls wait(mutex):
  mutex = 0, not > 0 → CPU2 WAITS (blocked)
 
CPU1 calls signal(mutex):
  mutex = 0 + 1 = 1
 
CPU2 is unblocked:
  mutex = 1 > 0 → proceed
  mutex = 0
  CPU2 enters critical section
```
 
---
 
#### Counting Semaphore
 
A semaphore with a value **greater than 1** — used to manage a pool of resources.
 
**Example:** 3 identical resources (e.g., 3 printers, 3 database connections)
 
```
Semaphore resources = 3   ← 3 available
 
Any process wanting to use a resource:
  wait(resources)    ← decrements count, blocks if 0
  [Use resource]
  signal(resources)  ← increments count, unblocks a waiting process
```
 
When resources = 0, any further wait() calls will block until a signal() is called.
 
---
 
#### Semaphore vs Spin Lock — Comparison
 
| Feature | Spin Lock | Semaphore |
|---|---|---|
| Waiting behavior | CPU spins (busy-wait) | Process is blocked (sleeps) |
| CPU usage while waiting | High (wastes cycles) | Zero (process is suspended) |
| Best for | Very short critical sections | Longer critical sections |
| Context switch | No | Yes (on block/unblock) |
| Complexity | Simple | Moderate |
| Values | Binary (0 or 1) | Any non-negative integer |
 
---
 
### 9. Synchronization Summary
 
**Race Condition:**
```
Multiple processors access shared data simultaneously
Result depends on execution order
Leads to incorrect results
```
 
**Critical Section:**
```
Code region accessing shared resource
Only one processor at a time must execute it
Requires mutual exclusion
```
 
**Solutions:**
```
Disable Interrupts → Only for single processor, not multiprocessor
Test-and-Set       → Hardware atomic instruction, basis for all locks
Spin Lock          → Busy-wait, fast, wastes CPU, for short sections
Binary Semaphore   → Mutual exclusion, process sleeps while waiting
Counting Semaphore → Resource pool management (N resources)
```

### Diagram 1 — Daisy Chaining
 
```
 Arbiter
    |
   BG ──> [CPU1] ──BG──> [CPU2] ──BG──> [CPU3] ──BG──> [CPU4]
    |        |               |               |               |
    +--BR----+---------------+---------------+---------------+
             (all BR lines connect back to arbiter)
Priority: CPU1 > CPU2 > CPU3 > CPU4
```
 
### Diagram 2 — Independent Requesting
 
```
CPU1 ──BR1──> |         | ──BG1──> CPU1
CPU2 ──BR2──> | Arbiter | ──BG2──> CPU2
CPU3 ──BR3──> |         | ──BG3──> CPU3
CPU4 ──BR4──> |         | ──BG4──> CPU4
All requests received simultaneously — fastest method
```
 
### Diagram 3 — Race Condition on Counter
 
```
Initial: counter = 5
 
CPU1:  LOAD R1 = counter (R1=5)
CPU2:  LOAD R2 = counter (R2=5)   ← both read 5
CPU1:  ADD  R1 = R1+1   (R1=6)
CPU2:  ADD  R2 = R2+1   (R2=6)
CPU1:  STORE counter = R1 (counter=6)
CPU2:  STORE counter = R2 (counter=6)  ← overwrites!
 
Expected: 7    Actual: 6    One increment LOST
```

```
mutex = 1 (initial)
```
