# Cache Coherence

---

## 1. The Cache Coherence Problem

### 1.1 Recall — Single Processor Cache

In a single processor system, there is one CPU and one cache. When the CPU writes to a memory location:
- Cache is updated (write-through or write-back)
- Eventually RAM is updated
- No other CPU is involved — no problem

### 1.2 The Problem in Multiprocessors

In a multiprocessor system, each processor has its own private cache. They all share the same main memory (RAM).

**The problem arises when multiple processors cache the same memory location and one of them writes to it.**

```
Initial state: Memory[A] = 10

CPU1 cache: A = 10    (loaded from RAM)
CPU2 cache: A = 10    (loaded from RAM)
CPU3 cache: A = 10    (loaded from RAM)

Now CPU1 writes: A = 99

CPU1 cache: A = 99    (updated)
CPU2 cache: A = 10    (STALE — old value!)
CPU3 cache: A = 10    (STALE — old value!)
Main Memory: A = 10   (STALE — if write-back used)
```

CPU2 and CPU3 have a stale (outdated) copy of A. If they read A, they get the wrong value 10 instead of 99.

**This is the cache coherence problem.**

---

### 1.3 Formal Definition

A memory system is **cache coherent** if:

1. A read by processor P to location X, after a write to X by P with no write by any other processor in between, always returns the value written by P.

2. A read by processor P to location X, after a write to X by another processor Q, returns the value written by Q — provided the write and read are separated by sufficient time and no other writes occur.

3. Writes to the same location are **serialized** — all processors see them in the same order.

Simply stated: **All processors must see the most recent value of any memory location at all times.**

---

### 1.4 Three Sources of Incoherence

#### Source 1 — Write-Back Caches

When a processor writes to its cache (write-back policy), the value is not immediately written to RAM. Other processors reading from RAM or their own cache get the old value.

```
CPU1 writes A=99 to its cache (write-back)
RAM still has A=10
CPU2 reads A from RAM → gets 10 (wrong)
```

#### Source 2 — Write-Through Caches (partial problem)

Even with write-through (CPU1 writes to both cache and RAM), other processors may have already cached the old value in their private caches.

```
CPU2 cached A=10 before CPU1 wrote
CPU1 writes A=99 (write-through: cache + RAM updated)
CPU2 still has A=10 in its own cache → reads wrong value
```

#### Source 3 — I/O Device Access (DMA)

When a DMA controller writes data directly to RAM (bypassing CPU), the CPU's cached copy of that memory region becomes stale.

```
DMA writes new data to RAM address 0x5000
CPU1 has old data from 0x5000 in its cache
CPU1 reads from cache → gets old (stale) data
```

---

### 1.5 Coherence vs Consistency

These two terms are often confused.

| Term | Meaning |
|---|---|
| **Cache Coherence** | All processors see the same value for a single memory location. Concerned with one variable at a time. |
| **Memory Consistency** | The order in which writes to different locations become visible to all processors. Concerned with ordering of operations across multiple variables. |

Cache coherence is a necessary condition for correct multiprocessor operation.
Memory consistency is a stronger requirement about ordering.

This course focuses on **cache coherence**.

---

## 2. Solutions to Cache Coherence

Two broad approaches:

1. **Software-based** — compiler marks memory regions as non-cacheable or inserts flush instructions. Rarely used now.

2. **Hardware-based** — hardware automatically maintains coherence. Two methods:
   - **Snooping Protocols** — for bus-based (tightly coupled) systems
   - **Directory-based Protocols** — for large-scale systems without a shared bus

---

## 3. Snooping Protocols

### 3.1 Core Idea

In a **bus-based multiprocessor**, all processors share a common bus. Every bus transaction is **visible to all processors** — because all are connected to the same bus.

**Snooping** means each processor's cache controller **monitors (snoops) all bus transactions** to detect when another processor reads or writes a shared memory location.

```
CPU1  CPU2  CPU3  CPU4
 |     |     |     |
 +-----+-----+-----+  ← Shared Bus
             |
           RAM

All cache controllers watch EVERY bus transaction.
```

When a cache controller sees a relevant bus transaction — it takes appropriate action to maintain coherence.

---

### 3.2 Two Types of Snooping Protocols

#### Write-Invalidate Protocol

**Rule:** When a processor wants to write to a block, it first **invalidates all other copies** of that block in other caches. Then it writes exclusively.

```
CPU1 wants to write to block A:

Step 1: CPU1 broadcasts INVALIDATE(A) on bus
Step 2: All other caches snoop and see INVALIDATE(A)
Step 3: CPU2, CPU3, CPU4 mark their copy of A as INVALID
Step 4: CPU1 writes to A in its own cache
Step 5: CPU1 now has the only valid copy

When CPU2 later reads A:
Step 6: CPU2 sees its copy is invalid — cache miss
Step 7: CPU2 fetches latest A from CPU1 (or RAM)
```

**Key property:** At any time, either one processor has a modified (exclusive) copy, or multiple processors have identical read-only copies.

**Analogy:** Like a library book — when you want to write in it, you take it out exclusively. All other copies must be returned (invalidated). Others must borrow it fresh when they need it.

**Examples:** MESI protocol, most modern processors (Intel, AMD)

**Advantages:**
- Only one invalidation message needed per write (regardless of how many copies exist)
- Efficient when writes are followed by multiple writes by the same processor
- Less bus traffic for multiple writes to the same location

**Disadvantages:**
- Every read after an invalidation causes a cache miss
- High miss rate if the same location is frequently read by multiple processors after being written

---

#### Write-Update Protocol (Write-Broadcast)

**Rule:** When a processor writes to a block, it **broadcasts the new value** to all caches that have a copy. All copies are updated simultaneously.

```
CPU1 wants to write A = 99:

Step 1: CPU1 broadcasts UPDATE(A, 99) on bus
Step 2: All caches that have A update their copy to 99
Step 3: RAM is also updated (or flagged for update)

CPU2 reads A → gets 99 from its own cache (hit, correct value)
```

**Analogy:** Like a WhatsApp group — when one person updates information, everyone in the group gets the update immediately. No one needs to ask again.

**Advantages:**
- No cache misses after a write (all copies updated)
- Good when multiple processors frequently read a shared location after it is written

**Disadvantages:**
- Every write generates bus traffic to update all copies
- If only one processor reads the shared location, others receive unnecessary updates
- More bus bandwidth consumed

---

#### Write-Invalidate vs Write-Update — Comparison

| Feature | Write-Invalidate | Write-Update |
|---|---|---|
| On write | Invalidate all other copies | Update all other copies |
| Bus traffic per write | One invalidate message | One update per cached copy |
| Read after write | Cache miss (must fetch) | Cache hit (already updated) |
| Multiple writes by same CPU | Only first write causes traffic | Every write causes traffic |
| Best for | Write-heavy patterns | Read-after-write patterns |
| Used in practice | More common (MESI) | Less common |

## 4. MSI Protocol

MSI is the simplest coherence protocol. Each cache block can be in one of three states.

### 4.1 Three States

| State | Full Name | Meaning |
|---|---|---|
| **M** | Modified | This cache has the only valid copy. It has been written to. RAM is out of date. |
| **S** | Shared | This cache has a valid read-only copy. Other caches may also have valid copies. RAM is up-to-date. |
| **I** | Invalid | This cache block is invalid (stale or not present). Any access will cause a miss. |

---

### 4.2 MSI State Transitions

#### For a Read Operation:

```
I → S:  Cache miss. Processor reads block from RAM (or another cache).
         Block enters Shared state.

S → S:  Cache hit. Read served from local cache. State unchanged.

M → M:  Cache hit. Read served from local cache. State unchanged.
```

#### For a Write Operation:

```
I → M:  Cache miss. Processor fetches block, then writes to it.
         Issues INVALIDATE on bus. Block enters Modified state.

S → M:  Cache hit but must upgrade to write.
         Issues INVALIDATE on bus to invalidate all other copies.
         Block enters Modified state.

M → M:  Cache hit. Write served locally. State unchanged.
         No bus traffic needed.
```

#### When Another Processor Writes (Snoop on Bus):

```
S → I:  Another processor issued INVALIDATE for this block.
         This cache must invalidate its copy.

M → I:  Another processor wants this block (read or write).
         This cache must write back Modified data to RAM first.
         Then invalidate its own copy.
```

---

### 4.3 MSI State Transition Diagram

```
                 Own Read Miss
         +--------→ S ←--------+
         |          |           |
         |   Own    | Other     |
         |  Write   | Processor |
    I ←──+──────────+ Writes    |
    |    |          ↓           |
    |    +-----→ M              |
    |         Own Write         |
    |                           |
    +---------------------------+
        Own Read Miss
```

Simplified transitions:

```
I ---(own read)--→  S
I ---(own write)--→ M
S ---(own write)--→ M  (upgrade, send INVALIDATE)
S ---(other writes)→ I (snooped INVALIDATE)
M ---(other reads)--→ S (write back to RAM, share)
M ---(other writes)→ I (write back to RAM)
```

## 5. MESI Protocol

MESI is an extension of MSI that adds one more state — **Exclusive** — to improve performance for clean (unmodified) private data.

### 5.1 Four States

| State | Full Name | Meaning |
|---|---|---|
| **M** | Modified | Only valid copy. Written to. RAM is stale. Dirty. |
| **E** | Exclusive | Only valid copy. NOT written to yet. RAM is up-to-date. Clean. |
| **S** | Shared | Valid copy. Other caches may also have copies. RAM up-to-date. |
| **I** | Invalid | Not valid. Any access is a miss. |

---

### 5.2 Why is Exclusive State Needed?

In MSI, when a processor reads a block that no other processor has cached, it enters **Shared** state. If it then writes to that block, it must issue an INVALIDATE on the bus — even though no other processor has a copy (the invalidate is unnecessary!).

The **Exclusive** state represents a block that only this cache has — but it has not been modified yet. If the processor writes, it can silently move to Modified state **without any bus transaction** (no invalidate needed because no other cache has a copy).

```
MSI:  Read (alone) → S → Write → must broadcast INVALIDATE (wasteful)

MESI: Read (alone) → E → Write → silently become M (no bus traffic)
```

This reduces unnecessary bus traffic significantly.

---

### 5.3 MESI State Transitions

#### Read Operations:

```
I → E:  Cache miss. Processor reads block. No other cache has it.
         Block enters Exclusive (only copy, clean).

I → S:  Cache miss. Processor reads block. Other caches have copies.
         Block enters Shared.

E → E:  Cache hit. Read served locally. No state change.

S → S:  Cache hit. Read served locally. No state change.

M → M:  Cache hit. Read served locally. No state change.
```

#### Write Operations:

```
E → M:  Cache hit. Block was exclusive (only copy, clean).
         Write locally without any bus transaction.
         Block becomes Modified (dirty).

S → M:  Cache hit. Must invalidate other copies.
         Broadcasts INVALIDATE on bus.
         Block becomes Modified.

I → M:  Cache miss. Fetch block, then write.
         Broadcasts INVALIDATE on bus.
         Block becomes Modified.

M → M:  Cache hit. Write served locally. No bus transaction.
```

#### Snooped Bus Transactions (Another Processor Acting):

```
E → S:  Another processor reads same block.
         This cache shares it. Both become Shared.

E → I:  Another processor writes to same block.
         This cache invalidates its copy.

S → I:  Another processor writes (broadcasts INVALIDATE).
         This cache invalidates its copy.

M → S:  Another processor reads same block.
         This cache must write back modified data to RAM first.
         Then both have Shared copies.

M → I:  Another processor writes to same block.
         This cache must write back modified data to RAM first.
         Then invalidate its own copy.
```

---

### 5.4 MESI State Summary Table

| State | Only Copy? | Modified? | RAM Valid? | On Own Write | On Other Read | On Other Write |
|---|---|---|---|---|---|---|
| **M** | Yes | Yes | No | Stay M | Write back → S | Write back → I |
| **E** | Yes | No | Yes | Become M (silent) | Become S | Become I |
| **S** | No | No | Yes | Become M + INVAL | Stay S | Become I |
| **I** | — | — | — | Fetch → M + INVAL | Fetch → E or S | Fetch → M + INVAL |

---

### 5.5 MESI State Transition Diagram

```
              Read (others have it)
         ┌────────────────────────────┐
         |                            ↓
    I ───+──── Read (no others) ───→  E
    ↑    |                            |
    |    +──── Write ──────────────→  M ←─── Write (from E, silent)
    |                                 |
    |         Other reads             |
    |    ┌────────────────────────────┘
    |    ↓
    |    S ←─── Read (others have it)
    |    |
    +────+ (other writes INVALIDATE)
```

Simplified state machine:

```
         Read hit    Write hit
    M ──────────→ M ──────────→ M
    |                           ↑
    | other reads               | Write (silent upgrade)
    ↓                           |
    S ←── Read hit ──── S      E ← Read (no sharers)
    |                           |
    | other writes              | other reads/writes
    ↓                           ↓
    I ←──────────────────────── I
```

---

### 5.6 MESI — Worked Example

**Scenario: CPU1, CPU2, CPU3 accessing memory block A**

```
Initial: All caches empty. RAM: A = 10.

Step 1: CPU1 reads A
  CPU1 cache: A = 10, state = E  (only copy, clean)
  CPU2 cache: empty
  RAM: A = 10

Step 2: CPU2 reads A
  CPU1 snoops CPU2's read request
  CPU1: A transitions E → S
  CPU2: A = 10, state = S
  CPU1: A = 10, state = S
  RAM: A = 10

Step 3: CPU1 writes A = 99
  CPU1 broadcasts INVALIDATE(A)
  CPU2 snoops → invalidates its copy: A state = I
  CPU1: A = 99, state = M
  CPU2: A = I
  RAM: A = 10  (stale, not updated yet — write-back)

Step 4: CPU2 reads A
  CPU2 cache miss (state = I)
  CPU2 broadcasts read request on bus
  CPU1 snoops → must supply data (write-back to RAM)
  CPU1 writes A = 99 back to RAM
  CPU1: A = 99, state = S (or I depending on protocol variant)
  CPU2: A = 99, state = S
  RAM: A = 99  (now updated)
```

## 6. Directory-Based Protocols

### 6.1 Why Directory-Based?

Snooping protocols rely on a **shared bus** that broadcasts all transactions to all processors. This works well for small systems (up to ~32 processors).

**For large systems:**
- Hundreds or thousands of processors
- No single shared bus — uses interconnection networks
- Broadcasting to all processors on every memory access is too expensive and does not scale

**Solution → Directory-based coherence**

Instead of broadcasting, a **directory** keeps track of which caches have a copy of each memory block. Only the relevant caches are notified.

---

### 6.2 Directory Structure

A **directory** is a data structure (stored in memory, often distributed) that records, for each memory block:

1. **Whether the block is cached** (and where)
2. **The state of the block** (modified, shared, invalid)
3. **Which processors have a copy** (presence bits or a list)

```
Directory Entry for Block A:
+--------+---------+------------------+
| State  | Owner   | Sharers Bitmap   |
+--------+---------+------------------+
| Shared | —       | 1 0 1 0  (CPU1,CPU3 have copies) |
+--------+---------+------------------+

Directory Entry for Block B:
+--------+---------+------------------+
| State  | Owner   | Sharers Bitmap   |
+--------+---------+------------------+
| Modified | CPU2  | 0 1 0 0          |
+--------+---------+------------------+
```

For N processors, each directory entry needs N bits (presence bits) to track which processors have copies.

---

### 6.3 Directory States

Each block in the directory can be in one of three states:

| State | Meaning |
|---|---|
| **Uncached** | No processor currently has this block in cache |
| **Shared** | One or more processors have a read-only (clean) copy |
| **Modified** | Exactly one processor has the block, and it has been written to (dirty) |

---

### 6.4 Directory Protocol — How it Works

#### Read Request (Block is Uncached)

```
Step 1: CPU1 sends read request to Home Node (memory + directory)
Step 2: Directory finds block is Uncached
Step 3: Directory fetches block from RAM
Step 4: Directory sends block to CPU1
Step 5: Directory updates: State = Shared, CPU1 bit = 1
```

#### Read Request (Block is Shared)

```
Step 1: CPU1 sends read request to Home Node
Step 2: Directory finds block is Shared (already cached by others)
Step 3: Directory sends block to CPU1
Step 4: Directory updates: CPU1 bit = 1 (adds CPU1 to sharers list)
```

#### Read Request (Block is Modified — another CPU owns it)

```
Step 1: CPU1 sends read request to Home Node
Step 2: Directory finds block is Modified — CPU2 owns it
Step 3: Directory sends FETCH message to CPU2
Step 4: CPU2 writes back modified data to RAM (home node)
Step 5: CPU2 transitions its state: M → S
Step 6: Directory sends block to CPU1
Step 7: Directory updates: State = Shared, CPU1 and CPU2 bits = 1
```

#### Write Request (Block is Shared)

```
Step 1: CPU1 wants to write to block A
Step 2: CPU1 sends write request to Home Node
Step 3: Directory finds block is Shared — CPU2 and CPU3 have copies
Step 4: Directory sends INVALIDATE to CPU2 and CPU3
Step 5: CPU2 and CPU3 invalidate their copies, send ACK
Step 6: Directory sends block + write permission to CPU1
Step 7: Directory updates: State = Modified, Owner = CPU1
         CPU2 and CPU3 bits = 0
```

#### Write Request (Block is Modified — another CPU owns it)

```
Step 1: CPU1 wants to write to block A
Step 2: CPU1 sends write request to Home Node
Step 3: Directory finds block is Modified — CPU2 owns it
Step 4: Directory sends INVALIDATE + FETCH to CPU2
Step 5: CPU2 writes back data to RAM, invalidates its copy, sends ACK
Step 6: Directory sends block + write permission to CPU1
Step 7: Directory updates: State = Modified, Owner = CPU1
```

---

### 6.5 Advantages and Disadvantages of Directory-Based

**Advantages:**
- **Scalable** — no broadcasting; only relevant caches are contacted
- Works with **any interconnection topology** — not limited to bus
- Suitable for **large systems** (hundreds to thousands of processors)
- Reduced bandwidth — targeted messages instead of broadcasts

**Disadvantages:**
- **Directory storage overhead** — for N processors and M memory blocks, directory size = N × M bits
- **Higher latency** — each coherence action requires multiple messages through directory
- **Directory is a potential bottleneck** — if centralized
- **Complex implementation** — many message types and state combinations

---

## 7. Snooping vs Directory-Based — Full Comparison

| Feature | Snooping Protocol | Directory-Based Protocol |
|---|---|---|
| Interconnect required | **Shared bus** | Any network |
| Communication method | **Broadcast** to all | **Unicast** to specific caches |
| Scalability | **Poor** (bus saturation) | **High** (no broadcast) |
| Latency | Low (bus broadcast fast) | Higher (directory lookup + messages) |
| Hardware cost | Simple (no directory) | Higher (directory storage) |
| Best for | Small systems (2–32 CPUs) | Large systems (32+ CPUs) |
| Bandwidth usage | High (all see all transactions) | Low (targeted messages) |
| Examples | Intel Core (shared L3 bus) | SGI Origin, large NUMA servers |
