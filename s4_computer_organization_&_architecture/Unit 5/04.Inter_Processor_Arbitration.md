# Inter-processor Arbitration And Arbitration Method



## 1. Inter-processor Arbitration
 
### 1.1 What is Arbitration and Why is it Needed?
 
In a multiprocessor system, multiple processors share common resources — the **system bus**, **shared memory**, and **I/O devices**.
 
A conflict arises when two or more processors try to use the same resource at the same time. Only one can use it at a time.
 
**Arbitration** is the process of deciding **which processor gets access to the shared resource** when multiple processors request it simultaneously.
 
Without arbitration — two processors simultaneously driving the bus would cause electrical conflicts and data corruption.
 
**Example:**
```
CPU1 wants to write to RAM at address 0x1000
CPU2 also wants to write to RAM at address 0x1000
Both at the same time → Who goes first?
Arbitration decides this.
```
 
---
 
### 1.2 Key Terms
 
| Term | Meaning |
|---|---|
| **Bus Master** | The processor currently controlling the bus |
| **Bus Request (BR)** | Signal a processor raises to request bus access |
| **Bus Grant (BG)** | Signal sent back to processor giving it permission |
| **Arbiter** | Hardware unit that receives all requests and grants access to one |
| **Priority** | Which processor gets access when multiple request simultaneously |
 
---
 
### 1.3 Types of Arbitration
 
Arbitration is classified in two ways:
 
**By location of arbiter:**
- **Centralized** — one dedicated arbiter handles all requests
- **Distributed** — no single arbiter; processors collectively decide
 
**By method:**
- **Daisy Chaining**
- **Polling (Centralized Parallel)**
- **Independent Requesting**
 
---
 
## 2. Arbitration Methods
 
### 2.1 Daisy Chaining
 
#### Structure
 
All processors are connected in a **linear chain**. The bus grant signal travels through the chain from the processor closest to the arbiter outward.
 
```
Arbiter
   |
  BG line (Bus Grant passes through chain)
   |
[CPU1] --BG--> [CPU2] --BG--> [CPU3] --BG--> [CPU4]
   |             |              |               |
   +-------------+--------------+---------------+
                       BR line (all connect back to arbiter)
```
 
#### How Daisy Chaining Works
 
```
Step 1: One or more CPUs raise Bus Request (BR) signal to arbiter
Step 2: Arbiter sees bus is free → asserts Bus Grant (BG) signal
Step 3: BG signal enters CPU1 first
Step 4a: If CPU1 wants the bus → CPU1 takes it, does NOT pass BG further
Step 4b: If CPU1 does not want bus → CPU1 passes BG to CPU2
Step 5: Same logic at CPU2, CPU3, CPU4
Step 6: First CPU in chain that wants bus takes it
Step 7: That CPU uses bus, then releases it
Step 8: Arbiter can issue next BG
```
 
#### Priority in Daisy Chaining
 
Priority is determined by **physical position** in the chain.
 
- CPU1 (closest to arbiter) = **highest priority**
- CPU4 (farthest from arbiter) = **lowest priority**
 
If CPU1 and CPU4 both request simultaneously, CPU1 always wins.
 
#### Advantages
 
- **Simple hardware** — very few connections needed
- **Easy to implement** — just a chain of connections
- **Cheap** — low hardware cost
 
#### Disadvantages
 
- **Priority is fixed** — lower priority devices may never get bus (starvation)
- **Not fair** — CPU1 can monopolize the bus
- **Failure propagation** — if one CPU fails, all CPUs after it in chain are cut off
- **Slow** — BG must propagate through every CPU in chain

### 2.2 Polling (Centralized Parallel Arbitration)
 
#### Structure
 
A **centralized arbiter** connects to all processors. The arbiter actively **polls** each processor in sequence to check if it wants the bus.
 
```
            Arbiter
           /   |   \
          /    |    \
       CPU1  CPU2  CPU3  CPU4
       
Arbiter sends poll signal to each in turn:
"CPU1, do you want the bus?"
"CPU2, do you want the bus?"
...
```
 
#### How Polling Works
 
```
Step 1: Arbiter sends a poll address on the bus (e.g., address 01 = poll CPU1)
Step 2: CPU1 checks — does it want the bus?
        If yes → CPU1 asserts a "bus busy" signal and takes the bus
        If no  → Arbiter moves to next address (CPU2)
Step 3: Arbiter repeats for each processor in polling sequence
Step 4: First processor that wants the bus gets it
Step 5: Processor uses bus, releases it, arbiter resumes polling
```
 
#### Priority in Polling
 
Priority is determined by **polling order** set by the arbiter.
 
- The arbiter can be programmed to change priority dynamically
- Can implement **round-robin** — each processor gets equal turns
- Can implement **priority-based** — higher priority polled first
 
#### Advantages
 
- **Flexible priority** — can be changed by reprogramming arbiter
- **No starvation possible** with round-robin
- **Failure of one CPU** does not affect others
 
#### Disadvantages
 
- **Slower** than daisy chain — arbiter must poll each device
- **Polling overhead** — time spent asking devices even when none need bus
- **Arbiter is a single point of failure**
- **Wasted cycles** when no device needs the bus
 
---
 
### 2.3 Independent Requesting
 
#### Structure
 
Each processor has **its own dedicated Bus Request (BR) line** to the arbiter and its own dedicated **Bus Grant (BG) line** from the arbiter.
 
```
CPU1 ---BR1---> |         | ---BG1---> CPU1
CPU2 ---BR2---> | Arbiter | ---BG2---> CPU2
CPU3 ---BR3---> |         | ---BG3---> CPU3
CPU4 ---BR4---> |         | ---BG4---> CPU4
```
 
#### How Independent Requesting Works
 
```
Step 1: One or more CPUs assert their individual BR lines simultaneously
Step 2: Arbiter receives all requests at the same time (parallel)
Step 3: Arbiter uses priority logic to select one winner
Step 4: Arbiter asserts the corresponding BG line for the winner
Step 5: That CPU takes the bus and begins transfer
Step 6: CPU releases bus when done
Step 7: Arbiter can now process next request
```
 
#### Priority in Independent Requesting
 
Priority is encoded in the arbiter's logic:
- Fixed priority — higher priority CPUs always win
- Dynamic priority — priority rotates to ensure fairness
 
#### Advantages
 
- **Fastest response** — all requests received simultaneously, no propagation delay
- **Flexible priority** — arbiter can implement any priority scheme
- **Single CPU failure** does not affect others
 
#### Disadvantages
 
- **Most expensive** — requires dedicated BR and BG lines for each CPU
- **Arbiter becomes complex** as number of processors increases
- **Arbiter is still a single point of failure**
 
---
 
### 2.4 Arbitration Methods — Comparison Table
 
| Feature | Daisy Chaining | Polling | Independent Requesting |
|---|---|---|---|
| Arbiter type | Centralized (simple) | Centralized | Centralized |
| Priority | Fixed by position | Flexible (programmable) | Flexible (programmable) |
| Hardware cost | Cheapest | Moderate | Most expensive |
| Speed | Slow (chain propagation) | Moderate (polling delay) | Fastest (parallel) |
| Starvation risk | High (low priority CPUs) | Low (round-robin possible) | Low |
| Failure impact | Chain broken after failure | Only failed CPU affected | Only failed CPU affected |
| Scalability | Poor | Moderate | Poor (wires per CPU) |
| Complexity | Simplest | Moderate | Most complex |
 
---
 
### 2.5 Centralized vs Distributed Arbitration
 
#### Centralized Arbitration
 
- A **single dedicated arbiter** handles all bus grant decisions
- All three methods above (daisy chain, polling, independent) are centralized
- Simple to implement
- Single point of failure
 
```
        [Arbiter]
       /    |    \
    CPU1  CPU2  CPU3
```
 
#### Distributed Arbitration
 
- **No single arbiter** — each processor participates in the arbitration process
- Processors collectively decide who wins by exchanging signals
- More fault-tolerant — no single point of failure
- More complex
 
**One method — Self-Selection:**
Each processor that wants the bus broadcasts its ID. All processors observe all IDs. Each processor applies the same priority logic. All agree on the winner without a central unit.
