# 1. Introduction — Why I/O is Needed

A computer is not just a CPU and memory. It needs to communicate with the outside world — keyboard, mouse, monitor, disk, printer, network. This communication happens through **Input/Output (I/O).**

**The Core Problem:**

| Device | Speed |
|---|---|
| CPU | GHz — billions of operations per second |
| RAM | Nanoseconds |
| Keyboard | Human typing speed — very slow |
| HDD | Milliseconds |
| Printer | Very slow mechanical device |

CPU is extremely fast. I/O devices are extremely slow. If CPU directly managed every I/O device — it would waste almost all its time waiting.

**Solution → I/O Interface**

---

## 2. I/O Interface

### 2.1 What is an I/O Interface?

An **I/O Interface** (also called an **I/O module** or **I/O controller**) is a circuit/chip that sits between the CPU/memory and the I/O devices.

It acts as a **bridge** — handling the communication between the fast CPU/memory side and the slow I/O device side.

```
CPU / Memory Bus
      │
┌─────▼─────────────────┐
│     I/O Interface     │  ← I/O Controller / I/O Module
│  ┌─────────────────┐  │
│  │  Data Register  │  │
│  │ Status Register │  │
│  │Control Register │  │
│  └─────────────────┘  │
└─────────┬─────────────┘
          │
    ┌─────▼──────┐
    │  I/O Device│  ← Keyboard, Printer, Disk, etc.
    └────────────┘
```

---

### 2.2 Why is I/O Interface Needed?

| Reason | Explanation |
|---|---|
| **Speed mismatch** | CPU is GHz fast, devices are very slow — interface buffers this difference |
| **Data format mismatch** | CPU uses parallel data (multiple bits at once), some devices use serial data (one bit at a time) |
| **Voltage mismatch** | Devices may operate at different voltage levels than CPU |
| **Protocol differences** | Different devices have different communication protocols — interface handles translation |
| **Multiple devices** | CPU should not hardwire logic for every device — interface provides standard connection |

---

### 2.3 Functions of I/O Interface

1. **Data buffering** — temporarily holds data while speed difference is managed
2. **Error detection** — checks for errors in data transfer
3. **Communication with CPU** — via bus (address, data, control lines)
4. **Device control** — sends commands to the I/O device
5. **Status reporting** — tells CPU whether device is ready, busy, or has error
6. **Data conversion** — serial to parallel, voltage level shifting, format conversion

---

### 2.4 I/O Registers

Every I/O interface has three types of registers that the CPU communicates with:

#### Data Register
- Holds the **actual data** being transferred to/from the device
- Example: Keyboard data register holds the ASCII code of the key pressed
- CPU reads this to get input, writes this to send output

#### Status Register
- Holds the **current status** of the I/O device
- CPU reads this to know if device is ready
- Common status bits:

| Bit | Meaning |
|---|---|
| **BUSY** | Device is currently busy (1 = busy, 0 = ready) |
| **READY** | Device is ready to transfer data |
| **ERROR** | An error occurred during transfer |
| **DONE** | Transfer is complete |

#### Control Register
- CPU **writes commands** here to control the device
- Example: Tell a disk to seek to track 5, tell a printer to start printing
- Controls device mode, direction of transfer, interrupt enable/disable

---

### 2.5 I/O Ports

An **I/O port** is the address used to access an I/O register.

Each register (data, status, control) has a unique **port address** — just like memory has addresses.

The CPU uses special I/O instructions to read/write these port addresses:
- `IN` instruction — reads from an I/O port
- `OUT` instruction — writes to an I/O port

(In x86 architecture — `IN AL, port` reads from port into register AL)

## 3. Programmed I/O 

### 3.1 What is Programmed I/O?

**Programmed I/O** (also called **Polling** or **Busy-Wait I/O**) is the simplest I/O technique where:

> **The CPU directly controls the entire I/O operation and continuously checks (polls) the device status until the transfer is complete.**

The CPU is fully in charge — it does everything itself, step by step, through a program.

---

### 3.2 How Programmed I/O Works — Step by Step

**Example: CPU wants to read data from keyboard**

```
Step 1: CPU writes command to Control Register
        → "Start reading, prepare data"

Step 2: CPU reads Status Register
        → Is BUSY bit = 0? (Is device ready?)

Step 3: If BUSY = 1 (device not ready)
        → CPU loops back to Step 2
        → Keeps checking again and again
        → This loop is called POLLING or BUSY-WAIT

Step 4: If BUSY = 0 (device is ready / data available)
        → CPU reads Data Register
        → Data is transferred to CPU register

Step 5: CPU processes the data / continues program
```

**Flow diagram:**

```
CPU issues I/O command
        ↓
┌──────────────────┐
│  Read Status Reg │ ← Polling loop starts
└──────┬───────────┘
       │
  BUSY = 1?
   ↙        ↘
 Yes          No (Ready)
  ↑               ↓
  └─────────  Read Data Register
              ↓
         Data transferred to CPU
              ↓
         Continue program
```

---

### 3.3 Programmed I/O — Real Life Analogy

> Imagine you ordered food from a restaurant. You sit at the table and **keep asking the waiter every 30 seconds** — "Is my food ready? Is my food ready? Is my food ready?"
>
> You are doing nothing else — just waiting and asking repeatedly.
>
> Your entire time is wasted in polling. This is **Programmed I/O / Busy-Wait.**

---

### 3.4 Characteristics of Programmed I/O

| Feature | Value |
|---|---|
| CPU involvement | **Very high** — CPU does everything |
| Polling | CPU continuously checks device status |
| CPU efficiency | **Very low** — CPU wastes time waiting |
| Implementation | **Simplest** — easy to program |
| Hardware required | **Minimal** — no extra hardware needed |
| Suitable for | Simple embedded systems, very slow devices |

---

### 3.5 Advantages and Disadvantages

**Advantages:**
1. Very **simple to implement** — just a loop in software
2. **No extra hardware** needed
3. Good for **simple, slow systems** where CPU has nothing else to do
4. **Deterministic** — exact timing is predictable

**Disadvantages:**
1. **CPU is wasted** — keeps busy-waiting instead of doing useful work
2. **Very inefficient** for fast systems with multiple devices
3. **No parallelism** — CPU cannot do other tasks while waiting
4. Slows down the entire system

---

### 3.6 When to Use Programmed I/O?

- Simple microcontrollers (washing machine controller, traffic light controller)
- Systems where only one task runs at a time
- Real-time systems where precise timing matters
- When hardware cost must be minimized

## 4. I/O Addressing — Isolated I/O vs Memory Mapped I/O

Before understanding Memory Mapped I/O, understand the two ways I/O devices can be addressed.

### 4.1 Isolated I/O (Port-Mapped I/O)

In **Isolated I/O:**
- I/O devices have a **separate address space** from memory
- CPU has **two different sets of instructions** — one for memory, one for I/O
- Memory instructions: `LOAD`, `STORE`
- I/O instructions: `IN`, `OUT`
- The CPU uses a special control signal (`M/IO` line) to indicate whether the current bus transaction is for memory or I/O

```
Address Space:
┌────────────────────┐
│                    │  ← Memory Address Space (e.g., 0x00000 to 0xFFFFF)
│    RAM / ROM       │
│                    │
└────────────────────┘

┌────────────────────┐
│  I/O Port Space    │  ← Separate I/O Address Space (e.g., 0x000 to 0x3FF)
│  (I/O devices)     │     Accessed only via IN/OUT instructions
└────────────────────┘
```

**Key point:** Memory and I/O have **separate address spaces** — no overlap.

---

### 4.2 Memory Mapped I/O 

In **Memory Mapped I/O:**
- I/O device registers are **mapped into the same address space as memory**
- No separate I/O instructions — the same `LOAD` and `STORE` (or `MOV`) instructions used for memory are used for I/O too
- Certain memory addresses are **reserved for I/O devices**
- When CPU accesses these special addresses — it is actually communicating with the I/O device, not RAM

```
Unified Address Space:
┌────────────────────┐  ← Address 0x00000
│                    │
│    RAM / ROM       │
│                    │
├────────────────────┤  ← Address 0xF0000
│  I/O Device Regs  │  ← These addresses map to I/O devices, not RAM
│  (Keyboard)        │     Access them like normal memory locations
│  (Display)         │
│  (Disk Controller) │
└────────────────────┘  ← Address 0xFFFFF
```

**Example:**
- Address `0xFFF0` is mapped to keyboard data register
- CPU does `LOAD R1, 0xFFF0` → reads keyboard input (not RAM!)
- CPU does `STORE 0xFFF4, R2` → writes to display register (not RAM!)

---

### 4.3 How Memory Mapped I/O Works

```
CPU generates address 0xFFF0 (keyboard register)
              ↓
Address decoder checks: is this a RAM address or I/O address?
              ↓
0xFFF0 is in reserved I/O region → route to I/O controller
              ↓
I/O controller receives the read/write request
              ↓
Data transferred between CPU and I/O device
```

The CPU does not know (or care) whether it is accessing RAM or an I/O device — the hardware handles routing transparently.

---

### 4.4 Real Life Analogy for Memory Mapped I/O

> Imagine a city where **houses and offices share the same address numbering system.**
>
> House no. 1–900 = normal homes (RAM)
> House no. 901–1000 = reserved for government offices (I/O devices)
>
> When a postman delivers mail — he just uses the address. He does not need separate rules for offices vs homes. Same delivery system (LOAD/STORE) works for all.
>
> **This is Memory Mapped I/O** — same address space, same instructions, different destinations.

---

## 5. Memory Mapped I/O vs Isolated I/O 

| Feature | Memory Mapped I/O | Isolated I/O (Port Mapped) |
|---|---|---|
| Address space | **Shared** with memory | **Separate** from memory |
| Instructions for I/O | Normal LOAD/STORE/MOV | Special IN/OUT instructions |
| Address space size | Memory address space is reduced | Memory address space unchanged |
| Hardware | Simpler (no M/IO line needed) | Needs M/IO control signal |
| Programming | Simpler — use any memory instruction | Slightly more complex |
| Protection | Harder to protect I/O from accidental access | Easier — IN/OUT can be restricted |
| Used in | ARM processors, most embedded systems | x86 (Intel/AMD) processors |
| Flexibility | High — can use all addressing modes | Limited to IN/OUT |
| Speed | Same as memory access | Same as memory access |

---

## 6. Isolated I/O vs Memory Mapped I/O — Address Space Diagram

```
ISOLATED I/O:
Memory:     [0x00000 ─────────────────── 0xFFFFF]  (full memory space)
I/O Ports:  [0x000 ──── 0x3FF]                     (separate, smaller I/O space)
            Two separate buses / address decoders

MEMORY MAPPED I/O:
Combined:   [0x00000 ───────── 0xBFFFF]  ← RAM area
            [0xC0000 ──────── 0xFFFFF]   ← Reserved for I/O devices
            One unified address space — no separation
```
